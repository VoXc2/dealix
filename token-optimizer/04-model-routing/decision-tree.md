# شجرة القرار: أي نموذج لأي مهمة؟

## الخوارزمية

```
هل المهمة ميكانيكية ومتكررة؟
├── نعم → HAIKU
│   (إعادة تسمية، تنسيق، boilerplate، ترجمة)
│
└── لا → هل تحتاج تحليلاً عميقاً أو كود معقد جداً؟
    ├── لا → SONNET (الخيار الافتراضي لـ 80% من المهام)
    └── نعم → هل المهمة أكثر من 10 ملفات أو معمارية رئيسية؟
        ├── لا → SONNET (لا يزال كافياً)
        └── نعم → OPUS
```

## جدول سريع

| المهمة | النموذج | السبب |
|--------|---------|-------|
| إعادة تسمية متغيرات | Haiku | ميكانيكية |
| تنسيق الكود (prettier/ruff) | Haiku | ميكانيكية |
| إضافة type hints | Haiku | ميكانيكية |
| كتابة boilerplate | Haiku | متكررة |
| ترجمة رسائل خطأ | Haiku | ترجمة |
| إصلاح bug بسيط | Sonnet | يحتاج فهم |
| كتابة endpoint جديد | Sonnet | معياري |
| كتابة اختبارات | Sonnet | معياري |
| مراجعة PR (< 5 ملفات) | Sonnet | معياري |
| تفسير كود | Sonnet | معياري |
| refactor (ملف واحد) | Sonnet | معياري |
| تصميم معمارية | Opus | تحليل عميق |
| refactor (> 10 ملفات) | Opus | تعقيد عالي |
| تحليل أمني | Opus | دقة مطلوبة |
| debugging معقد (race conditions, memory leaks) | Opus | تحليل عميق |
| مراجعة PR كبيرة (> 10 ملفات) | Opus | شاملة |

## إعداد Subagents بنماذج أرخص

```bash
# في .claude/settings.json
CLAUDE_CODE_SUBAGENT_MODEL=haiku
```

Subagents كثيراً ما يؤدون مهام استكشافية (قراءة ملفات، بحث)
وHaiku يكفي لذلك، مما يوفر تكلفة كبيرة.
