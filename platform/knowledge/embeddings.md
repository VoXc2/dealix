# التضمين — طبقة المعرفة (الطبقة الرابعة)

> كيف تتحوّل المقاطع النصّيّة إلى متّجهات قابلة للبحث الدلالي، وكيف تُخزَّن وتُحدَّث ضمن نطاق المستأجر.

## 1. الغرض

خدمة التضمين تحوّل كل مقطع نصّي إلى متّجه عددي يمثّل معناه، ليصبح البحث الدلالي ممكنًا. التضمين هو الأساس التقني للاسترجاع الهجين، لكنه لا يُعد مصدرًا بذاته؛ المصدر يبقى الملف الأصلي.

## 2. النموذج والأبعاد

- النموذج المستخدم: `text-embedding-3-small` كما هو معرّف في `core/memory/embedding_service.py`.
- عدد الأبعاد: 1536.
- يُحسب التشابه بجيب التمام (cosine similarity) في كود Python نقي، فلا يلزم امتداد قاعدة بيانات في بيئة التطوير.

## 3. التخزين

- تُخزَّن متّجهات الحسابات في `AccountEmbeddingRecord` ومتّجهات المحادثات في `ConversationEmbeddingRecord`.
- كل سجلّ تضمين يحمل `tenant_id` واسم النموذج والنص المستخدم وطابعًا زمنيًّا.
- البحث مقيّد دائمًا بـ `tenant_id`؛ لا يوجد بحث عابر للمستأجرين.

## 4. التحديث وإعادة التضمين

- عند تغيّر المصدر تُحدَّث متّجهات المقاطع المتأثّرة عبر عمليّة upsert، لا بإنشاء سجلّات مكرّرة.
- التضمين الدفعي عبر `embed_batch` يعالج عدّة نصوص في استدعاء واحد لخفض الكلفة.
- تغيير اسم النموذج يُسجَّل في حقل `model_name`، فيمكن تمييز المتّجهات المُولَّدة بنماذج مختلفة.

## 5. المرونة عند الفشل

عند تعذّر استدعاء واجهة التضمين تُعيد الخدمة متّجهًا صفريًّا بدل أن تتعطّل، ويُسجَّل تحذير. المقطع ذو المتّجه الصفري لا يُطابَق في البحث الدلالي، لكنه يبقى قابلًا للاسترجاع المعجمي ويُعاد تضمينه لاحقًا.

## 6. الحوكمة

- لا يُستخدم نص فيه بيانات شخصيّة غير محجوبة في التضمين؛ الحجب يتم في الاستيعاب.
- التضمين لا يُنشئ معرفة جديدة؛ غياب المقطع المصدر يعني «أدلّة غير كافية».
- مفاتيح الواجهة الخارجيّة تُحفظ في متغيّرات البيئة، لا في الكود.

# English

# Embeddings — Knowledge Layer (Layer 4)

> How text chunks become searchable vectors, and how they are stored and updated within a tenant scope.

## 1. Purpose

The embedding service converts each text chunk into a numeric vector that represents its meaning, enabling semantic search. Embeddings are the technical foundation of hybrid retrieval, but they are not a source themselves; the source remains the origin file.

## 2. Model and dimensions

- Model used: `text-embedding-3-small` as defined in `core/memory/embedding_service.py`.
- Dimensions: 1536.
- Similarity is computed with cosine similarity in pure Python, so no database extension is required in the development environment.

## 3. Storage

- Account vectors are stored in `AccountEmbeddingRecord` and conversation vectors in `ConversationEmbeddingRecord`.
- Every embedding record carries a `tenant_id`, the model name, the text used, and a timestamp.
- Search is always scoped by `tenant_id`; there is no cross-tenant search.

## 4. Updates and re-embedding

- When a source changes, the vectors of affected chunks are updated via an upsert, not by creating duplicate records.
- Batch embedding via `embed_batch` processes multiple texts in a single call to reduce cost.
- A model name change is recorded in the `model_name` field, so vectors generated by different models can be distinguished.

## 5. Failure resilience

If the embedding API call fails, the service returns a zero vector instead of crashing, and a warning is logged. A chunk with a zero vector does not match in semantic search but remains retrievable lexically and is re-embedded later.

## 6. Governance

- Text containing un-redacted personal data is not used for embedding; redaction happens at ingestion.
- Embeddings create no new knowledge; absence of the source chunk means "insufficient evidence."
- External API keys are kept in environment variables, not in code.
