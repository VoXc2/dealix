## 1. تعريف القطاع

| البُعد | التفاصيل |
|--------|----------|
| **الوصف** | جهات حكومية، شركات مملوكة للدولة، صناديق استثمارية حكومية |

## 2. نقاط الألم الرئيسية

| الألم | الظهور الفعلي | الأثر |
|-------|---------------|-------|
| **إثبات الامتثال للرقابة** | ديوان المراقبة أو جهة رقابية تطلب دليلاً على مسار القرارات المالية | إجراءات تحقيق مُرهقة + مخاطر قانونية |
| **لا audit trail رقمي** | القرارات تُتخذ في اجتماعات — المحاضر يدوية، لا سجل رقمي قابل للتحقق | صعوبة إثبات الإجراء السليم |
| **PDPL في الخدمات الرقمية** | الجهات الحكومية تُعالج بيانات المواطنين — متطلبات PDPL صارمة | مخاطر غرامات SDAIA |
| **موافقات دورة الشراء** | كل عقد يمر بـ 6–10 مستويات موافقة — التأخير هو القاعدة | تأخير مشاريع + رضا منخفض للموردين |
| **تقارير الأداء للقيادة العليا** | التقارير يدوية، تتأخر، تحتاج جهداً كبيراً لإعدادها | قيادة تعمل بمعلومات منقوصة |

## 3. صنّاع القرار

| الشخصية | الدور | ما يهمه | كيف تصل إليه |
|---------|-------|---------|--------------|
| **مدير تقنية المعلومات** | بوابة التقييم التقني | Data residency، أمان، PDPL compliance | Technical session مسبق |
| **المستشار القانوني** | يُقيّم المخاطر القانونية | PDPL، SDAIA، عقود بنود واضحة | وثيقة compliance |
| **مدير الشراء / مشتريات** | يُنفّذ RFP والتقييم | معايير اختيار، مقارنة الحلول | تسجيل كمورد في Etimad |
| **المدير العام / نائب الرئيس** | القرار الاستراتيجي | مطابقة رؤية 2030، التحديث الرقمي | Referral رسمي |
| **مستشار الرقمنة / التحول** | يدفع للتغيير | تحقيق أهداف التحول الرقمي | المؤتمرات + LinkedIn |

## 4. الـ Wedge الأمثل

**الوتد الأقوى: audit trail للمراجع الحكومي**

```
الدخول:  "عند مراجعة ديوان المراقبة أو أي جهة رقابية — كيف تثبتون أن 
          قراراتكم المالية اتُّخذت وفق الصلاحيات المحددة؟"
المشكلة: لا دليل رقمي على مسار القرارات
الحل:    Trust & Evidence Layer + Approval OS
العرض:   Demo 4 — GCC Trust Layer
```

**الوتد الثاني: PDPL في الخدمات الرقمية**

```
الدخول:  "الخدمات الرقمية التي تُعالج بيانات المواطنين — كيف تُوثّقون 
          consent وحق الحذف وفق PDPL؟"
المشكلة: PDPL compliance غير كامل
الحل:    PDPL Consent Layer
العرض:   Demo 4
```

## 5. متطلبات ما قبل البيع في القطاع الحكومي

```
إلزامي قبل أي نقاش رسمي:
☐ التسجيل كمورد في Etimad (منصة المشتريات الحكومية)
☐ النسخة العربية الكاملة من كل الوثائق
☐ NDA حكومي (نموذج الجهة)
☐ وثيقة compliance PDPL/SDAIA/NCA كاملة
☐ خيار Data residency داخل KSA مُوثَّق
☐ مسودة RFP response أو استجابة للمتطلبات التقنية
```

## 7. نقاط الإثبات للتأكيد عليها

```
✅ PDPL مُدمج من اليوم الأول — data subject rights كاملة
✅ audit trail بـ hash chain — لا يمكن تعديله
✅ خيار hosting داخل KSA (Enterprise)
✅ تصدير تقرير audit في دقيقة — جاهز للمراجع
✅ عربي أولاً — واجهة وتقارير بالعربية الكاملة
✅ مُصمَّم مع متطلبات NCA وSDAIA وSAMA
❌ لا ادعاء SAMA approved أو NCA certified حالياً
```

## 9. استراتيجية التسلل الصحيح

```
الخطأ الشائع: البدء بـ C-level مباشرة
الصحيح:
  الخطوة 1: ابنِ علاقة مع مستشار التحول الرقمي أو مدير IT
  الخطوة 2: ارسم خريطة stakeholders كاملة
  الخطوة 3: قدّم ورقة موجزة (2 صفحات) تُوضح توافق الحل مع متطلبات الجهة
  الخطوة 4: اطلب جلسة تقنية مع IT/Legal — لا عرض بيع مبكراً
  الخطوة 5: دع الـ champion الداخلي يُقدّم للقيادة
```

## 10. نطاق Pilot المقترح (Government)

```
النطاق:     workflow موافقة داخلي واحد (مثل: موافقات العقود مع الموردين)
الفريق:    فريق المشتريات + IT (5–15 مستخدم)
المحيط:    sandbox معزولة — بيانات تجريبية في المرحلة الأولى
المقاييس:
  - وقت إعداد تقرير audit لمراجعة ديوان المراقبة (قبل/بعد)
  - تغطية PDPL consent في workflow المختار
  - رأي مدير IT والقانوني في التوافق التقني
المدة:       8–12 أسبوع (أطول بسبب متطلبات موافقة الجهة)
الاشتراط:   نقطة تواصل رسمية (Focal Point) من الجهة
```
