# اختبارات الطبقة ١١ — التحويل

**المالك:** قائد التحويل في Dealix.

هذا مواصفة اختبار بصيغة markdown — لا يحتوي على كود. كل حالة لها معايير قبول صريحة.

## ت-١: نموذج النضج ينتج درجة

- **الخطوة:** تشغيل نموذج النضج على شركة بإشارات معروفة.
- **القبول:** تُنتَج درجة ٠–١٠٠ ومستوى من المستويات الخمسة. كل بُعد من الأبعاد الثمانية له قيمة.
- **الفشل:** أي درجة بدون إشارة مصدر، أو درجة خارج النطاق ٠–١٠٠.

## ت-٢: كل حالة استخدام مرتبطة بفرضية عائد

- **الخطوة:** فحص خريطة تحويل مُنتَجة.
- **القبول:** ١٠٠٪ من حالات الاستخدام مكتوبة كفرضية «إذا… نقدّر…».
- **الفشل:** أي حالة استخدام بعائد مذكور كحقيقة أو ضمان.

## ت-٣: الأدوار الأربعة مسمّاة قبل المرحلة

- **الخطوة:** محاولة بدء مرحلة بدون أحد الأدوار الأربعة.
- **القبول:** البوّابة ترفض البدء وتطلب الدور الناقص.
- **الفشل:** بدء مرحلة بدور غير مسمّى.

## ت-٤: حدود القرار مفروضة

- **الخطوة:** فحص سير عمل مُعاد تصميمه.
- **القبول:** كل إجراء خارجي يمرّ بموافقة بشرية صريحة؛ لا إرسال تلقائي.
- **الفشل:** أي إرسال خارجي أو تواصل بارد بدون موافقة.

## ت-٥: البوّابات تُجتاز بمعيار موثّق

- **الخطوة:** محاولة الانتقال إلى مرحلة بدون اكتمال معيار البوّابة.
- **القبول:** الانتقال محجوب حتى توثيق المعيار.
- **الفشل:** تجاوز بوّابة بدون معيار.

## ت-٦: القيمة مصنّفة في طبقاتها الأربع

- **الخطوة:** فحص سجل قيمة وتقرير شهري.
- **القبول:** كل قيمة مصنّفة: مُقدَّرة، مُلاحَظة، مُتحقَّقة، أو مؤكَّدة من العميل.
- **الفشل:** عرض قيمة مُقدَّرة على أنها مُتحقَّقة.

## ت-٧: خريطة ٣٠–٦٠–٩٠ تطابق مسار البيع

- **الخطوة:** مقارنة الخريطة بسُلَّم الخدمات.
- **القبول:** المراحل تطابق التدقيق ثم التجربة ثم التحويل ثم العقد الدوري.
- **الفشل:** مرحلة بلا رتبة بيع مقابلة.

## ت-٨: إجراء التراجع يعمل

- **الخطوة:** محاكاة مرحلة بقيمة سلبية.
- **القبول:** الانتقال يتوقف، يعود الفريق لسير عمل مستقر، يُسجَّل السبب، ولا تُحذف القيمة السابقة.
- **الفشل:** التقدّم رغم القيمة السلبية.

## ت-٩: ثنائية اللغة وصدق الإفصاح

- **الخطوة:** فحص كل ملف في `transformation/`.
- **القبول:** قسما H1 (`# العربية` ثم `# English`)، متطابقان، وكل ملف للعميل ينتهي بإفصاح القيمة التقديرية.
- **الفشل:** قسم ناقص أو إفصاح مفقود.

القيمة التقديرية ليست قيمة مُتحقَّقة.

# Layer 11 Tests — Transformation

**Owner:** Dealix Transformation Lead.

This is a markdown test specification — it contains no code. Each case has explicit acceptance criteria.

## T-1: Maturity model produces a score

- **Step:** Run the maturity model on a company with known signals.
- **Accept:** A 0-100 score and one of the five levels is produced. Each of the eight dimensions has a value.
- **Fail:** Any score without a source signal, or a score outside the 0-100 range.

## T-2: Every use case is tied to an ROI hypothesis

- **Step:** Inspect a produced transformation map.
- **Accept:** 100% of use cases are written as an "If... we estimate..." hypothesis.
- **Fail:** Any use case with a return stated as fact or guarantee.

## T-3: The four roles are named before the stage

- **Step:** Attempt to start a stage without one of the four roles.
- **Accept:** The gate refuses to start and requests the missing role.
- **Fail:** A stage starting with an unnamed role.

## T-4: Decision boundaries are enforced

- **Step:** Inspect a redesigned workflow.
- **Accept:** Every external action passes explicit human approval; no auto-send.
- **Fail:** Any external send or cold outreach without approval.

## T-5: Gates are passed with a documented criterion

- **Step:** Attempt to transition to a stage without the gate criterion complete.
- **Accept:** The transition is blocked until the criterion is documented.
- **Fail:** A gate bypassed without a criterion.

## T-6: Value is classified into its four tiers

- **Step:** Inspect a value ledger and a monthly report.
- **Accept:** Each value entry is classified: estimated, observed, verified, or client-confirmed.
- **Fail:** An estimated value presented as verified.

## T-7: The 30-60-90 roadmap mirrors the sales path

- **Step:** Compare the roadmap against the service ladder.
- **Accept:** Phases map to Audit, then Pilot, then Transformation, then Retainer.
- **Fail:** A phase with no corresponding sales rung.

## T-8: The rollback procedure works

- **Step:** Simulate a stage with negative value.
- **Accept:** The transition halts, the team returns to a stable workflow, the cause is recorded, and prior value is not deleted.
- **Fail:** Advancing despite negative value.

## T-9: Bilingual coverage and disclosure honesty

- **Step:** Inspect every file in `transformation/`.
- **Accept:** Two H1 sections (`# العربية` then `# English`), mirrored, and every customer-facing file ends with the estimated-value disclosure.
- **Fail:** A missing section or a missing disclosure.

Estimated value is not Verified value.
