# معمارية نظام التشغيل الأساسي — نموذج التركيب القابض

**الطبقة:** Holding · Compound Holding Model
**المالك:** رئيس Dealix Core (CTO)
**الحالة:** مسودة
**آخر مراجعة:** 2026-05-13
**النسخة الإنجليزية:** [CORE_OS_ARCHITECTURE.md](./CORE_OS_ARCHITECTURE.md)

## السياق
نظام دياليكس الأساسي هو **المنصّة الواحدة المشتركة** التي تستهلكها كل وحدة أعمال. وهو التفعيل البنيوي لقاعدة القابض: لا وحدة تبني بنية موازية. المعمارية هنا عرض طبقي عالي المستوى يتوافق مع `docs/BEAST_LEVEL_ARCHITECTURE.md` ومع موقف الموثوقية في `docs/BACKEND_RELIABILITY_HARDENING_PLAN.md`. قرارات منظومة الذكاء الاصطناعي في `docs/AI_STACK_DECISIONS.md`.

### الطبقة 1 — العرض
- مساحة العميل (واجهة متعدّدة المستأجرين).
- وحدة التحكّم الإدارية (للمجموعة والوحدات).
- التقارير ولوحات المؤشرات (للوحدات والمجلس).
- المصادقة، RBAC، حدود المستأجرين.

### الطبقة 2 — التطبيق
- وحدة تطبيق واحدة لكل BU.
- كل وحدة تكشف خدماتها عبر عقود نظام التشغيل نفسه.

### الطبقة 3 — التحكم بالذكاء الاصطناعي
- LLM Gateway مع توجيه النماذج (`docs/AI_MODEL_ROUTING_STRATEGY.md`).
- سجل البرومبت، سجل التقييمات.
- سجل التكلفة؛ ميزانيات لكل مستأجر.
- قياس برج التحكم بالذكاء (`docs/product/AI_CONTROL_TOWER.md`).

### الطبقة 4 — الحوكمة
- سجل السياسات (`docs/product/GOVERNANCE_AS_CODE.md`).
- محرّك مسار الموافقة.
- سجل تدقيق غير قابل للتعديل.
- ضوابط PDPL: الإقامة، الاحتفاظ، طلبات الأصحاب، الإخلال (`docs/ops/PDPL_BREACH_RUNBOOK.md`، `docs/ops/PDPL_RETENTION_POLICY.md`).

### الطبقة 5 — البيانات
- سجل المصادر (مجموعات بيانات، مالكون، حداثة).
- درجة جودة البيانات لكل مصدر.
- رسم الأصل (Lineage).
- كشف PII ووسمه.

### الطبقة 6 — التكامل
- موصِّلات (CRM، البريد، Slack، WhatsApp، الفوترة).
- ويب هوكس صادرة.
- ناقل أحداث المجال (انظر [`EVENT_DRIVEN_OPERATING_MODEL.md`](../product/EVENT_DRIVEN_OPERATING_MODEL.md)).
- ETL / Reverse-ETL إلى أنظمة العميل.

### الطبقة 7 — التعلم
- منظومة التقييمات (`docs/EVALS_RUNBOOK.md`).
- دائرة التغذية الراجعة: النتائج تُغذّي بيانات التقييم.
- ضبط البرومبت والنموذج.
- قواعد اختيار النموذج (`docs/AI_STACK_DECISIONS.md`).

## قواعد المعمارية

1. **مونوليث وحداتي أولاً.** الميكروسيرفس فقط حين يبرّره التوسّع المستقل أو الفرق أو العزل.
2. **مصادقة واحدة، تعدّد مستأجرين واحد، سجل تدقيق واحد.** لا تكرار لكل وحدة.
3. **الأحداث صنف أول.** كل تغيّر حالة ذي مغزى يُصدر حدث مجال.
4. **PDPL افتراضياً.** الإقامة والاحتفاظ افتراضاً متوافقَين؛ الاستثناءات تحتاج موافقة المستشار العام.
5. **عقد عام لكل طبقة.** لا وحدة تعبر الطبقات لتفصيل خاص.
6. **الكلفة والجودة مرصودتان لكل نداء.** Gateway يسجّل الكلفة ودرجة التقييم لكل برومبت.
7. **Capital Ledger قابل للكتابة من أي طبقة.** يجب تسجيل الأصول من أي مكان.

## عقود بين الطبقات

| من | إلى | العقد |
|---|---|---|
| العرض | التطبيق | جلسة مُصادَقة + سياق المستأجر |
| التطبيق | التحكم بالذكاء | `model_call` بمعرّف سياسة وبرومبت |
| التحكم بالذكاء | الحوكمة | فحص سياسة + قيد سجل تدقيق |
| التطبيق | البيانات | استعلام عبر سجل المصادر؛ احترام وسم PII |
| أي طبقة | التكامل | نشر حدث على الناقل |
| أي طبقة | التعلم | إيداع سجل تقييم |

## مضادّات النمط
- **تجاوز BU لطبقة الحوكمة** للسرعة → حجب نشر تلقائي.
- **نداء مزوّد LLM مباشرة** من التطبيق → ممنوع، Gateway فقط.
- **لوحات لكل وحدة** لا تقرأ من مخزن مقاييس Core OS → رفض في مراجعة الكود.

## الواجهات

## المقاييس
- **توفّر Core OS** — % توفّر لكل طبقة.
- **زمن الاستجابة p95** — على Gateway.
- **كلفة Gateway / نداء** — تكلفة الرموز لكل استدلال.
- **اكتمال سجل التدقيق** — % نداءات النموذج المسجَّلة.
- **معدّل نجاح التقييم** — % برومبتات السجل المجتازة.
- **تغطية وسم PII** — % حقول السجل ذات وسم PII.
