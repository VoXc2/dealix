# صفحة الحلول العامة — هندسة المعلومات (Solutions Page IA)

المبدأ الصارم: **لا نعرض 30 نظامًا في الموقع**. نعرض بساطة تُفهم في ثوانٍ:

```txt
5 أنظمة رئيسية
+ حلول حسب القطاع
+ تشخيص قصير يحدد نقطة البداية
```

المسار المقترح: `/ar/solutions` (والإنجليزي لاحقًا `/en/solutions`).
مصدر بيانات الصفحة (جاهز للربط): `data/business_need_intelligence/sector_solutions.yaml`.

---

## 1. أقسام الصفحة

1. **Hero**: جملة واحدة — "نقترح لك أول Sprint يناسب أكثر نقطة تعطّل عندك اليوم."
2. **الأنظمة الخمسة**: 5 بطاقات (Revenue · Executive Command · Follow-up Recovery ·
   WhatsApp Client · Proposal & Proof) — كل بطاقة سطر واحد.
3. **حلول حسب القطاع**: بطاقات قطاعات (وكالات، تدريب، عيادات، عقار، خدمات مهنية،
   توظيف، SaaS، لوجستيات) — كل بطاقة: احتياج بسطر + النظام الأساسي + أول Sprint.
4. **تشخيص قصير**: سؤال واحد ("أين يتعطّل العمل أكثر؟") → نقطة بداية (احتياج)،
   وليس حكمًا نهائيًا.
5. **CTA**: "ابدأ Sprint تشخيصي" بدون أي وعد بنتيجة.

---

## 2. لماذا هذا التبسيط؟

- العميل يرى شيئًا واحدًا بسيطًا، بينما يعرف Dealix داخليًا: الاحتياج، النظام،
  التخصيص، من نكلّم، ماذا نرسل، ماذا نسلّم.
- التعقيد (30 نظامًا + 20 سبرنت + متغيّرات تسليم) يبقى في طبقة
  `business_need_intelligence` الداخلية، لا في الواجهة.

---

## 3. الربط التقني (مقترح، غير منفّذ في هذا التغيير)

- إضافة مسار `/solutions` (و`/ar/solutions`) في `src/App.tsx`.
- صفحة `src/pages/Solutions.tsx` تقرأ `sector_solutions.yaml` (أو نسخة JSON
  مولّدة منه) وتعرض البطاقات والتشخيص.
- لا تُعرض الحقول الداخلية (الأنظمة المتخصصة، الإشارات، Need Fit Score) للعموم.

> ملاحظة تنفيذية: لم نُضِف صفحة React في هذا التغيير لتجنّب شحن واجهة غير مُتحقَّق
> من بنائها (حزم npm غير مثبّتة في هذه البيئة). البيانات والهندسة جاهزتان للربط
> فور توفّر بيئة بناء.
