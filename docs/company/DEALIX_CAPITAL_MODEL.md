## Context
Dealix is not a billable-hours consultancy and is not a generic AI shop.
Every client engagement must compound the company. This file defines the
five capital types that Dealix accumulates with every project and the
rule that converts cash work into permanent assets. It is the root of
Layer 1 and the reference that every other Layer-1 document inherits
from. Read alongside `docs/V14_COMPREHENSIVE_STRATEGIC_PLAN.md` and
`docs/DEALIX_OPERATING_CONSTITUTION.md`, which set the strategic and
governance boundaries this model operates inside.

## The five capitals

Dealix grows by accumulating five types of capital:

1. **Service Capital** — Reusable offers, scopes, checklists, delivery
   systems, and QA standards. Anything that lets the next sprint run
   faster, cleaner, and at higher quality than the last.
2. **Product Capital** — Internal tools, modules, APIs, dashboards, and
   automation that reduce delivery effort. Code and software that turn
   labor into leverage.
3. **Knowledge Capital** — Playbooks, sector insights, objection maps,
   data patterns, governance risks, and templates. The institutional
   memory of the firm.
4. **Trust Capital** — Proof packs, case studies, testimonials, audit
   logs, governance records, and safety reputation. Evidence that
   Dealix delivers, and that the market can verify.
5. **Market Capital** — Audience, partners, referrals, benchmarks, and
   authority content. The gravitational pull that brings the next deal
   without paid acquisition.

## The two-capital rule

Every project must increase at least two of the five capitals. A
project that produces only revenue and no capital is treated as
incomplete, even if the invoice is paid.

After every project the team asks six questions:

1. What service did we improve?
2. What product did we ship?
3. What knowledge did we add?
4. What proof did we build?
5. What market did we understand better?
6. What expansion did we unlock?

If three or more of these answers are blank, the project is logged as a
capital-debt event and routed to remediation in
`docs/ledgers/CAPITAL_LEDGER.md`.

## Why this model exists

A normal services firm scales linearly with headcount because each
project consumes time and returns only cash. Dealix is built so each
project consumes time and returns cash **plus** at least two reusable
assets. Over twelve to twenty-four months the asset base — not the
team size — becomes the primary source of leverage. This is what makes
the Academy, Platform, and Enterprise paths viable later in the
roadmap.

## How this connects to the other layers

- L1 Capital → defines what is accumulated.
- L2 Operating system → defines how it is accumulated.
- L3 Revenue engine → defines what we sell to fund accumulation.
- L4 Delivery and IP → defines how each delivery promotes a raw output
  into a graduated asset.
- L5 Market and growth → defines how the accumulated capital is
  converted into authority, audience, and inbound demand.

## Metrics
- Capitals-increased-per-project — count of the five capitals that
  measurably grew during the engagement; target ≥ 2.
- Capital-debt rate — share of closed projects in a quarter with fewer
  than two capitals increased; target ≤ 10%.
- Trust-asset velocity — number of new proof packs or case studies
  added per quarter; target ≥ 3.
- Productized-asset count — number of Stage-4 or higher assets in the
  IP Registry; target +1 per quarter.
