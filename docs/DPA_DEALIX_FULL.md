## 2. التعريفات · Definitions

| المصطلح | التعريف |
|---|---|
| **البيانات الشخصيّة** | أيّ معلومة، مهما كان مصدرها أو شكلها، من شأنها أن تؤدي إلى التعرّف على الفرد تحديداً، أو تجعل التعرّف عليه ممكناً (مطابق لـ PDPL Article 1) |
| **البيانات الحسّاسة** | البيانات التي يكون لها طبيعة تكشف عن الأصل العرقي أو القبلي، أو المعتقد الديني، أو الفكري، أو السياسي، أو البيانات الأمنية الجنائية، أو البيانات الحيويّة، أو البيانات الوراثيّة، أو الصحيّة، أو التي تدلّ على أن الفرد مجهول الأبوين أو أحدهما |
| **المعالجة** | أيّ عمليّة تتمّ على البيانات الشخصيّة، آليّاً أو غير آليّاً، شاملاً: الجمع، التسجيل، الحفظ، الفهرسة، التنظيم، التشكيل، التخزين، التعديل، التحديث، الاسترجاع، الاستخدام، الإفصاح، النقل، النشر، المشاركة، الربط، الحجب، الحذف، أو الإتلاف |
| **صاحب البيانات** | الفرد الذي تعود إليه البيانات الشخصيّة محلّ المعالجة |
| **النقل عبر الحدود** | إخراج البيانات الشخصيّة من المملكة، أو إتاحة إمكانيّة الوصول إليها من خارج المملكة |
| **الخرق الأمني** | أيّ حادثة تؤدّي إلى الإفصاح غير المشروع عن البيانات الشخصيّة، أو إلى تدمير، فقد، تغيير، أو وصول غير مصرّح به |
| **معالج فرعي (Sub-processor)** | أيّ طرف ثالث يكلّفه المعالج للقيام بجزء من معالجة البيانات الشخصيّة |

## 3. موضوع المعالجة ومدّتها · Subject Matter &amp; Duration

### 3.1 موضوع المعالجة (Purpose)

يقوم Dealix (المعالج) بمعالجة البيانات الشخصيّة المقدّمة من العميل (المتحكّم) للأغراض التالية حصراً:

- استقبال وتصنيف الـ leads عبر القنوات المتفق عليها (WhatsApp · email · CSV)
- توليد مسودّات ردود (Saudi-Arabic + English) — لا إرسال خارجي بدون موافقة المتحكّم
- توليد تقارير إثبات (Proof Pack) داخلية للعميل
- تشغيل لوحة Customer Portal للعرض على المتحكّم وموظّفيه المرخّصين

**خارج النطاق:** Dealix لا يعالج البيانات لأيّ غرض تسويقي ذاتي، ولا يبيع البيانات، ولا يستخدمها لتدريب نماذج LLM خاصّة بـ Dealix.

### 3.2 مدّة المعالجة

- **بدء المعالجة:** تاريخ توقيع DPA + استلام أوّل مدفوعات Sprint
- **انتهاء المعالجة:** انتهاء العقد + ٣٠ يوم احتفاظ + إعادة/حذف البيانات
- **الاحتفاظ القانوني:** الفواتير + سجلّات المدفوعات تحفظ ٦-١٠ سنوات (متطلّبات ZATCA السعوديّة)

### 3.3 طبيعة وأغراض المعالجة

| النشاط | البيانات المعنيّة | الأساس القانوني |
|---|---|---|
| استقبال leads | اسم العميل المحتمل، بريده، رقمه، شركته | تنفيذ عقد المعالج-المتحكّم |
| تصنيف الـ intent | محتوى رسالة العميل المحتمل | تنفيذ العقد |
| توليد مسودّات | محتوى الرسالة + سياق الشركة | تنفيذ العقد |
| تخزين Proof Events | موافقات، مدفوعات، نتائج (مع redaction) | تنفيذ العقد + التزام قانوني (ZATCA) |
| تقارير لـ Customer Portal | aggregates + عيّنات بعد redaction | تنفيذ العقد |

## 4. أنواع البيانات والأفراد المعنيّون · Data &amp; Subjects

### 4.1 أنواع البيانات الشخصيّة المعالجة

- **بيانات تعريف:** اسم، عنوان وظيفي، شركة
- **بيانات اتّصال:** بريد إلكتروني، رقم جوّال (E.164)، LinkedIn URL (إن قدّمها صاحبها)
- **بيانات سياقيّة:** قطاع، حجم الشركة، اهتمام، مرحلة في funnel
- **سجلّات تواصل:** نسخ من الرسائل المتبادلة (مع redaction للأسرار)
- **بيانات مدفوعات:** مرجع تحويل، تاريخ، مبلغ — **لا يخزّن Dealix أرقام بطاقات** (تمرّ عبر Moyasar مباشرة)

### 4.2 الأفراد المعنيّون (Data Subjects)

- العملاء المحتملون لـ المتحكّم (B2B prospects)
- جهات اتّصال داخل شركات الـ B2B prospects
- موظّفو المتحكّم الذين يستخدمون Customer Portal

### 4.3 البيانات الحسّاسة (Special categories)

**Dealix لا يعالج بيانات حسّاسة** كما عرّفها PDPL Article 1 (دينيّة، سياسيّة، صحيّة، إلخ). إن صادف العميل أنه أرسل بيانات حسّاسة عرضاً، يلتزم Dealix بحذفها فوراً وإخطار المتحكّم.

## 5. التزامات المعالج (Dealix) · Processor Obligations

### 5.1 الامتثال لتعليمات المتحكّم

- يعالج Dealix البيانات وفقاً لتعليمات المتحكّم الموثّقة فقط
- إذا اعتبر Dealix أن تعليمة معيّنة تخالف PDPL → يخطر المتحكّم خلال ٤٨ ساعة ويوقف التنفيذ حتى يحسم الأمر

### 5.2 السريّة

- جميع موظّفي ومتعاقدي Dealix الذين يصلون للبيانات يلتزمون بـ Non-Disclosure Agreement (NDA)
- التدريب على PDPL مسؤوليّة Dealix لكل من يصل للبيانات

### 5.3 الإجراءات الأمنيّة (Security Measures)

| المتطلّب | التطبيق في Dealix |
|---|---|
| تشفير البيانات أثناء النقل | TLS 1.3 على كل API endpoint |
| تشفير البيانات أثناء التخزين | Postgres encryption at rest (Railway managed) |
| التحكّم في الوصول | Role-based access; founder-only admin; service tokens |
| سجلّ التدقيق (Audit log) | كل عمليّة معالجة لها correlation_id غير قابل للتغيير |
| الحدود الأمنيّة (Hard gates) | ٨ بوّابات أمان مفروضة في الكود (لا تُطفأ من الإعدادات) |
| نسخ احتياطي | Railway managed daily backups + retention 30 days |
| اختبار الاسترجاع | quarterly drill (per `docs/SUPABASE_STAGING_RUNBOOK.md`) |
| مراقبة الاختراق | تنبيهات على failed-login + audit-log anomalies |

### 5.4 المعالجين الفرعيّين (Sub-processors)

- يحتفظ Dealix بقائمة محدّثة على `dealix.me/subprocessors.html`
- يخطر المتحكّم بأيّ تغيير قبل ١٤ يوم على الأقلّ
- يضمن تطبيق نفس مستوى الحماية على جميع المعالجين الفرعيّين عبر اتفاقيّات عقديّة موثّقة

**القائمة الحاليّة:**
- Railway Inc. (US) — استضافة backend
- Cloudflare Inc. (US) — DNS + CDN
- GitHub Inc. (US) — استضافة landing pages
- Anthropic PBC (US) — LLM inference (Claude)
- Groq Inc. (US) — LLM inference (Llama 3.3)
- Google LLC (US) — Gemini API (إن فُعّل)
- Hunter.io (FR) — enrichment (عند تفعيل HUNTER_API_KEY)
- Moyasar (KSA) — معالجة المدفوعات (عند تفعيل live)
- Meta Platforms (US/EU) — WhatsApp Business API (عند موافقة Meta)
- Supabase Inc. (US) — Postgres managed (عند الاستخدام)

تفاصيل النقل عبر الحدود في `docs/CROSS_BORDER_TRANSFER_ADDENDUM.md`.

### 5.5 المساعدة في حقوق صاحب البيانات

- يساعد Dealix المتحكّم في الاستجابة لطلبات الأفراد (DSAR) خلال ٧ أيّام عمل من إخطار المتحكّم
- التفاصيل في `docs/PDPL_DATA_SUBJECT_REQUEST_SOP.md`

### 5.6 إخطار الخرق الأمني

- يخطر Dealix المتحكّم خلال **٢٤ ساعة** من اكتشاف خرق أمني يطال بياناته
- المتحكّم مسؤول عن إخطار SDAIA + الأفراد المعنيّين خلال **٧٢ ساعة** (PDPL Article 21)
- التفاصيل في `docs/PDPL_BREACH_RESPONSE_PLAN.md`

### 5.7 تدقيق المتحكّم

- يحقّ للمتحكّم طلب تدقيق سنوي على إجراءات Dealix
- التدقيق يتمّ عبر زيارة عن بعد (Zoom screen-share + استعلامات وثائقيّة) لتقليل التكاليف
- في حالة الإخفاق المادّي → خطّة تصحيح خلال ٣٠ يوم

### 5.8 إعادة أو حذف البيانات

عند انتهاء العقد:
- المتحكّم يختار: **إعادة كل البيانات** (export JSON/CSV) أو **حذفها بالكامل** خلال ٣٠ يوم
- Dealix يقدّم **شهادة حذف** موقّعة بعد الحذف
- البيانات في النسخ الاحتياطيّة تحذف خلال دورة الـ retention الطبيعيّة (٣٠ يوم)

## 8. المسؤوليّة والتعويض · Liability &amp; Indemnification

### 8.1 الحدّ الأقصى للمسؤوليّة

- مسؤوليّة Dealix محدودة بـ **مجموع الرسوم المدفوعة في الـ ١٢ شهر السابقة** للحادثة
- تستثنى من هذا الحدّ: حالات الإهمال الجسيم + سوء النيّة + خرق العقد المتعمّد

### 8.2 الإعفاء المتبادل

كل طرف يعفي الآخر من المطالبات الناشئة عن:
- خرق الطرف الآخر لـ DPA
- فعل أو امتناع غير قانوني من الطرف الآخر

### 8.3 PDPL Compliance

- إذا فُرضت غرامة PDPL على المتحكّم بسبب فعل أو إهمال من Dealix، يتحمّل Dealix الغرامة (حتى الحدّ الأقصى للمسؤوليّة §8.1)
- إذا فُرضت بسبب فعل أو إهمال من المتحكّم، يتحمّلها المتحكّم

## Appendix A — Subprocessor Flow-Down Clauses

Each subprocessor in §5.4 has been required (via their own DPA / Terms of Service / Standard Contractual Clauses) to:

1. Process Customer Personal Data only per Controller's documented instructions
2. Maintain confidentiality
3. Implement appropriate security measures
4. Assist with data subject rights
5. Notify on security breach within 24-72 hours
6. Return or delete data on termination

Customer accepts that Dealix's contractual reach into US/EU subprocessors follows the standard SaaS-stack pattern (provider's own DPA + standard contractual clauses).
