# قابلية التراجع والتعويض — الحوكمة المؤسسية

**الطبقة:** L5 · الحوكمة المؤسسية
**المالك:** مالك التسليم
**الحالة:** مسودة
**آخر مراجعة:** 2026-05-13
**النسخة الإنجليزية:** [REVERSIBILITY_ROLLBACK.md](./REVERSIBILITY_ROLLBACK.md)

## السياق
أيّ إجراء بمساعدة الذكاء الاصطناعي قد يكون خاطئًا. سؤال الحوكمة ليس "هل قد يخطئ؟" بل "هل يمكننا التراجع أو التخفيف إن أخطأ؟". تحدد هذه الوثيقة التزامات ديليكس بقابلية التراجع والتعويض. ترتبط بـ `docs/ops/INCIDENT_RUNBOOK.md` و`docs/V6_OBSERVABILITY_AND_INCIDENT_RUNBOOK.md` و`docs/BACKEND_RELIABILITY_HARDENING_PLAN.md`. غايتها صون مبدأ: *لا إجراء بدون مسار تراجع أو تخفيف معروف*.

## المبدأ الأساسي
يجب تحديد التراجع أو التخفيف *قبل* التنفيذ. إن تعذّرا، يُحال الإجراء لإعادة التصميم أو يُرفَع لصنف ذي ضبط أعلى (اعتماد يدوي، قناة مقيدة، أو حظر كامل).

## الإجراءات الداخلية
لأي إجراء داخلي قاده الذكاء الاصطناعي (تحديث CRM، مهمة داخلية، كتابة بيانات):

- **حفظ الحالة السابقة.** يلتقط التشغيل حالة الكيان قبل الإجراء، مع نافذة احتفاظ كافية للتراجع.
- **إنشاء سجل تغيير.** قيد بكل عملية: الفاعل (الوكيل)، المعتمِد، الزمن، الفروقات، والربط بالتشغيلة.
- **تعيين مالك تراجع.** دور ديليكس مُسمّى مسؤول عن تنفيذ التراجع داخل النافذة.
- **إعلان نافذة التراجع.** نافذة متوقعة لكل صنف (دقائق إلى ساعات).

## الإجراءات الخارجية
لأي إجراء خارجي (بريد، رسالة، نشر، استدعاء API لطرف ثالث):

- **تسجيل الاعتماد.** كل إجراء صنف D يحمل رمز اعتماد صريحًا مرتبطًا بإنسان مُسمّى.
- **تأكيد المعاينة.** الحمولة النهائية وقائمة المستلمين تُؤكَّد لحظة الاعتماد لا لحظة المسودة.
- **مراجعة قائمة المستلمين.** لا إرسال لقائمة لم تُقارن بسجل عدم الاتصال والموافقات.
- **خطة تخفيف جاهزة.** إن أخطأ الإجراء يُنفَّذ التخفيف الموثّق (بريد تصحيح، اعتذار، متابعة).

## التقارير
للتقارير المُسلَّمة للعميل:

- **تاريخ الإصدارات.** كل تقرير مُسلَّم بإصدار وبصمة وأصل تشغيل.
- **عملية تصحيح.** تدفّق موثّق يُصدر v2 ويُبلغ العميل.
- **إبلاغ العميل.** إذا غيّر التصحيح حقيقة مؤثرة على القرار، يُبلَّغ العميل عبر مالك التسليم خلال SLA.

## قاعدة صارمة
**لا إجراء بلا تراجع أو تخفيف.** إن تعذّرا، لا يُعتمد الإجراء.

## دليل التراجع
تسرد بطاقة الوكيل دليل التراجع لكل صنف يلامسه:

| Action Class | Reversible? | Rollback Mechanism | Owner | Window |
|---|---|---|---|---|
| A | n/a | Versioned output replacement | QA Reviewer | n/a |
| B | Yes | Issue v2 with correction note | Delivery Owner | 48h |
| C | Yes | Restore previous state from change log | Technical Owner | 24h |
| D | Mitigation | Correction / retraction message | Delivery Owner | 24h |
| E | Blocked | — | — | — |

## الواجهات
| المدخلات | المخرجات | الملاك | الإيقاع |
|---|---|---|---|
| بطاقة الوكيل وتعريف سير العمل | خطة تراجع لكل صنف | التسليم + التقنية | لكل إصدار |
| تقارير الحوادث | تراجع أو تخفيف منفَّذ | مالك الحادثة | ضمن SLA |
| محفّزات إبلاغ العميل | تصحيح بإصدار جديد + اتصال | التسليم + نجاح العميل | حسب الحاجة |

## المقاييس
- Rollback-Plan Coverage — نسبة سير العمل ذي خطة تراجع موثّقة (الهدف 100%).
- Mean-Time-to-Rollback — دقائق من اكتشاف الحادثة إلى تنفيذ التراجع/التخفيف.
- Mitigation-After-Send Rate — نسبة الإجراءات الخاطئة التي نُفِّذ لها تخفيف داخل النافذة.
- Client-Notified-After-Correction Rate — نسبة التصحيحات الجوهرية التي أعقبها إبلاغ رسمي للعميل.
