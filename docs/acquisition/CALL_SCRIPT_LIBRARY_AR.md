# Call Script Library — مكتبة سكربتات الاتصال (AR)

> سكربتات اتصال جاهزة **لمتصل بشري فقط**. لا اتصال آلي. كل السكربتات تتجنب الوعود المضمونة وتربط المكالمة بإيميل حقيقي مُرسل.
> مرجع المصطلحات: `AGENTS.md`. الأنظمة الخمسة وزواياها في `AGENTS.md §2`.

---

## 1. الافتتاح العام (General Opener) — حرفيًا

يُستخدم هذا الافتتاح كما هو، مع استبدال [الاسم] و[النظام المناسب] والجزء بين الأقواس حسب النظام:

```txt
السلام عليكم، معك [الاسم] من Dealix. أرسلنا لكم رسالة مختصرة بخصوص [النظام المناسب]. الفكرة ليست بيع أداة، بل Sprint صغير يوضح أين يتعطل [الإيراد/المتابعة/واتساب/العروض/القرار] عندكم. سؤالي السريع: هل هذا الجانب عندكم ماشي بنظام واضح، أو يعتمد غالبًا على اجتهاد الفريق؟
```

> القاعدة: لا تبدأ المكالمة بـ "Re:" أو ادعاء سابق غير حقيقي. اربطها بالإيميل الذي أُرسل فعلًا. لا تَعِد بنتائج مضمونة.

## 2. أسئلة الاكتشاف لكل نظام (Per-System Discovery Questions) — حرفيًا

| النظام | أسئلة الاتصال |
|---|---|
| `revenue_os` | هل كل lead له next action؟ هل الإدارة ترى تقريرًا أسبوعيًا للفرص؟ |
| `executive_command_os` | ما أهم تقرير يومي عند الإدارة؟ هل التقرير يقول القرار التالي؟ |
| `followup_recovery_os` | هل عندكم متابعة موحدة بعد أول تواصل؟ هل الرسائل جاهزة حسب حالة العميل؟ |
| `whatsapp_client_os` | هل واتساب قناة رئيسية؟ هل الطلبات داخله مصنفة؟ متى يتم التصعيد لإنسان؟ |
| `proposal_proof_os` | كم يستغرق تجهيز العرض؟ هل يحتوي على scope وproof؟ ما أكثر اعتراض يتكرر؟ |

## 3. الافتتاح + الخطوة التالية لكل نظام

كل صف يجمع: الزاوية، جملة الافتتاح المخصّصة (الجزء المتغيّر من الافتتاح العام)، وأفضل خطوة تالية.

### 3.1 `revenue_os` — نظام تشغيل الإيرادات

- **الزاوية:** أين تضيع الفرص؟ من يحتاج متابعة؟ ما الخطوة التالية؟
- **افتتاح مخصّص:** "...بخصوص نظام تشغيل الإيرادات. الفكرة Sprint صغير يوضح أين يتعطل **الإيراد** عندكم. هل كل فرصة عندها خطوة تالية واضحة، أو غالبًا اجتهاد فريق؟"
- **أفضل خطوة تالية:** مكالمة 20 دقيقة + تجهيز Mini Proposal لبناء lead status model و next action (بعد موافقة founder).

### 3.2 `executive_command_os` — نظام القيادة التنفيذية

- **الزاوية:** التقارير كثيرة، لكن القرار اليومي غير واضح.
- **افتتاح مخصّص:** "...بخصوص نظام القيادة التنفيذية. الفكرة Sprint صغير يوضح أين يتعطل **القرار** اليومي عندكم. ما أهم تقرير يومي عندكم، وهل يقول لكم القرار التالي؟"
- **أفضل خطوة تالية:** عرض شكل Daily Command view في مكالمة قصيرة + Mini Proposal (بعد موافقة founder).

### 3.3 `followup_recovery_os` — نظام استرجاع المتابعات

- **الزاوية:** آخر متابعة لم تحدث قد تكون أغلى فرصة.
- **افتتاح مخصّص:** "...بخصوص نظام استرجاع المتابعات. الفكرة Sprint صغير يوضح أين تتعطل **المتابعة** عندكم. هل عندكم متابعة موحّدة بعد أول تواصل؟"
- **أفضل خطوة تالية:** مكالمة 20 دقيقة لفهم مصادر الاستفسارات + 7-day Follow-up Recovery Sprint (بعد موافقة founder).

### 3.4 `whatsapp_client_os` — نظام عملاء واتساب

- **الزاوية:** واتساب ليس فقط محادثات؛ يحتاج flows وaction cards وhandoff آمن.
- **افتتاح مخصّص:** "...بخصوص نظام عملاء واتساب. الفكرة Sprint صغير يوضح أين يتعطل **واتساب** عندكم. هل الطلبات داخله مصنّفة، ومتى يتم التصعيد لإنسان؟"
- **أفضل خطوة تالية:** عرض نموذج flow + action cards لأكثر طلب متكرر + Mini Proposal (بعد موافقة founder).

### 3.5 `proposal_proof_os` — نظام العروض والإثبات

- **الزاوية:** العرض المقنع يحتاج Proof وليس كلامًا أكثر.
- **افتتاح مخصّص:** "...بخصوص نظام العروض والإثبات. الفكرة Sprint صغير يوضح أين تتعطل **العروض** عندكم. كم يستغرق تجهيز العرض، وهل يحتوي scope وproof؟"
- **أفضل خطوة تالية:** إرسال نموذج عرض من صفحة واحدة لخدمة واحدة + Mini Proposal (بعد موافقة founder).

## 4. قواعد الاستخدام (Usage Rules)

```txt
- متصل بشري فقط — لا اتصال آلي.
- اربط المكالمة بالإيميل المُرسل فعلًا — لا Re:/Fwd: مزيّف.
- لا وعود مضمونة (لا "مضمون" / "نضاعف" / "guarantee").
- الألم يُصاغ كاحتمال إذا كان الدليل L0/L1 (غالبًا / قد يكون).
- استخدم خطًا منشورًا علنًا فقط — لا أرقام جوال شخصية.
- أي Mini Proposal بعد المكالمة يحتاج موافقة founder قبل الإرسال.
```

> بعد المكالمة: حدّث `status` في الـ Call Brief (`called` / `no_answer` / `interested` / `not_interested` / `callback_scheduled`) وسجّل الخطوة التالية. راجع `EMAIL_TO_CALL_HANDOFF_AR.md`.
