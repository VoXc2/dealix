# Company Research Policy — سياسة بحث الشركات (AR)

> كل بحث في Dealix يعتمد **مصادر عامة فقط**، يحترم suppression list وdo-not-contact، ويتوافق مع PDPL/SDAIA. لا أسرار ولا PII في أي مخرج بحثي.
> مرجع المصطلحات: `AGENTS.md §6`. مرجع الحوكمة: `company_os/governance/data_handling_checklist.md` و`company_os/governance/pdpl_checklist.md`.

---

## 1. الغرض (Purpose)

البحث يغذّي الـ Company Intelligence Pack. جودة الـ pack وأمانه القانوني يبدآن من **مصدر** المعلومة. هذه السياسة تحدد ما هو مسموح، وما هو ممنوع، وكيف نصوغ المعلومة حسب قوة مصدرها.

---

## 2. المصادر المسموحة (Allowed Sources)

```txt
- website (الموقع الرسمي للشركة)
- LinkedIn public page (الصفحة العامة)
- Google results (نتائج بحث عامة)
- directories (أدلة الأعمال العامة)
- job posts (إعلانات وظائف منشورة)
- news (أخبار عامة)
- public contact page (صفحة تواصل عامة)
- public social profiles (حسابات عامة)
```

| المصدر | ما نستخرجه عادة | مستوى الدليل الأرجح |
|---|---|---|
| website | القطاع، الخدمات، قنوات تواصل عامة، إشارات (واتساب ظاهر، برامج متعددة) | L1 |
| public contact page | قنوات تواصل عامة (نموذج تواصل، خط عمل منشور) | L1 |
| LinkedIn public page | حجم تقريبي، أدوار عامة منشورة | L1–L2 |
| job posts | إشارات نمو/فجوات (مثلاً توظيف متابعة مبيعات) | L2 |
| news | أحداث عامة (توسّع، إطلاق) | L2 |
| Google / directories | تأكيد عبر عدة مصادر متوافقة | L3 |

---

## 3. المصادر الممنوعة (Forbidden Sources)

```txt
- private data (أي بيانات خاصة غير منشورة)
- scraping مخالف (انتهاك شروط الخدمة)
- purchased lists (قوائم مشتراة)
- leaked databases (قواعد بيانات مسرّبة)
- unverified emails (إيميلات غير موثّقة)
```

> أي pack مبني على مصدر ممنوع **مرفوض** ولا يُحتسب send-ready. لا أرقام جوال شخصية (نمط 05XXXXXXXX)، لا هويات وطنية، لا مفاتيح/أسرار.

## 4. مستويات الدليل L0–L4 + قاعدة الصياغة

| level | المعنى | قاعدة الصياغة |
|---|---|---|
| `L0` | تخمين قطاعي | **احتمال صريح**: "في هذا النوع من الشركات غالبًا…" |
| `L1` | موقع الشركة | **احتمال**: "غالبًا" / "قد يكون"، بناءً على ما يظهر. |
| `L2` | صفحة خدمة/وظيفة/خبر عام | إسناد لمصدر عام محدد؛ تبقى `likely_*` احتمالية. |
| `L3` | عدة مصادر عامة متوافقة | ثقة أعلى، بلا ادعاء يقين مطلق. |
| `L4` | بيانات مقدّمة من الشركة | حقائق مؤكدة من العميل نفسه. |

**القاعدة الصارمة:** أي معلومة عند `L0` أو `L1` تُصاغ كاحتمال (`غالبًا` / `قد يكون` / `في هذا النوع من الشركات`) ولا تُصاغ كحقيقة مؤكدة. الحقول التي تبدأ بـ `likely_` احتمالية بطبيعتها.

### مثال على فرق الصياغة

```txt
❌ خطأ (L1 مصاغ كحقيقة):  الشركة تخسر 40% من استفساراتها بسبب ضعف المتابعة.
✅ صحيح (L1 كاحتمال):     في هذا النوع من شركات التدريب، غالبًا يضيع جزء من الاستفسارات
                          بسبب متابعة غير منظمة بعد أول تواصل.
```

## 5. احترام Suppression List وdo-not-contact

```txt
- قبل أي بحث/تواصل: تحقق من suppression list.
- إذا الشركة في القائمة أو do_not_contact = true → لا بحث موجّه للتواصل، ولا إرسال.
- القرار نهائي ولا يُتجاوز بجودة الـ pack.
```

## 6. التوافق مع PDPL / SDAIA

البحث والتواصل يتبعان متطلبات PDPL (هيئة SDAIA). المرجع التشغيلي: `company_os/governance/data_handling_checklist.md`.

| المبدأ | كيف نطبّقه في البحث |
|---|---|
| Lawful basis / الأساس النظامي | مصادر عامة فقط؛ بيانات العميل بموافقته (اتفاق الخدمة). |
| Purpose limitation / تحديد الغرض | البحث لغرض revenue intelligence فقط. |
| Data minimization / تقليل البيانات | نجمع ما يلزم فقط؛ لا PII غير ضروري. |
| Retention / الاحتفاظ | بيانات العميل تُحذف بعد 90 يومًا (افتراضي). |
| Security / الأمان | لا paste لبيانات العميل في أدوات عامة؛ تخزين آمن. |
| Data subject rights / حقوق صاحب البيانات | احترام طلبات الحذف وdo-not-contact. |

## 7. لا أسرار ولا PII في المخرجات البحثية

```txt
- لا API keys أو أسرار في أي pack/report/log.
- لا أرقام جوال سعودية شخصية (05XXXXXXXX).
- لا هويات وطنية أو بيانات حساسة.
- قنوات التواصل عامة فقط (صفحة تواصل، خط عمل منشور، حساب عام).
- البيانات في data/**/*.jsonl عيّنات تركيبية (synthetic) بأسماء وهمية.
```

- المرجع التشغيلي: `company_os/governance/data_handling_checklist.md`, `company_os/governance/pdpl_checklist.md`.
- الفاحص: `scripts/acquisition_delivery_check.py` (`npm run os:check`).
