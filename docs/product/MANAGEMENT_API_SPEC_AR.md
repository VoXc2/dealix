# مواصفة Management API — نموذج تشغيل القدرات

**الطبقة:** L2 · نموذج تشغيل القدرات
**المالك:** قائد المنصة
**الحالة:** مسودة
**آخر مراجعة:** 2026-05-13
**النسخة الإنجليزية:** [MANAGEMENT_API_SPEC.md](./MANAGEMENT_API_SPEC.md)

## السياق
تُدير Dealix أعمالها كبرامج. قرارات اختيار العميل والتسليم والحوكمة
والتسعير لا تُترك للجداول — بل تظهر عبر Management API. هذه المواصفة
هي العقد خلف خارطة نظام التشغيل في `OPERATING_SYSTEM_MAP.md`، وتُكمِّل
`docs/BACKEND_RELIABILITY_HARDENING_PLAN.md`، وتُوسِّع الأساس في
`docs/API_REFERENCE.md`، وتجعل الإدارة نفسها برمجيات، انسجامًا مع
`docs/BEAST_LEVEL_ARCHITECTURE.md`.

## مبادئ التصميم
- **غرض واحد لكل endpoint.** كل endpoint يملك قرارًا أو قراءة واحدة.
- **كتابات Idempotent.** كل POST يقبل ترويسة `Idempotency-Key`.
- **قابلة للتدقيق بشكل افتراضي.** كل كتابة تُصدر حدث حوكمة وحدث إثبات
  عند اللزوم.
- **ACL في كل استدعاء.** مطابقة الصلاحيات تُفرض عبر البوابة لا عبر
  الـ endpoint.
- **عقود مُصدَّرة.** بادئة `/v1/...` ومهلة إلغاء ≥ 90 يومًا.

## Endpoints (مستقبلية)
مُجمَّعة حسب طبقة خارطة نظام التشغيل.

### الجاهزية
- `GET /readiness/company` — بطاقة درجات جاهزية العميل العامة.
- `GET /readiness/services` — الجاهزية لكل خدمة قابلة للاستهلاك.
- `POST /readiness/service-score` — تقديم/تحديث درجة جاهزية من
  تشخيص.

### الطلبات
- `POST /requests` — إنشاء طلب وارد جديد.
- `GET /requests` — قائمة وتصفية الطلبات.
- `POST /requests/{id}/decision` — تسجيل قرار قابلية البيع/النطاق.

### العملاء
- `POST /clients` — إنشاء عميل + مساحة عمل.
- `GET /clients/{id}/health` — درجة صحة العميل واتجاهها.
- `GET /clients/{id}/capability-roadmap` — مستويات القدرة الحالية
  والمستهدفة مع Sprints المخطَّطة.

### الحوكمة
- `POST /governance/check` — تشغيل فحوصات وقت التشغيل لإجراء ذكاء.
- `POST /governance/approval` — تقديم/حسم طلب موافقة.
- `GET /governance/events` — استعلام أحداث الحوكمة للتدقيق.

### الجودة
- `POST /quality/score` — تقديم درجة مراجعة جودة.
- `GET /quality/reviews` — قائمة مراجعات الجودة لمشروع.

### الإثبات
- `POST /proof-pack/generate` — توليد حزمة إثبات التسليم.
- `GET /proof-ledger` — قراءة قيود سجل الإثبات.

### التعلم
- `POST /post-project-review` — تسجيل مراجعة ما بعد المشروع.
- `GET /feature-candidates` — قائمة الميزات المرشَّحة.

## ترويسات شاملة
- `Authorization` — bearer token من SSO.
- `X-Workspace-Id` — سياق مساحة العمل.
- `Idempotency-Key` — مطلوبة في كل POST.
- `X-Correlation-Id` — تُمرَّر إلى السجلات والمراقبة.

## نموذج الأخطاء
- `400` — مدخل غير صالح، مع `code` و `field` و `message`.
- `401` / `403` — فشل مصادقة / تصريح.
- `409` — تعارض idempotency.
- `422` — خرق قاعدة أعمال، مع `rule_id`.
- `5xx` — داخلي؛ يظهر في المراقبة.

## الإصدار والإلغاء
- التغييرات الكاسرة تُشحَن خلف إصدار مسار جديد.
- إشعارات الإلغاء تتضمن تاريخ الغروب ورابط الترحيل.
- كل إصدار يذكر الطبقة المتأثرة من خارطة نظام التشغيل.

## المقاييس
- **توفر API** — نسبة الطلبات الناجحة شهريًا (الهدف ≥ 99.5%).
- **P95 Latency** — الوقت عند المئوية 95 لكل مجموعة endpoint (الهدف ≤
  500 ms لطلبات GET، ≤ 1.5 s لطلبات الحوكمة).
- **معدل تعارض idempotency** — نسبة POSTs المُعيدة 409 (الهدف ≤ 1%).
- **اكتمال التدقيق** — نسبة استدعاءات الكتابة المطابِقة لحدث حوكمة
  (الهدف = 100%).
