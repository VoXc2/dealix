# تصميم انعكاس الصلاحيات — الحوكمة المؤسسية

**الطبقة:** L5 · الحوكمة المؤسسية
**المالك:** المالك التقني
**الحالة:** مسودة
**آخر مراجعة:** 2026-05-13
**النسخة الإنجليزية:** [PERMISSION_MIRRORING_DESIGN.md](./PERMISSION_MIRRORING_DESIGN.md)

## السياق
أكثر الطرق شيوعًا لكسر أمن المؤسسة هي أن يتصرف الذكاء الاصطناعي كمستخدم مميّز بينما المستخدم الطالب ليس كذلك. تمنع ديليكس هذا عبر انعكاس الصلاحيات: لا يستطيع الوكيل الوصول أو التصرف خارج ما يحق للمستخدم الطالب. تحدد هذه الوثيقة تصميم انعكاس الصلاحيات داخل ديليكس، بما يتوافق مع `docs/DPA_DEALIX_FULL.md` و`docs/DATA_RETENTION_POLICY.md` والموضع المعماري في `docs/BEAST_LEVEL_ARCHITECTURE.md`.

## المبدأ
**لا يصل الذكاء الاصطناعي ولا يتصرف إلا فيما يُسمح للمستخدم الطالب.** لا يرث الوكيل صلاحيات أعلى من صاحب الطلب. الوكيل ليس بابًا خلفيًا لمساحة العمل.

## الفحوصات اللازمة
يجب أن تجتاز كل تشغيلة قبل التنفيذ:

- **هوية المستخدم** — المستخدم مُعرَّف وحاضر.
- **مساحة العمل** — للمستخدم وصول، والوكيل مسموح بالعمل فيها.
- **الدور** — دور المستخدم مُسجَّل وكافٍ للقدرة المطلوبة.
- **صلاحيات البيانات** — نطاق بيانات الوكيل الفعلي = تقاطع صلاحيات الوكيل وصلاحيات المستخدم.
- **صلاحيات الأدوات** — قائمة الأدوات الفعلية = تقاطع كذلك.
- **صلاحيات الإجراءات** — صنف الإجراء المطلوب مسموح لدور المستخدم وسياسة المساحة.
- **متطلب الاعتماد** — إن استلزم الصنف اعتمادًا يُفحص الرمز ويرتبط بمعتمِد مُسمّى.

## التنفيذ
يُهيّأ كل تشغيل بسياق مكتوب:

```
{
  user_id,
  workspace_id,
  role,
  allowed_source_ids,     // intersect with agent.data_access
  allowed_tool_ids,       // intersect with agent.tool_access
  allowed_action_classes  // intersect with agent.action_permissions
}
```

ترفض طبقة الحوكمة وقت التشغيل أيّ استدعاء يتعدى التقاطعات. تُسجَّل الرفوضات مع السبب وتُعرض للمستخدم بشرح مفهوم.

## واجهة برمجية مستقبلية
عند فتح خدمة الحوكمة لإنفاذ من جانب العميل:

```
POST /governance/permission-check
{
  "user_id": "...",
  "workspace_id": "...",
  "agent_id": "...",
  "requested_sources": ["..."],
  "requested_action": "ClassC.update_crm_stage"
}
```

الاستجابة: `allowed` مع معرّف تدقيق، أو `requires_approval` مع دور المعتمِد، أو `denied` مع السبب ومرجع السياسة.

## حالات خاصة
- **التفويض.** إن فوّض المستخدم إجراءً للوكيل غير متزامن، يكون التفويض مُحدَّد المدة ومحصورًا في نطاق المستخدم وقت التفويض. توسعات لاحقة لا تنطبق رجعيًا.
- **حسابات الخدمة.** الحسابات المجدوَلة لها ملف صلاحيات خاص وتُعامَل كمبدأ مُسمّى، ليس "صلاحية مطلقة".
- **مسؤول المساحة.** يمنح الصلاحيات لأدوار لا لوكلاء؛ الوكلاء يرثون دائمًا عبر مستخدمين.

## الواجهات
| المدخلات | المخرجات | الملاك | الإيقاع |
|---|---|---|---|
| هوية مستخدم + بطاقة وكيل + سياسة مساحة | سياق تشغيل بنطاقات متقاطعة | المالك التقني | لكل تشغيل |
| استدعاء API التحقق | allowed / requires_approval / denied | خدمة الحوكمة | لكل طلب |
| سجلات الرفض | مراجعة مجلس عند التكرار | الحوكمة | شهري |

## المقاييس
- Mirroring Compliance — نسبة التشغيلات التي مرّت عبر دالة التحقق (الهدف 100%).
- Over-Scope Attempts — عدد التشغيلات المحجوبة لتجاوز النطاق حسب الوكيل.
- Approval-Required Latency — وسيط الثواني من محاولة الإجراء إلى قرار الاعتماد.
- Audit-ID Coverage — نسبة التشغيلات المسموحة التي تلقّت معرّف تدقيق مربوطًا بأصل التشغيل.
