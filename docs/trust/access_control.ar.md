# التحكّم بالوصول والهويّة في ديليكس

## 1. السياق

تشرح هذه الوثيقة كيفيّة تعامل ديليكس مع إدارة الهويّة والوصول (IAM) لعملاء المؤسّسات السعوديّة، بما يتماشى مع متطلّبات إطار الأمن السيبرانيّ لمؤسّسة النقد (SAMA CSF) وضوابط الأمن السيبرانيّ الأساسيّة (NCA ECC-1:2018).

## 2. الجمهور

- فِرَق الأمن وإدارة الهويّة لدى العميل.
- المدقّقون.
- فِرَق المشتريات.
- فِرَق ديليكس الداخليّة (الهندسة، الدعم، تمكين الشركاء).

## 3. القرارات والمحتوى

### 3.1 اتّحاد الهويّة

تدعم ديليكس SAML 2.0 وOpenID Connect (OIDC). تمّ التحقّق من التكامل مع:

- Microsoft Entra ID (Azure AD)
- Okta
- Ping Identity
- Google Workspace
- JumpCloud
- ADFS داخل بنية العميل

دعم SCIM 2.0 للتزويد التلقائيّ وإلغاء التزويد، ومعيَّن المجموعات إلى الأدوار قابل للتهيئة لكلّ مستأجر.

### 3.2 المصادقة متعدّدة العوامل (MFA)

- إلزاميّة افتراضيًّا لجميع الأدوار الإداريّة.
- العوامل المدعومة: TOTP، WebAuthn/FIDO2، تطبيقات المصادقة، SMS (احتياطيّ فقط).
- يمكن للعميل تحديد حدّ أدنى من قوّة العامل لكلّ دور.

### 3.3 التحكّم بالأدوار (RBAC)

| الدور | الغرض |
|---|---|
| مالك المستأجر | تحكّم إداريّ كامل |
| مسؤول المستأجر | إدارة المستخدمين والأدوار والتكاملات والفوترة |
| مسؤول مساحة العمل | تهيئة المحتوى داخل مساحة العمل |
| مستخدم مبيعات | قراءة وكتابة في المنطقة المخصَّصة |
| مدقّق للقراءة فقط | قراءة على مستوى المستأجر للتدقيق |
| حساب خدمة API | وصول برمجيّ بمفاتيح محدَّدة النطاق |
| شريك تنفيذ | دور تنفيذ مع تفويض موثَّق |

تُنشر مصفوفة الصلاحيّات في وحدة تحكّم المسؤول.

### 3.4 عزل المستأجرين

- عزل على مستوى التطبيق: نطاق إلزاميّ للمستأجر في كلّ استعلام.
- عزل على مستوى البيانات: سياسات أمان مستوى الصفّ (RLS).
- عزل تشغيليّ: لا يملك أيّ مهندس وصولًا متبادلًا بين المستأجرين.
- خيار النشر أحاديّ المستأجر متاح في خطّة المؤسّسات.

### 3.5 أدوات مسؤول العميل

- إدارة المستخدمين والأدوار بسجلّ تدقيق.
- تهيئة SSO/SCIM الذاتيّة.
- سياسة MFA لكلّ دور.
- سياسات الجلسة (مهلة قصوى، خمول، إعادة مصادقة للأفعال الحسّاسة).
- إصدار مفاتيح API بنطاق محدَّد.
- قائمة بيضاء لعناوين IP لخطّة المؤسّسات.
- عارض سجلّات التدقيق مع تصدير CSV.

### 3.6 إدارة الوصول المتميِّز (داخليًّا)

- لا يملك مهندسو ديليكس وصولًا دائمًا إلى بيانات الإنتاج.
- وصول في الوقت المناسب (JIT): تذكرة بمبرّر مرتبط بالعميل، موافقة من خارج خطّ الإبلاغ المباشر، منحة مؤقّتة (حدّ أقصى 8 ساعات؛ 30 دقيقة للدعم الاعتياديّ)، إلغاء تلقائيّ، تسجيل جميع الأفعال.

### 3.7 سجلّات التدقيق

- تدفّق غير قابل للتعديل ومحميّ السلامة.
- **سنة نشطة + 7 سنوات أرشيف بارد.**
- التخزين البارد مشفَّر، والوصول مقتصر على فريق صغير معتمَد.
- تصدير محدود بالمستأجر للعملاء.

### 3.8 أمان الجلسة وواجهات البرمجة

- عمر الجلسة الافتراضيّ 12 ساعة، قابل للتهيئة.
- خمول 30 دقيقة افتراضيًّا.
- إعادة مصادقة عند الأفعال الحسّاسة.
- مفاتيح API ذات نطاق، قابلة للتدوير والإنهاء، مُجزَّأة عند التخزين.
- حدود معدّل لكلّ مستأجر مع تنبيهات شاذّة.

### 3.9 مفاتيح يديرها العميل (BYOK)

متاحة في خطّة المؤسّسات، مع إجراءات موثَّقة للتدوير والإبطال (التمزيق التشفيريّ).

### 3.10 أحداث دورة الحياة

- التهيئة: عبر SCIM أو SSO JIT.
- تغيير الدور: عبر وحدة التحكّم، مع تسجيل التدقيق.
- إلغاء التهيئة: SCIM يلغي خلال دقائق؛ يدويًّا خلال 4 ساعات عمل.
- التعليق/الإنهاء: نافذة تصدير ثمّ حذف لا رجعة فيه.

## 4. مؤشّرات الأداء

- 90% من عملاء المؤسّسات يستخدمون SSO خلال 90 يومًا.
- 100% من الوصول الداخليّ لبيانات العميل عبر JIT.
- توافر مصادقة ≥ 99.95%.

## 5. التبعيّات

- نشر ADR 0003 لعزل المستأجرين.
- تقوية موصِّل SCIM.
- وحدة تحكّم إداريّة ثنائيّة اللغة.
