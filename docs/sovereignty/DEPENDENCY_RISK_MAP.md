# Dependency Risk Map

Anything Dealix depends on must be measured and controlled. The Dependency Risk Map is the canonical inventory.

## 1. The map

| Dependency | Risk | Control |
| --- | --- | --- |
| Single LLM provider | Price / quality / policy shock | LLM Gateway + model router |
| Founder delivery | Bottleneck | Delivery OS + checklists |
| Custom projects | Agency trap | Productized offers |
| Weak data sources | Bad outputs | Source Passport |
| Unsafe automation | Trust loss | Governance Runtime |
| One sales channel | Growth fragility | Partners + content + academy |
| One client segment | Market fragility | Portfolio strategy |
| No proof | Weak retention | Proof Pack |
| No capital capture | No compounding | Capital Ledger |

## 2. The rule

> Every dependency must have a fallback or a control.

A dependency with neither is a constitutional violation.

## 3. Severity tiers

| Severity | Definition | Required response |
| --- | --- | --- |
| Critical | Loss of dependency stops revenue or breaks governance | Active fallback + monthly drill |
| High | Loss of dependency degrades a Business Unit | Active fallback + quarterly drill |
| Medium | Loss of dependency increases delivery friction | Documented workaround |
| Low | Loss of dependency is inconvenient | Acknowledged, no action |

## 4. Operating discipline

- The Intelligence OS reviews the map quarterly.
- Every BU lead signs off on their critical dependencies.
- Drills test fallback paths — no fallback exists until it has been used.
- A dependency added without a control row is rejected by code review.

## 5. The principle

> What you cannot replace, controls you.
