# معيار Dealix للتسليم ونظام الجودة

## 1. الفكرة

البيع وحده لا يكفي. ميزة Dealix الدفاعية هي **ثبات الجودة والتسليم لكل عميل**. هذه الوثيقة تحدد:
- معيار التسليم بثماني مراحل لكل مشروع.
- نظام الجودة بخمس بوابات قبل التسليم.
- درجة جودة المشروع (0–100) وحدّها الأدنى للتسليم.
- خمس مزايا تنافسية مفعّلة تشغيليًا.

## 2. مراحل التسليم الثماني

كل مشروع — Sprint / Pilot / Retainer / Enterprise — يمر بنفس المراحل:

```
Discover → Diagnose → Design → Build → Validate → Deliver → Prove → Expand
```

1. **Discover** — فهم الشركة، القنوات، الأدوات، أين يضيع الوقت والفلوس والمخاطر.
2. **Diagnose** — تسجيل جاهزية البيانات، نضج العمليات، أفضل حالات الاستخدام، تقدير ROI، Quick Wins.
3. **Design** — تصميم Workflow، Schema، Prompts، الموافقات، Dashboard، المخرجات، KPIs.
4. **Build** — تنفيذ بأقل تعقيد، إعادة استخدام وحدات OS، Human-in-the-loop، Audit Logs منذ اليوم 1.
5. **Validate** — اختبار الجودة، السلامة، الدقة، السرعة، الحالات الحدية، نبرة العربي/الإنجليزي، PII.
6. **Deliver** — تسليم Dashboard أو Workflow، تقرير تنفيذي، SOP، تدريب موثّق، Handoff Document.
7. **Prove** — قياس الأثر خلال 14 يومًا: ساعات موفّرة، فرص مؤهلة، Tickets مصنّفة، Pipeline، جودة بيانات.
8. **Expand** — فتح المحادثة التالية (Retainer / Workflow إضافي / عمود جديد / Enterprise).

## 3. بوابات الجودة الخمس

أي FAIL يوقف التسليم.

### Business QA
- هل المشكلة واضحة؟
- هل المخرج يهم الإدارة؟
- هل فيه KPI رقمي؟
- هل فيه Next Action؟
- هل فيه مسار توسّع؟

### Data QA
- هل كل مصدر بيانات موثّق؟
- هل التكرارات مُعالجة؟
- هل الحقول الناقصة موثّقة؟
- هل PII مكتشفة وأُخفيت عند اللزوم؟
- هل أساس PDPL (المادة 5) موثّق؟
- هل درجة جودة البيانات محسوبة ومُشاركة؟

### AI QA
- هل المخرجات دقيقة (مقارنة بمجموعة حقيقية)؟
- هل تم تجنّب الهلوسة (إجابات بمصادر)؟
- هل المصادر مُستشهدة؟
- هل نبرة العربي مناسبة للقطاع ومستوى المشتري؟
- هل الحالات الحدية مختبَرة؟

### Compliance QA
- هل توجد ادعاءات مبالغ فيها؟
- هل التواصل البارد متوافق مع PDPL (13 إشعار، 14 موافقة)؟
- هل PII غائب عن التقارير؟
- هل الموافقة البشرية مُسجَّلة عند اللزوم؟
- هل سجل التدقيق كامل وقابل للاستعلام؟

### Delivery QA
- هل كل المخرجات في حزمة التسليم؟
- هل التقرير التنفيذي مفهوم لقارئ غير تقني؟
- هل العميل يعرف ماذا يفعل بعدها؟
- هل جلسة التسليم مُسجَّلة/مُكتملة؟
- هل عرض التجديد/الخطوة التالية مكتوب؟

## 4. درجة جودة المشروع (0–100)

الحد الأدنى للتسليم: **80**.

| المعيار | الوزن |
|---|---:|
| وضوح أثر الأعمال | 20 |
| جودة البيانات | 15 |
| جودة المخرجات (AR/EN) | 15 |
| سهولة استخدام العميل | 10 |
| الأمان والامتثال | 15 |
| قابلية التكرار كقالب منتج | 15 |
| قابلية الترقية إلى Retainer | 10 |
| **الإجمالي** | **100** |

أي مشروع أقل من 80 لا يُسلَّم حتى يُحسَّن.

## 5. المزايا التنافسية الخمس (مفعّلة تشغيليًا)

| الميزة | المعنى | كيف نُفعّلها |
|---|---|---|
| **Outcome-first** | كل خدمة بنتيجة رقمية | KPI لكل عرض في الكتالوج؛ النتيجة في Proof Ledger |
| **Saudi-localized** | عربي/إنجليزي، قطاعات، مدن، سياق | مخرجات ثنائية اللغة؛ Playbooks قطاعية؛ ZATCA/PDPL مدمجان |
| **Proof-backed** | كل مشروع يُنتج دليلًا | Proof Pack إلزامي في مرحلة 7؛ Proof Ledger مرجعي للمبيعات |
| **Governed AI** | مصادر، PII، موافقات، Audit | Governance OS يحرس كل إجراء؛ Decision Passport مختوم |
| **Productized delivery** | جودة ثابتة لكل عميل | Delivery OS + هذا النظام؛ لا تخصيص بدون 3 حالات إعادة استخدام |

## 6. Playbooks القطاعية

لكل قطاع (BFSI / تجزئة / صحة / تعليم / عقار / لوجستيات / مطاعم / مقاولات / عيادات / خدمات مهنية):
- 5 آلام رئيسية.
- ICP دقيق.
- الخدمات الموصى بها.
- 5 اعتراضات + ردود.
- قواعد تقييم خاصة.
- قوالب تواصل.
- KPIs.
- ملاحظات امتثال قطاعية (SAMA للبنوك، SFDA للصحة، نوبكو للمشتريات).

المرحلة 1: BFSI، تجزئة، صحة. المرحلة 2: تعليم، عقار، لوجستيات. المرحلة 3: الباقي.

## 7. Proof Ledger

سجل مركزي يحوي الأدلة قبل/بعد لكل مشروع:
- عدد السجلات المنظفة.
- ساعات موفرة أسبوعيًا.
- رسائل صيغت/أُرسلت/رُد عليها.
- Tickets مصنّفة.
- قيمة Pipeline.
- تقييم العميل.

المبيعات تستشهد بإدخالات Ledger في العروض — تحول تسليم Dealix نفسه إلى أقوى مرجع.

## 8. مؤشرات النجاح

- متوسط Quality Score: ≥ 85.
- نسبة المشاريع التي تمر بـ 5 بوابات من المحاولة الأولى: ≥ 70%.
- نسبة إكمال مرحلة 7 (Prove) خلال 14 يومًا: ≥ 95%.
- نسبة مرحلة 8 (Expand) التي تفتح محادثة لاحقة: 100%.
- زمن "Diagnose → Deliver" للـ Sprints: ≤ المخطط (7/10/14/21/30 يومًا).
