# نموذج إشارة المشاريع — الذكاء · الدماغ التشغيلي

**الطبقة:** Intelligence · الدماغ التشغيلي
**المالك:** الرئيس التنفيذي
**الحالة:** مسودة
**آخر مراجعة:** 2026-05-13
**النسخة الإنجليزية:** [VENTURE_SIGNAL_MODEL.md](./VENTURE_SIGNAL_MODEL.md)

## السياق
يُقيِّم نموذج إشارة المشاريع خطوط الخدمة والوحدات والمرشّحين الجدد
للترقية إلى مشاريع مستقلّة. يُجيب عن السؤال الذي يُحدّد ما إذا كانت Dealix
ستظل شركة واحدة متوقّفة أم قابضة تُنتج مشاريع: **هل هذه الوحدة جاهزة
للوقوف وحدها؟** يدمج النموذج جذب العملاء المدفوعين وعمق الاحتفاظ والقابلية
للتكرار ونضج المنتج والـPlaybook والهامش وجاهزية المالك في رقم واحد، ثم
يصنّفه في أربعة مخرجات. راجع `docs/V14_COMPREHENSIVE_STRATEGIC_PLAN.md`
لموقف القابضة و`docs/DEALIX_V3_AUTONOMOUS_REVENUE_OS.md` لمنصّة الإيراد
المُستضِيفة للمرشّحين.

## صيغة نتيجة جاهزية Venture
```
Venture Score =
  paid_clients * 0.15
+ retainers * 0.20
+ repeatability * 0.20
+ product_module_usage * 0.15
+ playbook_maturity * 0.10
+ margin * 0.10
+ owner_readiness * 0.10
```

كل مدخل على مقياس 0-100، مُعَيَّر كما يلي.

### تعريفات المدخلات
- **paid_clients** — عدد العملاء المدفوعين المميَّزين في آخر 12 شهراً،
  مُعَيَّر: 0 = 0، 1 = 30، 3 = 60، 5 = 80، 10+ = 100.
- **retainers** — احتفاظات شهرية نشطة، مُعَيَّرة: 0 = 0، 1 = 40، 2 = 70،
  3 = 85، 5+ = 100.
- **repeatability** — نسبة سير العمل المُقَوْلَب/المُمَنتج (من Capital
  Ledger).
- **product_module_usage** — % المشاريع في الخط التي تستخدم وحدة مشتركة
  (من Productization Ledger).
- **playbook_maturity** — تغطية Playbooks العرض والاستقبال والتسليم
  والإثبات والتسعير والـICP، كل منها بوزن 1/6 (0-100).
- **margin** — نسبة الهامش الإجمالي للوحدة، بحدّ أعلى 100.
- **owner_readiness** — وجود مالك مساءل بسلطة وفهم للـP&L: ثنائي 0 أو
  100، مضروب في 1.0 / 0.5 / 0.0 لحالة مكتبة الإثبات (ناضجة / جزئية / لا).

## نطاقات القرار
| النتيجة | النطاق | الفعل الافتراضي |
|---|---|---|
| 85+ | مرشّح Venture | فتح مراجعة الترقية؛ مدير، P&L، علامة مُحدَّدون |
| 70-84 | وحدة عمل | تشغيلها داخل Dealix بمالك وميزانية مُخصَّصَين |
| 55-69 | خط خدمة | الإبقاء في التنظيم الحالي مع دفع التمنتج |
| < 55 | إبقاء داخل الخدمات الأساسية | لا معاملة مستقلّة |

## المثال "أ" — Dealix Revenue (مرشّح Venture)
خط بإحصاءات:
- 6 عملاء مدفوعين خلال 12 شهراً (paid_clients = 85)
- احتفاظان نشطان (retainers = 70)
- repeatability = 80 (سير عمل Revenue OS مُقَوْلَب)
- product_module_usage = 75 (وحدة Revenue OS في 4/5 مشاريع)
- playbook_maturity = 85 (Playbook B2B ناضج عبر الأبعاد الستة)
- margin = 65
- owner_readiness = 90 (مالك موجود، مكتبة إثبات ناضجة)

الحساب:
```
Venture Score =
   85 * 0.15  = 12.75
 + 70 * 0.20  = 14.0
 + 80 * 0.20  = 16.0
 + 75 * 0.15  = 11.25
 + 85 * 0.10  =  8.5
 + 65 * 0.10  =  6.5
 + 90 * 0.10  =  9.0
 -----------------
            = 78.0
```
الحكم: **وحدة عمل (طرف عالٍ)** — عند 78 هي وحدة عمل قوية. العبور إلى
مرشّح Venture (85+) يستلزم رفع العملاء نحو 10، والاحتفاظات نحو 3،
والهامش فوق 70. وفق المصدر، Dealix Revenue يتأهّل venture عند "5+ عملاء،
2+ احتفاظ، وحدة Revenue OS مستخدمة، Playbook B2B ناضج، هامش صحّي، مالك
موجود، مكتبة إثبات موجودة" — هذه الشروط مجتمعة ترفع النتيجة فوق 85.

## المثال "ب" — مختبر التقييم الداخلي (إبقاء داخلي)
- 0 عملاء مدفوعين (paid_clients = 0)
- 0 احتفاظات (retainers = 0)
- repeatability = 60 (بعض حزم التقييم قابلة لإعادة الاستخدام)
- product_module_usage = 40 (مستخدمة في مشروعين)
- playbook_maturity = 30 (Playbook التقييم فقط)
- margin = 0 (مركز تكلفة داخلي)
- owner_readiness = 50 (مالك موجود، لا مكتبة إثبات خارجية)

الحساب:
```
Venture Score =
    0 * 0.15  =  0.0
 +  0 * 0.20  =  0.0
 + 60 * 0.20  = 12.0
 + 40 * 0.15  =  6.0
 + 30 * 0.10  =  3.0
 +  0 * 0.10  =  0.0
 + 50 * 0.10  =  5.0
 -----------------
            = 26.0
```
الحكم: **إبقاء داخل الخدمات الأساسية** — مختبر التقييم بنية تحتية لا
venture. يُغذّي `docs/EVALS_RUNBOOK.md` ومقاييس AI Run، ولا يُغلَّف
للترقية أبداً.

## مراجعة الترقية (85+)
عند بقاء وحدة عند 85+ لمدّة 30 يوماً:

1. يفتح الرئيس التنفيذي مراجعة الترقية.
2. يؤكّد المدير المالي هامش الوحدة واحتياجات رأس المال والمسار النقدي.
3. يؤكّد رئيس الحوكمة جاهزية PDPL والتعاقدية.
4. يؤكّد رئيس العلامة التسمية والهوية وعزل العلامة.
5. تُعاد جاهزية المالك بخطّة حوكمة مكتوبة.
6. تُعاد بوّابات `docs/governance/AUTONOMY_VALIDATION_GATES.md` و
   `AGENT_CONTROL_DOCTRINE.md` للكيان الجديد.

الناتج إما خطّة ترقية (كيان، قيادة، رأس مال، علامة، حوكمة) أو إبقاء مع
فجوات واضحة وموعد إعادة تقييم خلال 90 يوماً.

## إيقاع التقييم
- تراكم مستمر عبر Venture Signal Ledger.
- إعادة حساب شهرية لكل الوحدات المرشّحة.
- مراجعة إشارة المشاريع ربعية مع الرئيس التنفيذي.

## قواعد منع التلاعب
- لا تُقبل المدخلات بإقرار ذاتي. كل مدخل يُتتبَّع إلى قيد سجل.
- جاهزية المالك ثنائية على وجود المالك؛ يُرفض التلاعب بـ"مالك ظل" في
  المراجعة.
- يستخدم الهامش تكلفة الوحدة الكاملة، بما فيها العبء المنصّي والحوكمي
  المُحَمَّل.

## المقاييس
- **تغطية التقييم** — نسبة الوحدات المرشّحة بنتيجة حالية (الهدف: 100%).
- **إنتاجية الترقية** — عدد الوحدات المُرَقَّاة سنوياً (الهدف: ≥ 1).
- **معدّل الترقية الزائفة** — نسبة الوحدات المُرَقَّاة العائدة للأم خلال
  12 شهراً (الهدف: 0).
- **معدّل التتبّع التدقيقي** — نسبة مدخلات النتيجة المُتتبَّعَة لقيود
  السجلات (الهدف: 100%).
