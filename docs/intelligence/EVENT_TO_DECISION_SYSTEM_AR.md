# نظام الحدث إلى القرار — الذكاء · الدماغ التشغيلي

**الطبقة:** Intelligence · الدماغ التشغيلي
**المالك:** رئيس الديوان
**الحالة:** مسودة
**آخر مراجعة:** 2026-05-13
**النسخة الإنجليزية:** [EVENT_TO_DECISION_SYSTEM.md](./EVENT_TO_DECISION_SYSTEM.md)

## السياق
نظام الحدث إلى القرار هو خط الأنابيب الذي يُحوّل الأحداث التشغيلية الخام
إلى قرارات استراتيجية مُلزِمة. كل فعل ذي معنى داخل Dealix — تشغيل ذكاء،
موافقة، حدث إثبات، فوز بعقد احتفاظ — يجب أن يُصدر حدثاً مهيكلاً، يستقر في
سجل، يُسهم في مقياس، ويُحرّك قراراً عند تجاوز العتبات. يُعرّف هذا الملف
مفردات الأحداث، ومراحل خط الأنابيب، وأمثلة التدفّق. هو العقد التشغيلي بين
Core OS (الذي يُصدر الأحداث) وطبقة الذكاء (التي تستهلكها). راجع
`docs/V14_COMPREHENSIVE_STRATEGIC_PLAN.md` للموقف الاستراتيجي الذي يتيحه
هذا النظام، و`docs/AI_OBSERVABILITY_AND_EVALS.md` لعمود فقرات الرصد الذي
يحمل حمولات الأحداث.

## مراحل خط الأنابيب
```
+--------+    +-------+    +---------+    +--------+    +-------+
| الحدث  | -> | السجل | -> | المقياس | -> | القرار | -> | الفعل |
+--------+    +-------+    +---------+    +--------+    +-------+
  إصدار        إلحاق        حساب           عتبة         Core OS
```

1. **الإصدار** — يُنشر حدث JSON عبر ناقل الأحداث من قِبل كود Core OS أو
   الوكلاء أو البشر. كل حدث يحمل `event_type` و`project_id` و`client_id`
   و`created_at` وحمولة.
2. **الإلحاق** — يُكتب الحدث في سجل أو أكثر (راجع `LEDGER_ARCHITECTURE.md`).
   الكتابة إلحاقية فقط ومتوافقة مع PDPL.
3. **الحساب** — يُعيد محرّك المقاييس حساب المؤشرات المتأثّرة (راجع
   `METRICS_ENGINE.md`).
4. **العتبة** — تُقيّم قاعدة قرار المقياس مقابل نطاق، وتُطلق قراراً أو
   تصمت.
5. **الفعل** — يُرسَل القرار إلى ملاك Core OS (التسليم، الإيراد، المنتج،
   الحوكمة) للتنفيذ.

خط الأنابيب لاتزامني؛ التزامن الوحيد بين الإصدار والإلحاق. كل ما عداه
يُعاد حسابه على جدول (مستمر، ساعي، يومي، أسبوعي حسب المقياس).

## تصنيف الأحداث (حرفياً)
واحد وعشرون نوع حدث قانوني. أي نوع جديد يستلزم اقتراحاً مكتوباً لرئيس
الديوان وتحديث مخطّط السجل المقابل.

```
project_created
client_intake_completed
data_source_registered
dataset_uploaded
data_quality_scored
pii_detected
governance_checked
ai_run_completed
account_scored
draft_generated
approval_required
approval_granted
proof_event_created
report_delivered
capital_asset_created
feature_candidate_created
retainer_recommended
retainer_won
playbook_updated
partner_lead_created
venture_signal_detected
```

الأحداث خارج هذه القائمة ممنوع إصدارها إلى ناقل الذكاء. القياس التشغيلي
(السجلات، التتبّع، المترِك) يذهب إلى مكدّس الرصد لا إلى هنا.

## مظروف الحدث
جميع الأحداث تستخدم المظروف ذاته. يحدّد `event_type` الحقول المطلوبة في
الحمولة.

```json
{
  "event_type": "proof_event_created",
  "project_id": "PRJ-001",
  "client_id": "CL-001",
  "proof_type": "Revenue Value",
  "metric": "accounts_scored",
  "value": 50,
  "source": "Revenue Intelligence Report",
  "created_at": "2026-05-13T12:00:00Z"
}
```

قواعد الحقول:
- `event_type` — تطابق تام مع التصنيف أعلاه.
- `project_id` و`client_id` — مُعرّفات مُبهمة. لا تُرسل أسماء عملاء خام في
  الحمولة؛ يُحلّ الاسم عبر سجل المصادر.
- `created_at` — صيغة ISO 8601 بتوقيت UTC.
- ما يتجاوز المظروف خاص بنوع الحدث ويُحقَّق بمخطّط JSON في سجل التمنتج.

## تدفّقات نموذجية

### التدفّق "أ" — انفجار التكلفة إلى توجيه النموذج
```
الحدث: account_scored                  (يُصدره وكيل ذكاء الإيراد)
   -> السجل: AI Run Ledger             (إلحاق التكلفة والرموز والنموذج)
   -> المقياس: AI cost per project     (إعادة حساب)
   -> العتبة: تجاوز ميزانية النموذج    (إطلاق قاعدة قرار)
   -> القرار: التحويل لطبقة أرخص
   -> الفعل: مُوجّه نماذج Core OS يُحدّث ملف المشروع
```
النتيجة: يستمر سير العمل ذاته على نموذج أقل تكلفة. يُسجّل القرار في طبقة
القرار ويظهر في حزمة القرار الأسبوعية.

### التدفّق "ب" — الإثبات إلى التمنتج
```
الحدث: proof_event_created             (تسليم تقرير ذكاء الإيراد)
   -> السجل: Proof Ledger              (نوع الإثبات والمقياس والقيمة)
   -> المقياس: proof events per project
   -> العتبة: قابلية الإثبات عبر ≥ 3 مشاريع
   -> القرار: الخدمة قابلة للإثبات، رقّها
   -> الفعل: المنتج ينشئ مرشّح ميزة، الإيراد يُحدّث العرض
```
النتيجة: يُعترف بخط الخدمة كقابل للتكرار ويُوجَّه إلى التمنتج (راجع
`CAPITAL_ALLOCATION_SCORE.md`).

### التدفّق "ج" — الترقية إلى Venture
```
الحدث: retainer_won + venture_signal_detected
   -> السجل: Venture Signal Ledger
   -> المقياس: Venture Readiness Score
   -> العتبة: ≥ 85
   -> القرار: مرشّح Venture
   -> الفعل: الرئيس التنفيذي يفتح مراجعة الترقية
```
النتيجة: تدخل الوحدة مراجعة Venture وفق `VENTURE_SIGNAL_MODEL.md`.

## قواعد القرار
قاعدة القرار رباعية: (مقياس، عتبة، نوع قرار، مالك). القواعد مُصدَّرة
ومُسجَّلة مع مقياسها في `METRICS_ENGINE.md`. أمثلة:

| المقياس | العتبة | القرار | المالك |
|---|---|---|---|
| AI cost per project | تجاوز ميزانية النموذج لمدّتين | التحويل لطبقة أرخص | رئيس عمليات الذكاء |
| Proof events per project | ≥ 3 عبر ≥ 3 مشاريع | ترقية خط الخدمة | رئيس المنتج |
| Schema failure rate | > 2% خلال 7 أيام | إيقاف الوكيل وفتح Eval | رئيس عمليات الذكاء |
| Retainer win rate | < 20% خلال 30 يوماً | مراجعة التسعير والـICP | رئيس الإيراد |
| Venture Readiness Score | ≥ 85 لمدّة 30 يوماً | فتح مراجعة الترقية | الرئيس التنفيذي |

## أنماط ممنوعة
- إصدار أحداث بلا مُعرّف مشروع/عميل.
- التعبئة الرجعية للأحداث دون قيد في سجل التدقيق.
- تخطّي الناقل والكتابة المباشرة في السجل.
- لوحات مخصّصة تقرأ من السجلات (Logs) لا من السجلات (Ledgers).

## الواجهات
| المدخلات | المخرجات | المالكون | الإيقاع |
|---|---|---|---|
| إصدارات أحداث Core OS | كتابة في السجلات | الهندسة | مستمر |
| قيود السجلات | إعادة حساب المقاييس | محرّك المقاييس | مستمر / مجدول |
| قيم المقاييس + قواعد القرار | قرارات | طبقة القرار | عند تجاوز العتبة |
| القرارات | تعليمات Core OS | رئيس الديوان | أسبوعي |

## المقاييس
- **معدّل اجتياز مخطّط الحدث** — نسبة الأحداث المُصدَرَة التي تجتاز التحقّق
  (الهدف: ≥ 99.5%).
- **زمن الحدث إلى القرار** — الوسيط من الإصدار إلى الإرسال (الهدف: ≤ 24 ساعة
  للقواعد ذات الإيقاع الأسبوعي).
- **تغطية القرار** — نسبة العتبات المُطلَقة المُنتجَة لقرار موثَّق
  (الهدف: 100%).
- **معدّل الأحداث غير المُصنَّفة** — نسبة المحاولات لأنواع خارج التصنيف
  (الهدف: 0%).
