# خطة إطلاق Dealix في السوق السعودي — Focus 5 Launch Plan (30 يومًا)

خطة تشغيلية لإطلاق أنظمة Dealix الخمسة في السوق السعودي خلال 30 يومًا، تركّز على جاهزية الموقع، وتشغيل مصنع المسودات يوميًا، والوصول إلى أول عملاء عبر حِزم Sprint — بانضباط في الإرسال واحترام كامل لقواعد البريد والخصوصية.

> ملاحظة: هذه خطة تشغيل ومتابعة، وليست وعدًا بإيراد. كل الأرقام أدناه أهداف نشاط وحدود تشغيلية، لا نتائج مضمونة.

---

## الأهداف

أهداف واقعية قابلة للقياس خلال 30 يومًا:

- [ ] **جاهزية الموقع بالكامل**: صفحات `/systems` و`/systems/:slug` و`/pricing` و`/diagnostic` و`/start` تعمل بالعربية (RTL) وخالية من الأخطاء.
- [ ] **تشغيل مصنع المسودات يوميًا**: إنتاج 400 مسودة بريد بارد مخصّصة لكل شركة كل يوم عمل عبر `npm run outreach:factory`.
- [ ] **اكتمال بوابات الإرسال** قبل أي إرسال: توثيق الدومين (SPF/DKIM/DMARC) + قائمة موافِقة (opt-in) موثّقة + إلغاء اشتراك بنقرة واحدة.
- [ ] **مراجعة بشرية يومية** لأفضل 100 مسودة (Top 100) واعتماد المؤسس قبل أي إرسال.
- [ ] **الوصول إلى أول عملاء Sprint** وتسليم أول حزم خلال الأسبوعين الثالث والرابع.
- [ ] **بناء أول مادة إثبات (Proof)** من نتائج التسليم لاستخدامها في التوسعة.

الأنظمة الخمسة وحِزمها:

| النظام | السعر | مدة التنفيذ |
| --- | --- | --- |
| نظام تشغيل الإيرادات | 4,500 ريال | 7–10 أيام |
| لوحة القرار التنفيذي | 5,500 ريال | 7–14 يوم |
| نظام استرجاع المتابعات | 3,500 ريال | 7 أيام |
| نظام تشغيل العملاء على واتساب | 4,500 ريال | 7–10 أيام |
| نظام العروض والإثبات | 3,000 ريال | 5–7 أيام |

## بنية الجاهزية قبل الإطلاق

لا إرسال قبل اكتمال كل العناصر التالية:

**1. الموقع (Vite + React، عربي RTL):**
- [ ] صفحة `/systems` تعرض الأنظمة الخمسة بوضوح.
- [ ] صفحة `/systems/:slug` لكل نظام (الوصف، الفائدة، السعر، مدة التنفيذ).
- [ ] صفحة `/pricing` بالأسعار الخمسة.
- [ ] صفحة `/diagnostic` (تشخيص احتياج العميل).
- [ ] صفحة `/start` (نقطة بدء التواصل/الطلب).

**2. الدومين وتوثيقه (شرط إلزامي للإرسال):**
- [ ] **SPF** مضبوط للنطاق المُرسِل.
- [ ] **DKIM** مفعّل وموقّع.
- [ ] **DMARC** منشور مع سياسة (DMARC policy) تبدأ من `p=none` للمراقبة ثم تتدرّج نحو `quarantine`/`reject`.
- [ ] **إلغاء اشتراك بنقرة واحدة** (one-click unsubscribe) في كل بريد تسويقي.
- [ ] صندوق متابعة تقارير DMARC للمراقبة الأسبوعية.

**3. مصنع المسودات:**
- [ ] `scripts/daily-400-draft-factory.mjs` يعمل ويُخرج التقارير تحت `reports/outreach/`.
- [ ] التوزيع اليومي: الإيرادات 100، استرجاع المتابعات 90، القرار التنفيذي 70، واتساب 70، العروض 70 (إجمالي 400).
- [ ] المخرجات **مسودات فقط**، لا إرسال تلقائي.

**4. سجل الموافقات:**
- [ ] سجل يوثّق: تاريخ التوليد، المسودات المعتمدة، اسم المُعتمِد (المؤسس)، وقت الإرسال، والنتيجة.
- [ ] قائمة الموافقة (opt-in) موثّقة المصدر — **بدون قوائم مشتراة إطلاقًا**.

## الجدول الزمني

### الأسبوع 1 — البنية والمحتوى والتوثيق
- [ ] إنهاء وتدقيق صفحات الموقع الخمس بالعربية (RTL).
- [ ] إعداد الدومين المُرسِل وضبط SPF/DKIM/DMARC (`p=none` للبدء).
- [ ] تفعيل إلغاء الاشتراك بنقرة واحدة في قالب البريد التسويقي.
- [ ] تجهيز قائمة opt-in موثّقة المصدر وتنظيفها.
- [ ] اختبار تشغيل `npm run outreach:factory` والتأكد من مخرجات `reports/outreach/`.
- [ ] إنشاء سجل الموافقات الفارغ وجاهزيته.

### الأسبوع 2 — تشغيل المسودات + مراجعة Top 100 + بدء إرسال محدود (بعد اكتمال البوابات فقط)
- [ ] تشغيل مصنع المسودات يوميًا (400 مسودة/يوم عمل).
- [ ] ترتيب ومراجعة أفضل 100 مسودة يوميًا.
- [ ] اعتماد المؤسس للمسودات قبل أي إرسال.
- [ ] **بدء إرسال محدود ومتدرّج فقط بعد اكتمال: توثيق الدومين + قائمة opt-in + إلغاء الاشتراك.**
- [ ] مراقبة معدل الشكاوى والارتداد يوميًا.
- [ ] تسجيل كل إجراء في سجل الموافقات.

### الأسبوع 3 — أول Sprints وتسليم أول حِزم
- [ ] تحويل الردود المهتمة إلى اجتماعات تشخيص.
- [ ] إغلاق أول صفقات Sprint وفق الأسعار المعلنة.
- [ ] بدء تنفيذ أول الحِزم حسب مدد التنفيذ المحددة.
- [ ] الاستمرار في الحلقة اليومية (توليد → مراجعة → اعتماد → إرسال متدرّج).
- [ ] رفع حجم الإرسال تدريجيًا فقط إذا بقيت مؤشرات السمعة سليمة.

### الأسبوع 4 — مراجعة، Proof، توسعة
- [ ] تسليم أول الحِزم وجمع ملاحظات العملاء.
- [ ] بناء أول مادة إثبات (Proof) من نتائج فعلية بموافقة العميل.
- [ ] مراجعة مؤشرات الشهر ومعدلات الموافقة والردود.
- [ ] ضبط الرسائل والاستهداف بناءً على ما نجح.
- [ ] توسعة حجم الإرسال المعتمد بحذر مع الحفاظ على سمعة الدومين.

## التشغيل اليومي

الحلقة اليومية ثابتة وتُسجَّل بالكامل:

1. **توليد 400 مسودة** عبر `npm run outreach:factory` (مخرجات تحت `reports/outreach/`).
2. **ترتيب أفضل 100 مسودة** (Top 100) حسب الجودة والملاءمة.
3. **موافقة المؤسس** — مراجعة بشرية واعتماد صريح قبل أي إرسال.
4. **إرسال محدود ومتدرّج** فقط للقائمة الموافِقة (opt-in) وبعد توثيق الدومين بالكامل.
5. **تسجيل كل إجراء**: ما تم توليده، ما اعتُمد، ما أُرسل، ومؤشرات الشكاوى/الارتداد.

> 400 مسودة/يوم **مطلوبة**. أما 400 إرسالة/يوم فهي **غير مفعّلة افتراضيًا**، ولا تُفعّل إلا بموافقة المؤسس واكتمال جميع البوابات.

## القنوات

| مسموح | ممنوع / مؤجّل |
| --- | --- |
| البريد الإلكتروني لقائمة موافِقة (opt-in) موثّقة | البريد البارد لقوائم **مشتراة** |
| الموقع الإلكتروني (صفحات الأنظمة والتسعير والتشخيص) | أتمتة واتساب الباردة |
| الإحالات والتوصيات | أتمتة LinkedIn |
| المتابعة مع من تفاعل أو وافق سابقًا | رسائل بعناوين `Re:`/`Fwd:` مزيّفة |

قواعد ثابتة: لا قوائم مشتراة، لا واتساب بارد آلي، لا أتمتة LinkedIn، لا عناوين مزيّفة، لا وعود بإيراد.

## تدرّج الإرسال (Warm-up)

نبدأ بأحجام إرسال صغيرة ونزيدها تدريجيًا لحماية سمعة الدومين الجديد:

- الدومين حديث الإرسال يفتقر إلى سجل ثقة لدى مزوّدي البريد؛ القفز إلى أحجام كبيرة فجأة يرفع احتمال التصنيف كبريد مزعج.
- نبدأ بعدد صغير من الرسائل يوميًا ونزيده بنسبة محدودة كل يوم/أسبوع **فقط** عندما تبقى المؤشرات سليمة.
- **الربط بالمؤشرات**: ارتفاع معدل الشكاوى (يجب إبقاؤه **أقل من 0.3%**) أو ارتفاع الارتداد يوقف الزيادة فورًا ويستدعي العودة لحجم أقل.
- الجودة قبل الحجم: مسودة مخصّصة ومراجَعة لقائمة موافِقة أفضل من إرسال واسع غير مدروس.

## مؤشرات المتابعة الأسبوعية

مؤشرات نشاط للمتابعة والتعلّم — **وصفية لا وعود**:

- [ ] **عدد المسودات** المُولّدة (المستهدف: 400 × أيام العمل).
- [ ] **نسبة الموافقة**: المعتمد من المؤسس مقابل المُولّد.
- [ ] **عدد الرسائل المُرسلة فعليًا** (ضمن حدود التدرّج).
- [ ] **عدد الردود** ونوعها (مهتم/غير مهتم/إلغاء اشتراك).
- [ ] **عدد الاجتماعات** التشخيصية.
- [ ] **عدد الـ Sprints المباعة** والحِزم قيد التنفيذ/المسلّمة.
- [ ] **مؤشرات السمعة**: معدل الشكاوى (<0.3%) ومعدل الارتداد وتقارير DMARC.

## بوابات عدم الإطلاق (Stop conditions)

نوقف الإرسال فورًا عند أي من الحالات التالية:

- [ ] **ارتفاع الشكاوى** فوق 0.3% أو اقترابها من الحد.
- [ ] **ارتفاع معدل الارتداد** (Bounce) بشكل غير طبيعي.
- [ ] **نقص أو فشل التوثيق**: SPF أو DKIM أو DMARC غير مكتمل/متعطّل.
- [ ] **غياب موافقة المؤسس** على مسودات اليوم.
- [ ] **عدم وجود قائمة opt-in موثّقة** أو وجود شك في مصدرها.
- [ ] **تعطّل إلغاء الاشتراك بنقرة واحدة** في البريد التسويقي.

لا يُستأنف الإرسال إلا بعد معالجة السبب، وتوثيق الإجراء التصحيحي في سجل الموافقات، واعتماد المؤسس مجددًا.
