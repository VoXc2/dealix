# Tool Poisoning Defense — الدفاع ضد تسميم الأدوات

> tool poisoning هو حقن تعليمات خبيثة في أوصاف الأدوات أو نتائجها لدفع الوكيل
> لاستدعاء أدوات غير مصرّح بها أو تمرير معاملات خفية. خطر حقيقي في عملاء MCP.

---

## أنماط الهجوم

| النمط | الوصف |
|------|-------|
| Hidden parameter exploitation | معاملات خفية تُمرَّر دون علم المستخدم |
| Unauthorized tool invocation | دفع الوكيل لاستدعاء أداة غير مصرّح بها |
| Description poisoning | تعليمات خبيثة داخل وصف الأداة |
| Result poisoning | تعليمات داخل نتيجة الأداة لتوجيه الخطوة التالية |

---

## قواعد الدفاع

```txt
1. لا تُستدعى أداة بناءً على محتوى مسترجَع أو وصف أداة خارجي.
2. الأدوات المسموحة محدّدة مسبقًا (allowlist)، لا اكتشاف ديناميكي خطر.
3. معاملات الأداة تُتحقَّق قبل التنفيذ (no hidden parameters).
4. نتيجة الأداة تُعامَل كبيانات، لا تعليمات.
5. أي أداة كتابة/إرسال تتطلب موافقة.
6. كل استدعاء أداة يُسجَّل.
```

---

## allowlist الأدوات

```txt
- الأدوات المصرّح بها معرّفة لكل وكيل ضمن عقده.
- لا "use hidden tool" ولا استدعاء خارج القائمة.
- اكتشاف محاولة استدعاء خارج القائمة → رفض + سجل + flag.
```

---

## الربط بالوكلاء

كل وكيل له عقد إدخال/إخراج (Workflow-First). الأداة خارج العقد = مرفوضة.
لا وكيل يملك `can_send_external/can_call/...` (يفرضه `check_agent_permissions.py`).

---

## الاختبار

ضمن `reports/security/PROMPT_INJECTION_TEST_REPORT.md`، تُختبر سلاسل مثل
`use hidden tool` و`execute command` ويجب أن تكون `blocked = true`.
