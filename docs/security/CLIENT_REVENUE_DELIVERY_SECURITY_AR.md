## 1. مبدأ "لا أسرار" — القاعدة الذهبية

### ممنوع في أي ملف بيانات/تقرير/JSONL/GitHub/دردشة

| النمط المحظور | مثال على الانتهاك | البديل الآمن |
|--------------|------------------|--------------|
| مفاتيح OpenAI | `sk-abc123...` | `portal://secrets/openai-key` |
| مفاتيح AWS | `AKIA...` | `portal://secrets/aws-key` |
| مفاتيح Stripe | `sk_live_...` | `portal://secrets/stripe-key` |
| مفاتيح GitHub PAT | `ghp_...` | `portal://secrets/github-pat` |
| كلمات مرور | `password=MyPass123` | `portal://secrets/db-password` |
| Bearer tokens | `Bearer eyJ...` | `portal://secrets/api-token` |
| روابط دفع نشطة | `https://payment.co/pay/abc` | `portal://payment-links/PAY-1001` |
| أرقام جوال حقيقية | رقم جوال حقيقي كامل غير مُقنَّع | `+9665XXXXXXXX` |
| بريد إلكتروني شخصي | `user@gmail.com` | `REDACTED` أو `user@dealix.sa` |
| مفاتيح خاصة | نص يبدأ بـ BEGIN PRIVATE KEY | `portal://certs/private-key` |

### التطبيق الآلي
الفحص `scripts/client_revenue_delivery_check.py` يفحص كل ملف `.json`, `.jsonl`, `.yaml`, `.md`, `.csv` في `data/`, `reports/`, `schemas/` ويرفض الأنماط المحظورة. CI تفشل تلقائيًا عند أي انتهاك.

## 3. مبدأ أقل الامتياز

### مستويات الوصول في التسليم

| طبقة | الوصول المسموح | الوصول الممنوع |
|------|---------------|----------------|
| تحليل بيانات العميل | `read_only_crm` | كتابة · حذف · تعديل |
| رفع الملفات | `file_upload` (بوابة آمنة فقط) | وصول مباشر لملفات الإنتاج |
| أنظمة العميل | قراءة فقط بعد إذن موثَّق | أي وصول كتابة |
| سجلات الإجراءات | إضافة فقط (append-only) | تعديل · حذف السجلات |
| بيانات الإنتاج | لا وصول مستقل | أي تعديل في الإنتاج |

### قاعدة: كل صلاحية تُدقَّق
- كل طلب وصول موثَّق في `ai_action_ledger`.
- الصلاحيات تُمنح لمدة محددة (لا صلاحيات دائمة للتحليل).
- مراجعة أسبوعية لصلاحيات الوصول النشطة.

## 5. مرجع `secret_ref` — نمط البوابة

### الصيغة الصحيحة
```
secret_ref: "portal://secrets/[category]/[ref-id]"
```

### أمثلة صحيحة
```json
"payment_link_ref": "portal://payment-links/PAY-1001"
"api_access_ref": "portal://secrets/crm-readonly/DHO-1001"
"file_upload_ref": "portal://files/client-data/Digital-Rise/2026-06"
```

### أمثلة خاطئة (ترفضها الاختبارات)
```json
"payment_link": "https://bank.com/pay/xyz123"   ← انتهاك
"api_key": "sk-abc123xyz"                         ← انتهاك
"crm_password": "MyPassword123"                  ← انتهاك
```

## 6. أمن البيانات في طبقات النظام

### طبقة واتساب
- لا مفاتيح API في رسائل واتساب.
- لا بيانات PII عملاء في رسائل واتساب.
- روابط البوابة الآمنة فقط عند الحاجة لتبادل الملفات/الأسرار.

### طبقة الإيراد (عروض/مدفوعات/Proof Packs)
- السعر النهائي لا يظهر إلا بعد موافقة المؤسس.
- روابط الدفع تُولَّد في البوابة ولا تظهر في JSONL أو التقارير.
- `guaranteed_roi=false` إلزامي في كل Proof Pack.

### طبقة التسليم
- وصول `read_only` فقط لأنظمة العميل.
- بيانات العميل تُخفى قبل التحليل.
- تقارير القيمة تُرسَل عبر البوابة الآمنة.

### طبقة التجديد/الترقية
- لا تجديد/ترقية بلا `evidence_level ∈ {client_data, measured, verified}`.
- مسودات التجديد تنتظر موافقة المؤسس قبل أي تواصل.
