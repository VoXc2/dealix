# سياسة التعارض بين الوكلاء (Agent Collision Policy)

عند تعديل أكثر من وكيل لنفس الملف، أو عند تعارض المخرجات، تُطبَّق هذه السياسة.

## القواعد

1. **الحفاظ على بوابات الأمان الأحدث:** لا تُستبدل بوابة أمان أحدث بأخرى أقدم.
2. **عدم الكتابة فوق التعديلات اليدوية:** أي محتوى من المؤسس/بشري لا يُستبدل تلقائياً.
3. **تفضيل أصغر PR:** عند التعارض، يُفضَّل التغيير الأصغر والأكثر تحديداً.
4. **تقرير تعارض إلزامي:** يُنشأ تقرير في `reports/agents/` يصف التعارض والخيارات.
5. **قرار المؤسس عند الغموض:** إذا لم يكن الحل واضحاً، يُرفع للمؤسس عبر سؤال صريح.

## حالة موثّقة حالياً (مفتوحة)

🔴 **شجرة مكرّرة:** `company_os/company_os/` تكرار شبه كامل لـ `company_os/`
لكن بمحتوى حوكمة **مختلف** (انظر `reports/agents/AGENT_GOVERNANCE_REVIEW.md`).
- لم تُحذف (محتوى يدوي لم ننشئه).
- التوصية: توحيد المصدر في `company_os/` وإزالة التكرار **بقرار المؤسس**.

## إجراء التعامل مع التعارض

```
اكتشاف تعارض
  → هل أحد الطرفين بوابة أمان أحدث؟ → احتفظ بها
  → هل أحد الطرفين تعديل يدوي؟ → لا تكتب فوقه
  → أنشئ reports/agents/CONFLICT_<date>.md
  → إن بقي غموض → اسأل المؤسس (AskUserQuestion)
```

## ممنوع أثناء حل التعارض

- ❌ `force push` أو حذف عمل وكيل آخر.
- ❌ إضعاف اختبار أو إزالة بوابة أمان لحل التعارض.
- ❌ دمج تغييرات خارجية غير موثوقة بصلاحيات كتابة.
