# Vendor and Tool Adoption Policy

## Purpose
Prevent tool sprawl, hidden cost, data exposure, and unmanaged operational risk.

## When this applies
Use this policy before adding:

- AI providers
- Data providers
- CRM tools
- Email tools
- Analytics tools
- Monitoring tools
- Browser automation tools
- Scrapers or enrichment tools
- Internal productivity tools
- Customer-facing third-party services

## Approval checklist

| Check | Required answer |
|---|---|
| Business use case | What outcome does this improve? |
| Owner | Who owns this tool? |
| Data used | What data is sent, stored, or processed? |
| Access model | Who can access it? |
| Cost model | What can make cost grow? |
| Security review | What risks exist? |
| Alternative | What happens if we do not buy/build it? |
| Exit plan | How do we stop using it? |
| Review date | When do we revisit? |

## Decision levels

| Level | Example | Required approval |
|---|---|---|
| Low | Internal doc helper | Team owner |
| Medium | Tool with business data | Founder or workstream owner |
| High | Tool with customer or sensitive data | Founder and security review |
| Critical | Customer-facing or high-impact automation | Formal decision record |

## Stop rules

Retire a tool when:

- No owner exists.
- It duplicates another tool.
- It has unclear data handling.
- Cost grows without clear value.
- It is not used for 30 days.
- It creates operational or compliance risk.

## Founder rule

A tool is not adopted because it is impressive. It is adopted only if it improves speed, quality, trust, revenue, retention, or operating leverage.
