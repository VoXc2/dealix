# مصنع المسودات اليومي — 400 مسودة عبر الأنظمة الخمسة (Daily 400 Draft Factory)

> الغرض: إنتاج **400 مسودة بريد** يوميًا، موزّعة على أنظمة Dealix الخمسة، تمر كل واحدة عبر خط إنتاج موحّد وتنتهي في قائمة الموافقة (Top 100). كل مخرج يبقى **مسودة** حتى موافقة المؤسس، ولا إرسال تلقائي.

المصدر المرجعي للأنظمة والمصطلحات: `data/systems.json`. مخطط المسودة: `schemas/email_draft.schema.json`.

## القاعدة الصارمة (لا تتغير)

- المطلوب **مسودات فقط** — الإرسال **غير مفعّل افتراضيًا**.
- 400 مسودة/يوم نعم، لكن **400 إرسال/يوم لا**. كل بريد يبقى `draft` حتى يوافق المؤسس عليه يدويًا.
- لا قوائم مشتراة، ولا كشط بيانات خاصة، ولا أتمتة LinkedIn أو واتساب باردة.
- ممنوع أي وعد بإيراد مضمون. كل افتراض منخفض الدليل يُصاغ كاحتمال («غالبًا»، «قد»)، لا كحقيقة.

## خط الإنتاج (Pipeline) — كل مسودة تمر بسبع محطات

```
1. بحث الشركات          Company Search
        ↓
2. حزمة استخبارات الشركة  Company Intelligence Pack
        ↓
3. بطاقة حاجة العميل      Client Need Card
        ↓
4. النظام الموصى به       Recommended System  (واحد من خمسة فقط)
        ↓
5. مسودة البريد           Email Draft
        ↓
6. درجة الجودة            Quality Score (/100)
        ↓
7. قائمة الـ Top 100      Top 100 Queue  (≥ 75 فقط)
```

| المحطة | المدخل | المخرج |
|--------|--------|--------|
| Company Search | قطاع/مدينة/إشارة عامة | قائمة شركات مرشّحة |
| Company Intelligence Pack | معلومات عامة أو بيانات المؤسس (L4) | ملف بحثي موحّد لكل شركة |
| Client Need Card | الحزمة الاستخباراتية | ألم مرجّح + نظام مرشّح واحد |
| Recommended System | بطاقة الحاجة | معرّف نظام من الخمسة + سبب الاختيار |
| Email Draft | النظام + الإشارة + `email_angle` | مسودة بريد مخصّصة |
| Quality Score | المسودة + الـ rubric | درجة من 100 |
| Top 100 Queue | المسودات ≥ 75 | طابور موافقة المؤسس |

تفاصيل المحطة 2 في `docs/acquisition/COMPANY_INTELLIGENCE_PACK_AR.md`، والمحطة 3 في `docs/outreach/SYSTEM_BASED_CLIENT_NEED_CARD_AR.md`، والمحطتان 6–7 في `docs/outreach/TOP_100_APPROVAL_QUEUE_AR.md`.

## حقول مسودة البريد (Email Draft Fields)

كل مسودة كائن يطابق `schemas/email_draft.schema.json`:

| الحقل | المعنى |
|------|--------|
| `company` | اسم الشركة كما يظهر علنًا |
| `website` | الموقع الرسمي إن وُجد |
| `country` | الدولة (غالبًا السعودية) |
| `city` | المدينة إن أمكن من مصدر عام |
| `sector` | القطاع (تدريب، عقار، عيادات، وكالة، استشارات...) |
| `signal` | إشارة عامة قابلة للملاحظة تبرّر التواصل |
| `likely_pain` | الألم المرجّح، مرتبط بالإشارة لا بافتراض جازم |
| `recommended_system` | معرّف أحد الأنظمة الخمسة فقط |
| `why_this_system` | لماذا هذا النظام تحديدًا يناسب الإشارة والألم |
| `first_mission` | أول مخرج صغير يقدّمه النظام خلال أيام |
| `proof_angle` | زاوية الإثبات (من `ops.proof_angle` للنظام) |
| `email_angle` | زاوية الرسالة (من `ops.email_angle` للنظام) |
| `subject` | عنوان بريد قصير (≤ 90 حرفًا)، بلا مبالغة، بلا Re:/Fwd: مزيّف |
| `email_body` | نص المسودة المخصّص |
| `cta` | دعوة فعل واحدة واضحة (تشخيص/مكالمة قصيرة) |
| `risk_level` | `low` / `medium` / `high` |
| `evidence_level` | `L0` / `L1` / `L2` / `L3` / `L4` |
| `quality_score` | درجة الجودة (0–100)، تُملأ في المحطة 6 |
| `approval_status` | `draft` / `pending` / `approved` / `rejected` (الافتراضي `draft`) |
| `send_readiness` | `not_ready` / `ready_for_review` / `approved_draft` — **لا توجد حالة إرسال تلقائي** |

> ملاحظة: `send_readiness = approved_draft` تعني أن المسودة جاهزة وموافَق عليها للإرسال **اليدوي**، لا أن النظام سيرسلها بنفسه.

## بوابة موافقة المؤسس (Founder Approval Gate)

1. تُولَّد المسودات الـ 400 وتُقيّم آليًا.
2. تدخل المسودات ≥ 75 فقط طابور الـ Top 100.
3. يراجع المؤسس الطابور ويغيّر `approval_status` إلى `approved` لما يختاره.
4. الإرسال خطوة **بشرية منفصلة** تتم بعد الموافقة فقط — لا قبلها، ولا تلقائيًا.

## السكربت والمخرج

- السكربت: `scripts/commercial-daily-plan.js` (الأمر: `npm run commercial:plan`).
- يقرأ `data/systems.json` ويولّد التقرير: `reports/outreach/DAILY_400_SYSTEM_DRAFT_PRODUCTION.md`.
- التقرير يلخّص التوزيع (100/70/90/70/70)، حالة كل مسودة، ودرجات الجودة قبل المرور إلى طابور الموافقة.
