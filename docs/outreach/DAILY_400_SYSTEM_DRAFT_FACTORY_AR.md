# مصنع 400 مسودة يوميًا — Daily 400 System Draft Factory

نظام يومي ينتج **400 مسودة إيميل مخصصة** للشركات السعودية (B2B) عبر الأنظمة
الخمسة، يرتّب أفضل 100 للمراجعة، ويبني بطاقات احتياج العميل — كل ذلك كمسودات
محكومة ببوابات إرسال.

> **مبدأ التصميم:** 400 **مسودة**/يوم مطلوبة من اليوم الأول. أما 400 **إرسال**/يوم
> فهو خيار تشغيلي لاحق حسب جاهزية الدومين، وليس شرط تصميم. هذا حماية لسمعة قناة
> الإيميل.

## التشغيل

```bash
npm run outreach:factory
# أو بتاريخ محدد (نتائج قابلة للتكرار):
node scripts/daily-400-draft-factory.mjs --date=2026-06-03
```

بوابات الإرسال (كلها معطّلة افتراضيًا — متغيرات بيئة):

```bash
FOUNDER_APPROVAL=true DOMAIN_AUTH_READY=true VERIFIED_OPT_IN=true \
  node scripts/daily-400-draft-factory.mjs
```

## التوزيع اليومي

| النظام | عدد المسودات/يوم |
| --- | ---: |
| نظام تشغيل الإيرادات (Revenue Operating System) | 100 |
| نظام استرجاع المتابعات (Follow-up Recovery OS) | 90 |
| لوحة القرار التنفيذي (Executive Command OS) | 70 |
| نظام تشغيل العملاء على واتساب (WhatsApp Client OS) | 70 |
| نظام العروض والإثبات (Proposal & Proof OS) | 70 |
| **الإجمالي** | **400** |

التوزيع مضمون بالضبط عبر إسناد واعٍ بالملاءمة: لكل نظام تُختار الشركات الأعلى
ملاءمة لقطاعه وإشارته حتى تكتمل حصته.

## حقول كل مسودة

```txt
company · website · sector · country · city · signal · likely_pain
recommended_system · why_this_system · first_mission · proof_angle · email_angle
subject · email_body · cta
risk_level · evidence_level · approval_status · send_readiness
unsubscribe_required · unsubscribe_note
```

- `approval_status` = `pending_approval` دائمًا.
- `evidence_level` = `inferred` (بيانات العينة مُستنتجة؛ تصبح `verified` مع قائمة
  حقيقية موثّقة).
- `risk_level` = `Low`، أو `Medium` للقطاعات الحساسة (قانوني/طبي/مالي).

## كيف تُبنى المسودة

1. **عالم الشركات:** يُبنى عالم من 400 شركة (حاليًا عيّنة اصطناعية مُوسومة
   `synthetic-seed` بدومينات وهمية مثل `sample.example`).
2. **بطاقة احتياج العميل:** لكل شركة بطاقة تربط القطاع والإشارة بالألم المرجّح.
3. **اختيار نظام واحد:** بحسب أعلى ملاءمة (نظام واحد فقط لكل شركة).
4. **توليد الإيميل:** من قالب النظام في `scripts/lib/systems-data.mjs`.
5. **الترتيب:** درجة مركّبة من: ملاءمة النظام، وضوح الألم، التخصيص، قوة الإشارة،
   القدرة على الدفع، وانخفاض المخاطرة → أفضل 100 للمراجعة.
6. **بوابة الإرسال:** تُحسب `send_readiness`.

## بوابات الإرسال (Send Gates)

تكون المسودة `send_readiness = "ready"` فقط إذا تحققت **كل** الشروط:

1. موافقة المؤسس (`FOUNDER_APPROVAL`).
2. توثيق الدومين: SPF + DKIM + DMARC (`DOMAIN_AUTH_READY`).
3. قائمة موثّقة وموافِقة على التواصل (`VERIFIED_OPT_IN`).

وإلا تكون `blocked` مع ذكر الأسباب. القطاعات الحساسة (`Medium`) تبقى محجوبة حتى
بعد البوابات لحين مراجعة امتثال إضافية. افتراضيًا: **0 جاهزة / 400 محجوبة**.

## القواعد الصارمة

- 400 مسودة/يوم مطلوبة؛ 400 إرسال/يوم غير مفعّل افتراضيًا.
- نظام واحد فقط لكل إيميل؛ لا بيع للخمسة معًا.
- لا واتساب بارد، ولا أتمتة LinkedIn، ولا قوائم مشتراة.
- لا `Re:`/`Fwd:` مزيّفة، ولا عناوين مضللة، ولا وعود إيراد.
- لا أسرار أو بيانات حساسة (PII) في المسودات أو السجلات.
- لا أسماء وحدات داخلية في ما يصل للعميل.
- one-click unsubscribe عند الإرسال الفعلي للقنوات التسويقية.

## قبل أي إرسال حقيقي

1. استبدل العيّنة الاصطناعية بقائمة حقيقية موافِقة على التواصل.
2. أكمل توثيق الدومين وسياسة إلغاء الاشتراك.
3. ابدأ بإرسال متدرّج صغير (warm-up) لحماية السمعة وراقب الشكاوى والارتداد.
4. احصل على موافقة المؤسس على المسودات أولًا.
