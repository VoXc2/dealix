# بطاقة احتياج العميل المبنية على النظام — System-Based Client Need Card

البطاقة هي الأساس الذي يجعل كل إيميل قويًا ومخصصًا. لكل شركة مستهدفة بطاقة واحدة
تربط سياقها بالنظام الأنسب من الأنظمة الخمسة قبل كتابة أي رسالة.

## القالب

```txt
Company:            اسم الشركة
Website:            الموقع
Country:            الدولة (السعودية)
City:               المدينة
Sector:             القطاع
Signal:             الإشارة الظاهرة (ما الذي يدل على وجود فرصة/ألم؟)
Likely Pain:        الألم المرجّح
Recommended System: النظام المقترح (واحد فقط)
Why this system:    لماذا هذا النظام تحديدًا لهذه الشركة
First Mission:      أول مهمة/Sprint مقترح
Proof Angle:        زاوية الإثبات (قبل/بعد، نموذج، تقرير)
Email Angle:        زاوية الإيميل (الفكرة المحورية للرسالة)
CTA:                دعوة الإجراء
Risk Level:         مستوى المخاطرة (Low / Medium)
```

## تعريف الحقول

| الحقل | المعنى | ملاحظة الجودة |
| --- | --- | --- |
| Company / Website | هوية الشركة | يجب أن تكون شركة حقيقية موافِقة على التواصل قبل الإرسال |
| Country / City | الموقع الجغرافي | يربط الرسالة بالسياق المحلي |
| Sector | القطاع | يحدد أي نظام أنسب |
| Signal | إشارة ظاهرة علنًا (قنوات، حملات، نماذج تواصل) | لا تُستخدم بيانات خاصة أو مسرّبة |
| Likely Pain | فرضية الألم بناءً على القطاع والإشارة | فرضية لا جزم؛ تُصاغ باحترام |
| Recommended System | نظام واحد فقط | ممنوع اقتراح أكثر من نظام في البطاقة |
| Why this system | سبب الملاءمة | يربط الألم بأسرع قيمة |
| First Mission | أول Sprint | محدد المدة والمخرج |
| Proof Angle | كيف نثبت القيمة مبكرًا | قبل/بعد أو نموذج أو تقرير |
| Email Angle | الفكرة المحورية للرسالة | تتحوّل لاحقًا إلى صياغة الإيميل |
| CTA | دعوة إجراء واحدة | سؤال خفيف لا ضغط |
| Risk Level | حساسية القطاع/الامتثال | Medium للقطاعات الحساسة (قانوني/طبي/مالي) |

## مثال فعلي

```txt
Company:            شركة تدريب في الرياض
Website:            example.sa
Country:            السعودية
City:               الرياض
Sector:             التدريب
Signal:             برامج متعددة + واتساب ظاهر للتواصل
Likely Pain:        استفسارات التسجيل قد تضيع أو لا تُتابع بنفس الجودة
Recommended System: نظام استرجاع المتابعات (Follow-up Recovery OS)
Why this system:    أسرع قيمة تظهر في ترتيب المتابعة وتجهيز الرسائل
First Mission:      Follow-up Recovery Sprint (7 أيام)
Proof Angle:        قبل/بعد follow-up queue + تقرير استرجاع أسبوعي
Email Angle:        المشكلة ليست في عدد الاستفسارات، بل في آخر متابعة لم تحدث
CTA:                أرسل لك تصورًا مختصرًا لأول workflow؟
Risk Level:         Low
```

## كيف تتحول البطاقة إلى مسودة

1. تُبنى البطاقة من القطاع + المدينة + الإشارة + الألم المرجّح.
2. يُختار **نظام واحد** بحسب أعلى ملاءمة للقطاع والإشارة.
3. يُملأ قالب الإيميل الخاص بالنظام من حقول البطاقة.
4. تُحسب درجة الأولوية (وضوح الألم، التخصيص، ملاءمة القطاع، قوة الإشارة، القدرة
   على الدفع، انخفاض المخاطرة، ملاءمة النظام).
5. تبقى المسودة `pending_approval` حتى تكتمل بوابات الإرسال.

## الربط بالأتمتة

يولّد مصنع المسودات اليومي بطاقة احتياج كاملة لكل شركة ضمن المخرجات الآلية:
`company_os/outreach/daily_400_drafts.json`، وملخّص لأفضل 20 في
`reports/outreach/SYSTEM_BASED_CLIENT_NEED_CARDS.md`.

<sub>قاعدة: شركة حقيقية موافِقة على التواصل قبل أي إرسال — لا قوائم مشتراة، ولا
بيانات حساسة في البطاقة أو السجلات.</sub>
