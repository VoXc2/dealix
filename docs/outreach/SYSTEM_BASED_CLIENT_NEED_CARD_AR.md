# بطاقة حاجة العميل حسب النظام (System-Based Client Need Card)

> الغرض: تحويل ملاحظة عامة عن شركة إلى **نظام موصى به واحد** من الخمسة، عبر منطق واضح يربط الإشارة (signal) بالألم المرجّح (likely_pain) بالنظام الأنسب. البطاقة هي المحطة الثالثة في مصنع المسودات.

المصدر المرجعي: `data/systems.json`. المخطط: `schemas/client_need_card.schema.json`.

## القواعد

- **نظام واحد فقط** لكل شركة. لا نوصي بنظامين في الرسالة الأولى؛ نختار أقوى تطابق.
- الإشارة من **مصدر عام** فقط (موقع، صفحة وظيفة، خبر، زر واتساب معلن) أو بيانات يقدّمها المؤسس.
- الألم منخفض الدليل يُصاغ كاحتمال: «غالبًا»، «قد»، «في هذا النوع من الشركات». لا جزم، ولا وعد بإيراد مضمون.
- `recommended_system` يجب أن يكون أحد المعرّفات الخمسة فقط.

## حقول البطاقة

| الحقل | المعنى |
|------|--------|
| `company` | اسم الشركة |
| `sector` | القطاع |
| `recommended_system` | معرّف النظام (واحد من خمسة) |
| `likely_pain` | الألم المرجّح المرتبط بالإشارة |
| `why_this_system` | لماذا هذا النظام يناسب الإشارة |
| `first_mission` | أول مخرج صغير خلال أيام |
| `proof_angle` | ما سنكشفه أو نقيسه أولًا (من `ops.proof_angle`) |
| `starter_price_sar` | السعر الابتدائي للنظام |
| `risk_level` | `low` / `medium` / `high` |
| `evidence_level` | `L0`–`L4` |

## مستويات الدليل

| المستوى | المصدر |
|--------|--------|
| L0 | تخمين قطاعي |
| L1 | موقع الشركة |
| L2 | صفحة/وظيفة/خبر محدد |
| L3 | عدة مصادر عامة متوافقة |
| L4 | بيانات مقدّمة من الشركة |

## منطق الإشارة ← النظام (Signal → System)

| إشارة عامة ملحوظة | الألم المرجّح | النظام الموصى به |
|--------------------|---------------|------------------|
| leads/استفسارات تدخل من قنوات متعددة بلا next action واضح، غياب تقرير إيراد | إيراد يتسرب بهدوء عبر مراحل البيع | `revenue-operating-system` |
| القرار يعتمد على شخص واحد، بيانات مبعثرة بين أدوات، لا تقرير موحّد للمؤسس | قرارات تتأخر لغياب صورة يومية مجمّعة | `executive-command-os` |
| نموذج تسجيل + زر واتساب، اعتماد على استفسارات واردة بلا قائمة متابعة | فرص تُترك بلا متابعة في وقتها فتذهب لمنافس أسرع | `follow-up-recovery-os` |
| واتساب قناة رئيسية معلنة، حجم محادثات عالٍ، ردود متأخرة | فوضى محادثات بلا مسار وبلا سياسة تحويل للإنسان | `whatsapp-client-os` |
| شركة خدمات/استشارات تقدّم عروضًا متكررة، صيغ عروض غير موحّدة | عروض بطيئة وضعيفة الإثبات تؤخّر الصفقات | `proposal-proof-os` |

> القاعدة عند التداخل: إذا ظهرت أكثر من إشارة، اختر النظام الأقرب لأقوى إشارة قابلة للملاحظة. مثال: واتساب معلن **مع** استفسارات متروكة ← إن كان الثقل على القناة نفسها فـ `whatsapp-client-os`، وإن كان على غياب المتابعة فـ `follow-up-recovery-os`.

## أمثلة مختصرة (مثال لكل نظام)

### 1) نظام تشغيل الإيرادات — `revenue-operating-system`
- **الإشارة (L2):** الموقع يعرض leads عبر نموذج + متجر + واتساب، بلا أثر لمسار متابعة موحّد.
- **الألم المرجّح:** غالبًا تتوزع الفرص بين القنوات بلا next action، فيتسرب إيراد لا يظهر في تقرير.
- **why_this_system:** يكشف أين تتوقف الفرصة ويبني مسار متابعة وتقرير إيراد يومي.
- **first_mission:** خريطة تسرب الإيراد الأولى + مسار متابعة خلال 7–10 أيام. `starter_price_sar: 4500`.

### 2) نظام القيادة التنفيذية — `executive-command-os`
- **الإشارة (L1):** شركة بفروع/خطوط متعددة، والمؤسس يظهر كصاحب القرار الأوحد في كل التواصل العام.
- **الألم المرجّح:** قد تتأخر القرارات لأن الصورة غير مجمّعة في مكان واحد يومي.
- **why_this_system:** تقرير قيادة يومي يختصر «أهم فرصة / أكبر خطر / القرار التالي» في صفحة.
- **first_mission:** أول تقرير قيادة يومي + خريطة مؤشرات خلال 7–14 يومًا. `starter_price_sar: 5500`.

### 3) نظام استعادة المتابعة — `follow-up-recovery-os`
- **الإشارة (L2):** صفحة دورات تستقبل تسجيلات عبر نموذج + زر واتساب معلن.
- **الألم المرجّح:** في هذا النوع من شركات التدريب غالبًا تُترك بعض الاستفسارات بمتابعة متأخرة.
- **why_this_system:** يبني قائمة متابعة ورسائل تعيد تفعيل الفرص المتروكة.
- **first_mission:** قائمة متابعة + مجموعة رسائل خلال 7 أيام. `starter_price_sar: 3500`.

### 4) نظام عملاء واتساب — `whatsapp-client-os`
- **الإشارة (L2):** رقم WhatsApp Business معلن كقناة أساسية للحجوزات والدعم، وحجم محادثات واضح.
- **الألم المرجّح:** قد تتراكم المحادثات بلا مسار ولا أحد يعرف ما يحتاج ردًا الآن.
- **why_this_system:** يحوّل واتساب إلى مسارات وبطاقات إجراء وسياسة تحويل للإنسان.
- **first_mission:** خريطة مسارات واتساب + بطاقات إجراء خلال 7–10 أيام. `starter_price_sar: 4500`.

### 5) نظام العروض والإثبات — `proposal-proof-os`
- **الإشارة (L1):** شركة استشارات/خدمات B2B تذكر «عروض مخصّصة لكل عميل» بلا قالب موحّد ظاهر.
- **الألم المرجّح:** قد تخرج العروض بصيغ مختلفة وبلا إثبات يطمئن العميل فتتأخر الصفقات.
- **why_this_system:** قالب عرض + Proof Pack يسرّعان قرار العميل.
- **first_mission:** قالب عرض + Proof Pack خلال 5–7 أيام. `starter_price_sar: 3000`.
