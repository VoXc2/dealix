# Data Request — Lead Intelligence Sprint

**Layer:** Service Catalog · Operational Kit
**Owner:** Delivery Analyst
**Status:** Draft
**Last reviewed:** 2026-05-13
**Arabic mirror:** [data_request_AR.md](./data_request_AR.md)

## Context
This file is the **client-facing checklist** of everything Dealix needs before Day 1 kickoff of the Lead Intelligence Sprint. It removes the "we're waiting on you" friction that delays most consulting engagements. It enforces the lawful-basis posture required by `docs/DPA_DEALIX_FULL.md` and the cross-border rules in `docs/CROSS_BORDER_TRANSFER_ADDENDUM.md`, and the PDPL retention regime in `docs/ops/PDPL_RETENTION_POLICY.md`. Until every required item is delivered, the sprint does not start.

## Summary
The client must provide:
1. **One primary lead/account export** (CSV or XLSX).
2. **One ICP description document** (PDF, DOCX, or text).
3. **One current service/offer description** (PDF or text).
4. **One sample of previous outreach** (anonymized OK).
5. **One signed data handling acknowledgement** (auto-generated from intake).

Time budget for the client to assemble: **≤ 1 business day**.

## Item 1 — Primary Lead/Account Export

### Format
- CSV or XLSX. UTF-8 encoded. First row is headers.
- Up to 10,000 rows. Larger volumes are scoped separately.
- Single file. No zipped multi-tab spreadsheets unless agreed in advance.

### Required Columns
| Column | Type | Description | Required |
|---|---|---|---|
| `company` | string | Legal or trading name | yes |
| `contact_name` | string | First + last where available | recommended |
| `email` | string | Primary corporate email | recommended |
| `phone` | string | E.164 format ideally | optional |
| `city` | string | KSA city or GCC city | yes |
| `sector` | string | Industry tag | yes |
| `source` | string | Where the lead came from (event, referral, inbound, etc.) | yes |
| `last_contact` | date | ISO 8601 (YYYY-MM-DD) if known | optional |
| `notes` | text | Any free-text relevant info | optional |

### Optional Columns Improving Score Quality
- `headcount`, `revenue_band`, `decision_maker_role`, `language_preference`, `previous_outcome`.

### What We Will Do With It
- Profile, normalize, deduplicate, score.
- Add columns: `dedupe_id`, `quality_score`, `fit_score`, `intent_score`, `recommended_channel`, `recommendation_reason`.

### What We Will NOT Do With It
- Sell, resell, or share with any third party.
- Train any public AI model on it.
- Retain it past the retention period defined in `docs/DATA_RETENTION_POLICY.md`.
- Process it in any jurisdiction outside what is agreed in `docs/CROSS_BORDER_TRANSFER_ADDENDUM.md`.

## Item 2 — ICP Description Document

### Minimum Content
- Target sectors (top 3).
- Target geography (top 3 cities/regions).
- Target company size (headcount or revenue band).
- Decision-maker role(s) typically targeted.
- Disqualifiers (sectors/sizes you do NOT want).
- Recent wins description (2–3 named or anonymized examples) — these become the score anchor.

### Format
- PDF, DOCX, or plain text.
- 1–3 pages.

### If Not Available
The Sales Engineer drafts an ICP in the Day 2 intake call with the client.

## Item 3 — Current Service/Offer Description

The Dealix outreach drafts must reflect what the client actually sells.

### Minimum Content
- 1-page description of the primary offer.
- Pricing tier or band (rough OK).
- Top 3 buyer pains addressed.
- Top 3 buyer benefits delivered.
- Key proof points the sales team uses.

## Item 4 — Previous Outreach Sample

### Minimum Content
- At least **one** previous outreach email, message, or call script.
- Anonymized fine. Recipient names redacted.
- Sender voice and tone are the goal — Dealix will draft new content in the same register.

### Optional
- Reply rate data if available.
- One example reply that converted (anonymized).

## Item 5 — Data Handling Acknowledgement

Auto-generated by the intake system. Sponsor signs electronically. Confirms:
- The client has the lawful basis to process the dataset for the outreach implied.
- The client agrees to the DPA (`docs/DPA_DEALIX_FULL.md`).
- The client confirms the cross-border posture (`docs/CROSS_BORDER_TRANSFER_ADDENDUM.md`).
- The client acknowledges the retention policy (`docs/ops/PDPL_RETENTION_POLICY.md`).

## Sensitive Data Handling
If intake Q14 returned **yes**:
- Files must be encrypted in transit (SFTP or HTTPS) and at rest (Dealix vault).
- Restricted to a named two-person Dealix delivery team.
- Proof pack auto-anonymizes any field flagged in the intake.
- Retention reduced to project + 30 days (down from project + 365 days).

## Delivery Method
- Preferred: SFTP drop into Dealix-issued folder (path in welcome email).
- Acceptable: encrypted Google Drive or OneDrive shared link with named access.
- Not acceptable: WhatsApp file send, public link, personal email attachment unencrypted.

## SLA
- Client uploads expected within **1 business day** of kickoff.
- For each business day of delay, the sprint timeline slips by 1 business day.
- After 5 business days of delay, Dealix may pause the engagement and invoice a SAR 1,500 restart fee.

## Interfaces

|---|---|---|---|
| Welcome email + SFTP creds | Drop folder ready | Dealix Ops | T-2 days |
| Client upload | Receipt + integrity check | Dealix Analyst | Day 1 |
| Missing items | Chase email | Sales Engineer | Daily until resolved |
| Sensitive-data declaration | Vault placement | Dealix QA + Analyst | Day 1 |

## Metrics
- **On-time data provision** — `% sprints where all data items arrive ≤ 1 business day`. Target ≥ 80%.
- **Data quality** — `% rows requiring manual rework after dedupe`. Target ≤ 15%.
- **Lawful-basis acknowledgement** — `% sprints with signed acknowledgement`. Target = 100%.
