# قائمة تسليم — سبرنت مكتب الدعم بالذكاء الاصطناعي

**الطبقة:** كتالوج الخدمات · العدّة التشغيلية
**المالك:** قائد التسليم — العملاء
**الحالة:** مسودة
**آخر مراجعة:** 2026-05-13
**النسخة الإنجليزية:** [delivery_checklist.md](./delivery_checklist.md)

## السياق
سيناريو التشغيل اليومي لسبرنت مكتب الدعم مدة 14 يومًا. كل سبرنت قابل للتكرار والتدقيق وفق `docs/V14_COMPREHENSIVE_STRATEGIC_PLAN.md`، ومعيار الجودة في `docs/quality/QUALITY_STANDARD_V1.md`، ومصفوفة الإنسان-في-الحلقة في `docs/governance/HUMAN_IN_THE_LOOP_MATRIX.md`. كل خانة تُربَط بحدث في حزمة الإثبات.

## الأدوار
- **قائد التسليم (DL)** — محاسَب واحد.
- **المحلل (AN)** — تجميع الرسائل، التصنيف.
- **قائد الكتابة (CL)** — مكتبة الردود المُقترَحة.
- **مراجِع الحوكمة (GR)** — الحساسية، التصعيد، الأساس القانوني.
- **قائد الدعم (SL)** — نقطة الاتصال الوحيدة لدى العميل.
- **مراجِع الجودة (QA)** — مراجِع مستقل.
- **الراعي (SP)** — القبول النهائي.

## الخطة يومًا بيوم

### T-3 إلى T-1 — قبل الإطلاق
- [ ] إرسال بريد الترحيب. (DL)
- [ ] إنشاء لوحة السبرنت. (DL)
- [ ] إعادة تحقّق العلاوة، تطبيق clinics_playbook إن انطبق. (DL + GR)
- [ ] تهيئة حزمة الإثبات بـ `sprint_initialized`. (DL)

### الأيام 1–3 — التصنيف والفئات
- [ ] مكالمة إطلاق (45 دقيقة) مع SP وSL. (DL)
- [ ] استلام نماذج مُجهَّلة وفحص PII. (AN + QA)
- [ ] تشغيل تجميع الرسائل. (AN)
- [ ] صياغة نظام التصنيف. (AN)
- [ ] SL يراجع النظام ويُحسِّنه. (SL + AN)
- [ ] اختبار تغطية النظام على عيّنة احتجاز: ≥ 90%. (AN)
- [ ] حدث إثبات: `messages_classified = N`, `category_coverage = X%`. (AN)

### الأيام 4–7 — الردود المُقترَحة + FAQ
- [ ] CL يصوغ 1–3 ردود مُقترَحة لكل فئة بالعربية + الإنجليزية. (CL)
- [ ] معايرة نبرة العلامة مقابل نماذج العميل. (CL)
- [ ] هيكلة مُنشِئ FAQ من أعلى الفئات. (CL)
- [ ] كل الردود مُعلَّمة `draft_only`. (CL)
- [ ] لا نصيحة طبية / مالية مُولَّدة؛ الحالات الحساسة تتوقّف عند بطاقة التصعيد. (GR + CL)
- [ ] حدث إثبات: `replies_drafted = N`. (CL)

### الأيام 8–10 — قواعد التصعيد
- [ ] مراجعة سياسة الحالات الحساسة مع Sec/Legal العميل. (GR + SL)
- [ ] تنفيذ توجيه التصعيد: نوع الحالة → إنسان مُسمَّى + علامة مائية. (GR + AN)
- [ ] اختبار توجيه الحالات الحساسة بشكل صحيح في تشغيل تجريبي. (GR + QA)
- [ ] لا حالة حساسة يُردّ عليها تلقائيًا في النظام. (GR + QA)
- [ ] لعملاء العيادات: اختبار التصعيد السريري تحديدًا. (GR)
- [ ] حدث إثبات: `escalations_routed = N`, `sensitive_blocks = N`. (GR)

### الأيام 11–12 — متتبّع SLA
- [ ] توفير متتبّع SLA (Notion/Sheets). (DL)
- [ ] تحميل الأهداف لكل فئة. (DL + SL)
- [ ] توصيل حقول زمن أول ردّ وزمن الحل. (DL)
- [ ] حالات اختبار مُسجَّلة للتحقق من سلوك المتتبّع. (QA)

### الأيام 13–14 — الجودة والتسليم
- [ ] QA يشغّل قائمة الجودة الكاملة. (QA)
- [ ] أي إخفاق → إصلاح نفس اليوم. (DL)
- [ ] صياغة تقرير رؤى الدعم. (DL)
- [ ] صياغة تقرير الإثبات. (DL)
- [ ] التحقق من تجهيل الحقول الحساسة. (QA + GR)
- [ ] مكالمة تسليم (75 دقيقة) مع SP وSL ووكيلَيْن على الأقل. (DL)
- [ ] SP يوقّع مذكرة التسليم. (SP)
- [ ] حدث إثبات: `sprint_delivered`. (DL)
- [ ] تشغيل حركة البيع الإضافي. (DL + CSM)

## الضوابط الشاملة
- لا قدرة إرسال تلقائي مُسلَّمة في MVP أبدًا.
- حزمة الإثبات تُحدَّث باستمرار.
- النماذج الحساسة لا تُنقَل خارج الخزنة المُشفَّرة.
- عملاء العيادات يتطلّبون توقيع GR إضافة إلى QA.

## المقاييس
- **التسليم في الوقت** — الهدف ≥ 95%.
- **تغطية الفئات** — الهدف ≥ 90% على الاحتجاز.
- **دقّة تصعيد الحساسية** — الهدف = 100%.
- **لا إرسال تلقائي مُسلَّم** — الهدف = 100%.
