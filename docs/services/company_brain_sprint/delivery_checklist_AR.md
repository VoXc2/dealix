# قائمة تسليم — سبرنت دماغ الشركة

**الطبقة:** كتالوج الخدمات · العدّة التشغيلية
**المالك:** قائد التسليم — المعرفة
**الحالة:** مسودة
**آخر مراجعة:** 2026-05-13
**النسخة الإنجليزية:** [delivery_checklist.md](./delivery_checklist.md)

## السياق
سيناريو التشغيل أسبوعًا بأسبوع لسبرنت دماغ الشركة مدته 3–4 أسابيع. يجب أن يكون كل سبرنت قابلًا للتكرار والتدقيق وفق `docs/V14_COMPREHENSIVE_STRATEGIC_PLAN.md`، ونظام الحوكمة في `docs/governance/AI_INFORMATION_GOVERNANCE.md`، ومعيار الجودة في `docs/quality/QUALITY_STANDARD_V1.md`. كل خانة تُربَط بحدث في حزمة الإثبات.

## الأدوار
- **قائد التسليم (DL)** — محاسَب واحد.
- **مهندس المعرفة (KE)** — الاستيعاب، الفهرسة، ضبط الاسترجاع.
- **مراجِع الحوكمة (GR)** — الحساسية، الوصول، الأساس القانوني.
- **مالك المعرفة (KO)** — نقطة الاتصال الوحيدة لدى العميل.
- **مراجِع الجودة (QA)** — مراجِع مستقل.
- **الراعي (SP)** — القبول النهائي.

## الخطة أسبوعًا بأسبوع

### T-5 إلى T-1 — قبل الإطلاق
- [ ] إرسال بريد الترحيب مع المجلد المُشارَك وملخص الاستلام وجدول الأسبوع 1. (DL)
- [ ] إنشاء لوحة السبرنت الداخلية في Notion. (DL)
- [ ] إعادة تحقّق العلاوة مقابل الاستلام. (DL + ضابط الهامش)
- [ ] تهيئة حزمة الإثبات بـ `sprint_initialized`. (DL)

### الأسبوع 1 — سجل المصادر + الاستيعاب
- [ ] مكالمة إطلاق (60 دقيقة) مع SP وKO. (DL)
- [ ] استلام قائمة جرد المستندات. (KO + KE)
- [ ] بناء سجل المصادر وفق `docs/ledgers/SOURCE_REGISTRY.md`. (KE + GR)
- [ ] كل مصدر له: المالك، الاستخدام المسموح، الحساسية، الاحتفاظ. (GR)
- [ ] بدء الاستيعاب؛ استخراج النص الخام من كل مستند. (KE)
- [ ] وضع في خزنة مُشفَّرة للمصادر الحساسة. (GR + KE)
- [ ] أحداث إثبات: `documents_indexed = N`, `source_registry_complete`. (KE)

### الأسبوع 2 — الفهرس + ضبط الاسترجاع
- [ ] اختيار استراتيجية التقطيع لكل نوع مستند. (KE)
- [ ] توليد الـ embeddings وفق `AI_MODEL_ROUTING_STRATEGY.md`. (KE)
- [ ] تشغيل اختبارات استرجاع أوّلية بأسئلة نموذجية من KO. (KE + KO)
- [ ] ضبط الاسترجاع: حجم الـ chunk، top-k، حد التشابه. (KE)
- [ ] تنفيذ نموذج الصلاحيات وفق تعريفات المجموعات. (GR + KE)
- [ ] حدث إثبات: `index_tuned`. (KE)

### الأسبوع 3 — RAG + المراجع + الأدلّة-غير-الكافية
- [ ] كتابة قالب prompt يفرض ذكر المصدر. (KE)
- [ ] تنفيذ مسار الأدلّة-غير-الكافية: حين تنخفض درجات الاسترجاع تحت الحد، يقول المساعد "أدلّة غير كافية". (KE)
- [ ] فحص لا-مراجع-مهلوسة: المساعد يذكر فقط مصادر مُسترجَعة فعلًا. (KE + QA)
- [ ] KO يمشي عبر 30 سؤالًا اختباريًا. (KO)
- [ ] تسجيل الأخطاء ومعالجتها. (KE)
- [ ] أحداث إثبات: `answers_with_sources = N`, `insufficient_evidence_responses = N`. (KE)

### الأسبوع 4 — ضبط الوصول + الجودة + التسليم
- [ ] استقبال مجموعات المستخدمين. (GR + KO)
- [ ] اختبار الصلاحيات: كل مجموعة ترى فقط المستندات المسموحة. (GR + QA)
- [ ] سجل التدقيق يلتقط من-سأل-ماذا-وحصل-ماذا. (KE)
- [ ] صياغة تقرير جودة المعرفة: التغطية، الثغرات، الاستيعابات التالية المُقترَحة. (DL)
- [ ] صياغة تقرير الإثبات. (DL)
- [ ] التحقق من تجهيل الحقول الحساسة. (QA)
- [ ] تمرير الجودة النهائي. (QA)
- [ ] مكالمة تسليم (90 دقيقة) مع SP وKO وقادة المجموعات. (DL)
- [ ] SP يوقّع مذكرة التسليم. (SP)
- [ ] أحداث إثبات: `blocked_unauthorized_accesses = N`, `sprint_delivered`. (DL)
- [ ] تشغيل حركة البيع الإضافي. (DL + CSM)

## الضوابط الشاملة
- حزمة الإثبات تُحدَّث باستمرار.
- المحتوى الحساس لا يُنقَل خارج الخزنة المُشفَّرة.
- سجل التدقيق محتفَظ به وفق `DATA_RETENTION_POLICY`.
- لا تُسلَّم مُسلَّمة دون توقيع QA + GR.

## الواجهات
| المدخلات | المخرجات | المالكون | الإيقاع |
|---|---|---|---|
| SOW موقَّع + الاستلام | لوحة السبرنت | DL | T-5 |
| مراجعة أسبوعية | تحديثات لوحة السبرنت | DL | أسبوعيًا |
| نتائج QA + GR | تذاكر إصلاح أو توقيع | QA + GR + DL | نهاية كل أسبوع |
| المساعد + التقرير النهائي | التسليم | DL + SP | الأسبوع 4 |

## المقاييس
- **التسليم في الوقت** — الهدف ≥ 90%.
- **مراجع مهلوسة** — الهدف = 0.
- **إخفاقات ضبط الوصول** — الهدف = 0.
- **نجاح اختبار الصلاحيات** — الهدف = 100%.
