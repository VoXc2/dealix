# نموذج التشغيل — Dealix Operating Model (عربي أولًا)

> كيف تعمل المنظومة يومًا بيوم: الحلقة، الأدوار، البوابات، والافتراضات الآمنة.

---

## 1. الحلقة التشغيلية (The Loop)

```
إشارة/اهتمام → تصنيف → بطاقة إجراء (Draft) → بوابة موافقة المؤسس → تنفيذ يدوي/مُعتمد → تسجيل (ledger) → قياس → تعلّم
```

كل خطوة "خارجية" (إرسال، رابط، دفع، صلاحية) تمر إجباريًا عبر **بوابة موافقة**. لا استثناء.

## 2. الأدوار (الوكلاء)

| الوكيل | المستوى الأقصى | يفعل | لا يفعل |
|---|---|---|---|
| whatsapp client | L4 (بموافقة) | ترحيب/فحص/بطاقات | إرسال تلقائي، طلب أسرار |
| portal | L4/L5 (بموافقة) | روابط/رفع/صلاحيات | تخزين أسرار في سجل |
| revenue execution | L4 (بموافقة) | عرض/إثبات/تجهيز دفع | سعر نهائي/إرسال دفع بلا موافقة |
| delivery | L4 (بموافقة) | handoff/إعداد/تقارير | صفقة بلا handoff |
| renewal / success | L4 (بموافقة) | مسودات مبنية على قيمة | تجديد بلا قيمة، ضغط |
| governance | L2 (نصح) | مراجعة/تنبيه | تجاوز قرار بشري |

تفصيل المستويات L1–L5 في `AGENTS.md` §4، ومتوافق مع `company_os/governance/agent_permissions.md`.

## 3. الافتراضات الآمنة (Safe Defaults)

```yaml
dry_run: true            # محاكاة، لا أثر خارجي
approval_required: true  # تحتاج موافقة المؤسس
send_enabled: false      # الإرسال الخارجي مُعطّل في v1
```

لا يُفعَّل `send_enabled` إلا بعد اكتمال كل البوابات (موافقة + تسليم بشري حيث يلزم).

## 4. أنواع البوابات

| البوابة | متى | المرجع |
|---|---|---|
| `gate: approval` | أي إرسال/رابط/إجراء L4 | طابور الموافقة |
| `gate: portal` | أسرار/ملفات/صلاحيات/مراجعات | البوابة الآمنة |
| `gate: human_handoff` | سعر نهائي/دفع/قانوني/شكوى | التسليم البشري |

## 5. مقياس الأدلة في القرار
لا قرار يُبنى على `evidence_level=none`. التجديد/الترقية تتطلب `client_data` فأعلى. (التفصيل في `AGENTS.md` §6.)

## 6. التسجيل والقياس
- كل إجراء → `company_os/governance/ai_action_ledger.jsonl`.
- كل قرار مؤسس → `reports/founder/DECISION_LOG.md`.
- المقاييس → `reports/whatsapp/WHATSAPP_METRICS.md` ومراجعات التحويل.
