# نظام أمر المؤسس — Founder Command System (عربي أولًا)

> كيف يقود المؤسس المنظومة كلها من خلال **أمر واحد يومي**، **مراجعة أسبوعية**، و**سجل قرار** — دون أن يغرق في التفاصيل.

---

## 1. الفلسفة
الوكلاء يجهّزون؛ المؤسس **يقرّر**. كل ما يحتاجه المؤسس يصل إليه مُلخّصًا وقابلًا للتنفيذ بضغطة: approve / reject / edit / copy / mark-sent / nurture / do-not-contact / request-handoff.

## 2. الإيقاع اليومي — أمر المؤسس الواحد
المصدر: `reports/founder/DAILY_SUPER_COMMAND.md` (تغذّيه كل الطوابير).

يحتوي يوميًا على:
- حالة إنتاج المسودات.
- أهم إجراءات الموافقة.
- الردود الإيجابية الجديدة.
- بطاقات واتساب بانتظار الموافقة.
- طابور العروض + تسليمات الدفع.
- تسليمات (Sales→Delivery) + فرص التجديد.
- تنبيهات الخصوصية/الأمن + لقطة النقد/الـpipeline.
- **قرار حرج واحد** لليوم.

## 3. الإيقاع الأسبوعي — مراجعة المجلس
المصدر: `reports/founder/WEEKLY_BOARD_REVIEW.md`: المقاييس، صحة العملاء، المخاطر، الخصوصية، خط الإيراد، والقرارات الكبرى.

## 4. سجل القرار
كل قرار مؤسس (اعتماد/رفض/تعديل/تسعير/تسليم بشري) يُدوّن في `reports/founder/DECISION_LOG.md` مع: الوقت، النوع، المخاطرة، القرار، السبب. هذا يبني الذاكرة المؤسسية ويغذّي حلقة التعلّم.

## 5. مصفوفة التصعيد (متوافقة مع company_os)

| نوع القرار | المُعتمِد | ملاحظة |
|---|---|---|
| رسائل واتساب/تواصل | المؤسس | بطاقة إجراء |
| تسعير < 5,000 ريال | المؤسس | — |
| تسعير ≥ 5,000 ريال | المؤسس + تهدئة 24 ساعة | — |
| تسليم دفع | المؤسس + تسليم بشري | لا إرسال تلقائي |
| صلاحيات بيانات | المؤسس + فحص PDPL | عبر البوابة |
| قانوني/تعاقدي | تسليم بشري | لا وعد آلي |

(المرجع: `company_os/governance/agent_permissions.md`.)
