# منهج دياليكس — نموذج التركيب القابض

**الطبقة:** Holding · Compound Holding Model
**المالك:** رئيس Dealix Services + رئيس Dealix Core
**الحالة:** مسودة
**آخر مراجعة:** 2026-05-13
**النسخة الإنجليزية:** [DEALIX_METHOD.md](./DEALIX_METHOD.md)

## السياق
منهج دياليكس هو **المنهجية العامة** التي تُطبّقها كل وحدة على كل تكليف. هو تتابع الأفعال خلف العلامة: تشخيص ← تصميم ← بناء ← حوكمة ← تحقّق ← تسليم ← إثبات ← تشغيل ← تركيب. وهو المعيار الذي تقيس عليه [`DEALIX_STANDARD_METRICS`](./DEALIX_STANDARD_METRICS.md). وهو التعبير التشغيلي عن `docs/DEALIX_OPERATING_CONSTITUTION.md`، وله مرآة عربية للسوق السعودي عبر `docs/strategic/DEALIX_MASTER_OPERATING_MODEL_AR.md`.

## المراحل التسع

```
Diagnose → Design → Build → Govern → Validate → Deliver → Prove → Operate → Compound
```

### 1. تشخيص
- **الغرض:** إيجاد الفرصة الأعلى رافعة الآمنة والقابلة للبيع.
- **التسليمات:** خريطة فرص، قائمة قصيرة محسوبة، مقاييس أساس، جرد مخاطر.
- **معيار البوّابة:** للفرصة الأولى أساس قابل للقياس؛ تقييم أثر PDPL.

### 2. تصميم
- **الغرض:** تحديد مسار العمل، شكل البيانات، الحوكمة، فرضية العائد.
- **التسليمات:** مخطّط، عقد بيانات، ارتباطات السياسات، خطة KPI.
- **معيار البوّابة:** المسار معتمد من العميل + الحوكمة.

### 3. بناء
- **الغرض:** بناء المسار على Core OS لا على بنية مخصّصة.
- **التسليمات:** مسار حيّ في مساحة العميل؛ سياسات مهيّأة؛ منظومة تقييمات.
- **معيار البوّابة:** اجتياز التقييمات الأوّلية؛ لا بنية يتيمة خارج Core OS.

### 4. حوكمة
- **الغرض:** ربط المسار بالسياسات والموافقات والتدقيق وضوابط PDPL.
- **التسليمات:** ارتباطات السياسات؛ مصفوفة الموافقات؛ سجل تدقيق مُفعَّل؛ مسار DSR.
- **معيار البوّابة:** تأكيد Runtime لتغطية 100% لنداءات النموذج.

### 5. تحقّق
- **الغرض:** إثبات الجودة قبل الحمل الحيّ.
- **التسليمات:** نتائج تقييم؛ تشغيل تجربة؛ تقرير بوّابة جودة.
- **معيار البوّابة:** QA ≥ 85؛ معدّل تقييم فوق العتبة؛ لا عيوب حرجة.

### 6. تسليم
- **الغرض:** التحوّل إلى الإنتاج وتدريب فريق العميل.
- **التسليمات:** تحوّل إنتاج؛ تدريب؛ SLA و on-call؛ خطة تبنّي.
- **معيار البوّابة:** التزام SLA لأسبوع؛ ≥ 80% من المستخدمين نشطون.

### 7. إثبات
- **الغرض:** التقاط العائد كـ Proof Pack ودراسة حالة.
- **التسليمات:** Proof Pack (`docs/PROOF_PACK_V6_STANDARD.md`)؛ أدلّة ROI؛ مسودة دراسة.
- **معيار البوّابة:** الإيداع في Proof Ledger؛ توقيع العميل على الأرقام.

### 8. تشغيل
- **الغرض:** تشغيل المسار على احتفاظ شهري؛ ضبطه؛ توسيعه.
- **التسليمات:** SoW احتفاظ؛ تقرير قيمة شهري؛ قائمة مسارات جديدة.
- **معيار البوّابة:** توقيع الاحتفاظ خلال 60 يوماً من التحوّل.

### 9. تركيب
- **الغرض:** استخراج أصول قابلة لإعادة الاستخدام وتنميط العمل المتكرّر إلى Core OS.
- **التسليمات:** قيود Capital Ledger؛ مرشّحات تنميط؛ رؤى منشورة.
- **معيار البوّابة:** ≥ 2 أصول مسجَّلة؛ ≥ 1 مرشّح تنميط.

## كيف يُفرض المنهج

- **بوّابة جودة** عند نهاية كل مرحلة (`docs/product/QUALITY_AS_CODE.md`).
- **تسليمات قالبية** في Core OS لكل مرحلة — لا يُمكن تخطّيها بصمت.
- **Proof Pack** شرط لإصدار فاتورة المرحلة الأخيرة.
- **Productization Ledger** يجب أن يُسجّل قيداً واحداً لكل تكليف (`docs/product/PRODUCTIZATION_LEDGER.md`).

## التموضع العام
يُنشر المنهج خارجياً بوصفه منهجية دياليكس الموقَّعة، كما تنشر الشركات الفئوية منهجياتها. ويرسو عليه:

- العروض البيعية والمقترحات.
- قوالب SoW السبرنت والتجربة والاحتفاظ.
- منهج الأكاديمية.
- شهادة الشركاء.
