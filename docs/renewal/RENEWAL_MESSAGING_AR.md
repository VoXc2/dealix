# رسائل التجديد — Renewal Messaging

> **المستوى:** L3 Draft → L4 Act with Approval | **اللغة:** عربي أولًا | **الموافقة:** موافقة المؤسس إلزامية قبل أي إرسال | **القاعدة:** خطوة واحدة — بلا ضغط — مبنية على قيمة مُسلَّمة

---

## مبادئ رسائل التجديد

قبل كتابة أي رسالة تجديد، هذه القواعد غير قابلة للخرق:

| المبدأ | التطبيق |
|---|---|
| **القيمة أولًا** | كل رسالة تبدأ بما أنجزناه معًا، لا بما نريده |
| **خطوة واحدة فقط** | اقتراح واحد محدد، لا قائمة خيارات |
| **بلا ضغط** | لا "عرض محدود المدة"، لا "لا تفوّت"، لا إلحاح |
| **بلا ضمان ROI** | نستشهد بما قِسناه، لا نَعِد بما سيحدث |
| **موافقة المؤسس** | الرسالة لا تُرسل إلا بعد `approved_by` موثَّق |
| **مرجع REN-* محدد** | كل رسالة مرتبطة بسجل تجديد موجود في البيانات |

---

## 1. هيكل رسالة التجديد

كل رسالة تجديد تتبع هذا الهيكل:

```
[1] الافتتاح — إقرار بالتقدم المُحقَّق
[2] القيمة المُسلَّمة — الرقم + المصدر (بلا مبالغة)
[3] الخطوة الواحدة المقترحة — وصف بسيط ومحدد
[4] سؤال مفتوح — نمنح العميل زمام القرار
[5] الإغلاق — دافئ وبلا ضغط
```

## 2. قوالب الرسائل (عربي)

### القالب أ — تجديد بعد أول تسليم ناجح

**المحفز المناسب:** `first_successful_workflow` أو `weekly_value_proof`
**evidence_level المطلوب:** `measured` أو `verified`

---

> [اسم العميل]،
>
> نهاية الأسبوع الأول من تشغيل الـ Workflow جاءت بأرقام تستحق الوقفة.
>
> ما قِسناه معًا: **[مقياس رئيسي] تحوَّل من [before] إلى [after]** — وهذا رقم من بياناتكم الفعلية، لا تقدير.
>
> الخطوة التالية التي أراها مناسبة: **[خطوة واحدة محددة من الكتالوج]**.
>
> هل تودّون مناقشة ذلك في 15 دقيقة هذا الأسبوع؟
>
> [التوقيع]

**مثال مكتمل — Digital Rise Agency (REN-1001):**

> خالد،
>
> نهاية الأسبوع الأول من تشغيل workflow المتابعة جاءت بأرقام تستحق الوقفة.
>

>
> الخطوة التالية التي أراها مناسبة: **الاستمرار بباقة شهرية لتثبيت هذا المكسب وتوسيعه.**
>

### القالب د — اقتراح ترقية (Upsell) بناءً على UPS-*

**المحفز المناسب:** أي محفز مع توافر UPS-* معتمد
**evidence_level المطلوب:** `measured` أو `verified`

---

> [اسم العميل]،
>
> بعد [عدد] أسابيع من العمل معًا، أرى فرصة لأخذ الأمور خطوة أبعد.

## 3. ما يُحظر كتابته في رسائل التجديد

| العبارة | السبب |
|---|---|
| "هذا العرض لن يتكرر" | إلحاح مصطنع — محظور |
| "أنتم تفقدون X ريال كل يوم" | ضغط وتخويف — محظور |
| "ضمانًا لنتائج ممتازة" | ضمان ROI — محظور (AGENTS.md §2 و§7) |
| "وفّرنا لكم X ريال" | ادعاء غير مُثبَّت — إلا إذا `evidence_level: verified` |
| قائمة خيارات متعددة | يُشتّت ويُربك — خطوة واحدة فقط |
| "نأمل استمراركم" | تعبير ضعيف يُضعف المصداقية |
| أرقام غير مُستندة لـ WVR-* | ادعاء بلا دليل — محظور |

## 4. مسار اعتماد رسالة التجديد

```
1. تحقق من شروط التجديد (value_proof, evidence_level, cites_delivered_value)
2. اختر القالب المناسب للمحفز
3. اكتب الرسالة (L3 Draft) — الذكاء الاصطناعي يُسودّ
4. حدّد رقم REN-* المرتبط بالرسالة
5. أرسل المسودة للمؤسس للمراجعة
6. المؤسس يراجع ويعتمد → يُسجّل approved_by + approved_at في REN-*
7. المؤسس يُرسل الرسالة يدويًا (L4 Act with Approval)
8. تحديث status في REN-* → sent_manually
9. تسجيل الإجراء في ai_action_ledger
```

**v1 — لا إرسال آلي:** `send_enabled = false` في كل المسودات. الإرسال يدوي دائمًا.

## 5. التعامل مع الرد السلبي

إذا رفض العميل أو لم يرد:

| الحالة | الإجراء |
|---|---|
| لم يرد بعد 5 أيام | رسالة متابعة دافئة (غير مُلحّة) — بموافقة المؤسس |
| رفض صريح | تحديث REN-* → `status: declined` + توثيق السبب |
| رفض مع تعليق | توثيق التعليق + تحليل السبب + خطة تحسين |
| طلب تأجيل | تحديث REN-* → `status: draft` + تحديد توقيت جديد |

**القاعدة:** الرفض ليس فشلًا — هو معلومة. توثّقه وتعلّم منه.
