# قاعدة الخطوة التالية الواحدة لكل مرحلة — Dealix

المبدأ: لكل فرصة في كل مرحلة **خطوة تالية واحدة فقط** (`next_action`)، واضحة وقابلة
للتنفيذ، ومالك معروف. لا فرصة بلا خطوة تالية. هذا يطابق حقل `next_action` في
`schemas/opportunity.schema.json` وقيمه في `data/commercial/opportunities.jsonl`
(مثل `prepare_proposal`، `book_discovery`).

## لماذا خطوة واحدة؟
- يمنع التشتّت ويبقي الـ pipeline متحرّكاً.
- يجعل مراجعة الـ pipeline سريعة (انظر `reports/commercial/PIPELINE_REVIEW.md`).
- يفصل ما هو **آلي** (مسودّات، أبحاث، طابور) عمّا يتطلّب **موافقة المؤسّس** (إرسال، تسعير، عرض).

## جدول الخطوة التالية لكل مرحلة
| المرحلة | الخطوة التالية الواحدة | المالك | آلي/موافقة |
|--------|------------------------|--------|-----------|
| `signal_detected` | إنشاء سجل فرصة وبدء البحث | النظام | آلي |
| `researched` | تأهيل مبدئي حسب القواعد | النظام | آلي |
| `qualified` | تجهيز مسودّة تواصل أولى | النظام | آلي (مسودّة فقط) |
| `drafted` | عرض المسودّة على المؤسّس للموافقة | المؤسّس | موافقة |
| `approved_for_outreach` | إدراج في طابور الإرسال وفق سياسة التسليم | المؤسّس | موافقة (إرسال) |
| `contacted` | متابعة مجدولة وانتظار الرد | النظام | آلي (مسودّة متابعة) |
| `replied` | حجز مكالمة اكتشاف (`book_discovery`) | المؤسّس | موافقة |
| `discovery_scheduled` | إجراء مكالمة الاكتشاف | المؤسّس | يدوي |
| `discovery_completed` | تجهيز العرض إن اكتمل التأهيل (`prepare_proposal`) | المؤسّس | موافقة |
| `proposal_needed` | بناء مسودّة عرض مطابقة للمنتج والـ metric والنطاق | النظام | آلي (مسودّة) |
| `proposal_sent` | متابعة العرض وجدولة نقاش | المؤسّس | موافقة |
| `negotiation` | معالجة الاعتراضات + اعتماد السعر النهائي | المؤسّس | موافقة (تسعير) |
| `payment_handoff` | تسليم تفاصيل الدفع وتأكيد الالتزام | المؤسّس | موافقة |
| `won` | بدء تسليم العمل | المؤسّس | يدوي |
| `delivery_handoff` | نقل النطاق المتفق عليه لفريق التسليم | المؤسّس | يدوي |
| `active_delivery` | متابعة التنفيذ ضمن النطاق | المالك | يدوي |
| `renewal_candidate` | تقييم التجديد/الترقية (`DLX-L5`) | المؤسّس | موافقة |
| `renewed` | تأكيد الاشتراك وتحديث الخطة | المؤسّس | موافقة |
| `lost` | تسجيل سبب الخسارة + قرار `nurture` إن مناسب | النظام | آلي |
| `nurture` | إعادة تواصل لاحقة عند إشارة جديدة | النظام | آلي |
| `do_not_contact` | إيقاف كل تواصل وتثبيت السبب | النظام | آلي (إيقاف) |

## قواعد عامة
1. لا يجوز ترك `next_action` فارغاً لأي فرصة نشطة.
2. كل خطوة تتضمّن إرسالاً خارجياً أو تسعيراً أو عرضاً → **موافقة مؤسّس إلزامية**.
   الافتراضات: `dry_run=true`، `approval_required=true`، `send_enabled=false`.
3. الانتقال إلى `proposal_needed` ممنوع ما لم تكن الفرصة `qualified` ومُطابَقة لمنتج
   ولها `success_metric` ونطاق واضح (انظر `PROPOSAL_APPROVAL_POLICY_AR.md`).
4. إن ظهرت إشارة استبعاد → الخطوة التالية هي `disqualify` والانتقال إلى `lost`/`do_not_contact`.
5. كل تغيير في `next_action` يُسجّل مع التاريخ والمالك.

## «مثال توضيحي»
- OPP-001 (Digital Rise Agency) في `discovery_completed` → `next_action = prepare_proposal`.
- OPP-003 (TrainMe KSA) في `replied` وغير مؤهّل → `next_action = book_discovery`.
