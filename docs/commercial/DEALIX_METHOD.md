# منهجية Dealix — The Dealix Method

> **القاعدة:** Dealix لا يبيع أداة؛ يبيع منهجاً من ثماني خطوات. المنهجية نفسها تُعاد عبر الموقع، والـ demo، والـ Proof Pack — صياغة واحدة، ثلاثة استخدامات.

## النطاق / Scope

- **هذا الملف:** منهجية بيعية من 8 خطوات (Discover → Expand)، لكل خطوة غرضها ومدخلها ومخرجها وأي non-negotiable تحرسه.
- **ليس هذا الملف:** قائمة أسعار، ولا سكربت محادثة، ولا مواصفة هندسية.

## الخطوات الثماني — The 8 steps

```
Discover → Diagnose → Design → Draft → Approve → Deliver → Prove → Expand
 نفهم   →  نشخّص  → نصمّم  → نجهّز →  نوافق  →  نسلّم  → نثبت  → نوسّع
```

كل خطوة تُغذّي التي تليها؛ لا قفز فوق خطوة، ولا تقدّم بلا دليل من السابقة (قاعدة السلم الذهبية).

## تفصيل الخطوات

### 1) Discover — نفهم

| العنصر | التفاصيل |
|--------|----------|
| **الغرض** | فهم سياق العميل وألمه الحقيقي قبل أي عرض |
| **المدخل** | محادثة أولى، إجابات التشخيص المجاني rung [0] |
| **المخرج** | ملف فهم مختصر: القطاع، الـ ICP، الألم الأوضح |
| **non-negotiable يحرسه** | لا إجابة بلا مصدر — الفهم مبني على مدخلات موثقة لا افتراضات |

### 2) Diagnose — نشخّص

| العنصر | التفاصيل |
|--------|----------|
| **الغرض** | تحديد فجوات الإيراد والفرص بناءً على بيانات حقيقية |
| **المدخل** | عيّنة بيانات العميل، حالة الـ pipeline الحالي |
| **المخرج** | تقرير تشخيصي: أعلى فجوات، أعلى فرص، توصية درجة |
| **non-negotiable يحرسه** | لا scraping — التشخيص يعتمد بيانات العميل المرخّصة فقط |

### 3) Design — نصمّم

| العنصر | التفاصيل |
|--------|----------|
| **الغرض** | تصميم خطة تسليم وعرض مُوجَّه إلى درجة معتمدة |
| **المدخل** | تقرير التشخيص، مصفوفة العروض الداخلية |
| **المخرج** | نطاق scope محدد + درجة سلم معتمدة + خطة خطوات |
| **non-negotiable يحرسه** | لا وعود ROI أو نتائج مضمونة — التصميم يصف عملاً، لا أرقاماً مضمونة |

### 4) Draft — نجهّز

| العنصر | التفاصيل |
|--------|----------|
| **الغرض** | تجهيز المخرجات (رسائل، تقارير، playbook) كمسودات |
| **المدخل** | النطاق المعتمد، بيانات العميل |
| **المخرج** | مسودات outreach وتقارير جاهزة للمراجعة — لا مُرسَلة |
| **non-negotiable يحرسه** | لا مخرج AI للعميل بلا QA — كل مسودة تمرّ على مراجعة بشرية |

### 5) Approve — نوافق

| العنصر | التفاصيل |
|--------|----------|
| **الغرض** | الحصول على موافقة بشرية صريحة قبل أي إجراء خارجي |
| **المدخل** | المسودات المراجَعة |
| **المخرج** | سجل موافقة موثّق (`approved_manual`) قبل أي إرسال |
| **non-negotiable يحرسه** | موافقة بشرية لكل إجراء خارجي — لا live send، لا cold WhatsApp / LinkedIn automation |

### 6) Deliver — نسلّم

| العنصر | التفاصيل |
|--------|----------|
| **الغرض** | تنفيذ التسليم وفق النطاق ومعيار SOAEN |
| **المدخل** | المخرجات الموافق عليها |
| **المخرج** | تسليم منفّذ، فاتورة بعد قيمة حقيقية |
| **non-negotiable يحرسه** | لا live charge — الدفع بتأكيد يدوي؛ لا إجراء خارجي بلا موافقة |

### 7) Prove — نثبت

| العنصر | التفاصيل |
|--------|----------|
| **الغرض** | بناء Proof Pack مُتحقَّق يوثّق ما حدث فعلاً |
| **المدخل** | نتائج التسليم، أحداث الإثبات (evidence events) |
| **المخرج** | Proof Pack بعد QA + رؤية بأسلوب الحالة (anonymized) |
| **non-negotiable يحرسه** | لا fake proof، ولا تقدّم مرحلة بلا دليل مُتحقَّق؛ لا PII في السجلات |

### 8) Expand — نوسّع

| العنصر | التفاصيل |
|--------|----------|
| **الغرض** | تحويل الإثبات إلى upsell، أو إحالة، أو شريك |
| **المدخل** | Proof Pack، رضا العميل |
| **المخرج** | مقترح ترقية إلى درجة أعلى، إحالة، أو تعريف شريك |
| **non-negotiable يحرسه** | لا تقدّم مرحلة بلا دليل — الترقية تُفتح فقط بعد إثبات الدرجة السابقة |

## ملخص حراسة الـ non-negotiables

| الخطوة | الـ non-negotiable المحروس |
|--------|-----------------------------|
| Discover | لا إجابة بلا مصدر |
| Diagnose | لا scraping |
| Design | لا وعود ROI / نتائج مضمونة |
| Draft | لا مخرج AI بلا QA |
| Approve | موافقة بشرية / لا live send |
| Deliver | لا live charge |
| Prove | لا fake proof / لا PII في السجلات |
| Expand | لا تقدّم مرحلة بلا دليل |

## إعادة استخدام المنهجية / Reuse

المنهجية الواحدة تُستخدم في ثلاثة أماكن بصياغة متطابقة: على **الموقع** كقصة كيف نعمل، في الـ **demo** (12 دقيقة) كخريطة المحادثة، وفي الـ **Proof Pack** كهيكل التوثيق. هذا يجعل ما يسمعه العميل وما يراه وما يستلمه شيئاً واحداً.
