# Customer Portal Token — Setup Guide

**Status:** PRODUCTION_READY — token-based access via URL param
**Audience:** Sami (founder) · paying customers
**Companion:** `landing/customer-portal.html` · `landing/assets/js/access-gate.js` · `api/routers/customer_company_portal.py`
**Wave:** 7.5 §24.4

> Customer Portal access is token-based via URL parameter. Each paying customer gets a unique token generated by `dealix_customer_onboarding_wizard.py`. Token grants read-only access to their Operations Console, Executive Command Center, and Decision queue.

---

## How it works

### Customer side

1. Customer receives unique URL after onboarding:
   ```
   https://dealix.me/customer-portal.html?org=<handle>&access=<token>
   ```

2. First visit:
   - JavaScript stores token in `localStorage`
   - Customer can refresh page; token persists

3. Subsequent visits:
   - Customer just visits `https://dealix.me/customer-portal.html?org=<handle>`
   - localStorage retrieves the token automatically

### Sami side (token generation)

After onboarding wizard runs:

```
data/customers/<handle>/
├── customer_portal_token.txt   ← single-line token (gitignored)
├── integration_plan.md
└── env_vars_railway.txt
```

Token format: `dealix-cust-<24 url-safe chars>`

Sami shares the URL with customer via:
- 1Password vault link (recommended)
- Encrypted email (PGP)
- Personal WhatsApp message (acceptable for MVP, encrypted at-rest by Meta)

**NEVER share via:**
- Public GitHub issue
- Public Twitter / LinkedIn
- Plaintext email

## Token rotation (when needed)

### Voluntary rotation (every 90 days, recommended)

```bash
# Re-run wizard with same customer-handle (will rotate token)
python3 scripts/dealix_customer_onboarding_wizard.py \
  --customer-handle acme-real-estate \
  --inputs-json data/customers/acme-real-estate/_last_inputs.json
```

New token → new file `data/customers/<handle>/customer_portal_token.txt` (overwrites old).

Sami sends new URL to customer; old token becomes invalid.

### Emergency rotation (token leaked)

1. Sami immediately runs wizard rotation (above)
2. Sends new URL to customer with note: "rotated for security"
3. Updates `data/customers/<handle>/_audit.jsonl` with rotation event
4. If breach suspected → log in `docs/PDPL_DATA_SUBJECT_REQUEST_SOP.md` and notify lawyer per breach response plan

## State-based UX (3 modes)

| State | URL pattern | Data displayed |
|---|---|---|
| **DEMO** | `/customer-portal.html` (no params) | Realistic demo data showing every section |
| **SIGNED_UP** | `/customer-portal.html?org=<handle>` | Their company name + sector-personalized demo overlay |
| **ACTIVE** | `/customer-portal.html?org=<handle>&access=<token>` | Live data via existing APIs |

Token is what unlocks ACTIVE mode.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| Page shows DEMO MODE | Missing `?access=` param | Customer needs to use full URL (org + access) |
| 404 / "Customer not found" | Wrong handle | Verify handle case-sensitivity (always lowercase) |
| 401 / token rejected | Token rotated | Re-issue from `data/customers/<handle>/customer_portal_token.txt` |
| localStorage cleared | Customer cleared browser data | Re-share URL; localStorage repopulates on visit |
| Token sharing across browsers | Customer wants access on mobile | Same URL works everywhere; localStorage scoped per device |

## Hard rules

- ❌ NO public token publishing (never in marketing copy or screenshots)
- ❌ NO logging tokens in stdout / audit log (only token IDs / hashed values)
- ❌ NO sharing tokens via insecure channels (Slack public channels, plaintext SMS)
- ❌ NO multi-customer reuse (1 token = 1 customer)
- ✅ Tokens stored in gitignored `data/customers/<handle>/`
- ✅ Tokens regenerated per onboarding (uniqueness guaranteed by `secrets.token_urlsafe`)
- ✅ Customer can revoke voluntarily (Sami runs rotation)
