## What works today (manual workaround)

When a customer wants leads from email forwarded into Dealix:

### Option A — Forward-to-Sami (covers customers 1-3)

1. Customer sets up Gmail/Outlook forward rule:
   - From: their inbound lead inbox (e.g., `info@acme-real-estate.sa`)
   - Forward to: `dealix-leads@<sami-domain>` (Sami's email)
   - Mark forwarded copy as Read

2. Sami eyeballs each forwarded lead, runs the lead through Dealix:

```bash
# Manual ingestion via existing script (Wave 6)
python3 scripts/dealix_first_prospect_intake.py \
  --prospect-handle <derived> --sector <sector> --relationship warm \
  --note "Forwarded from <customer>"
```

3. Dealix Lead Inbox shows the entry; Sami follows the standard pipeline.

**Pros:** zero new code; works today.
**Cons:** founder-bottleneck; doesn't scale past 3 customers.

### Option B — Customer keeps email primary, only escalates hot leads via WhatsApp

If customer's email volume is low (~5-10 leads/week), simpler approach:

1. Customer reads inbound emails themselves
2. Customer copies hot leads to a private WhatsApp group with Sami
3. Sami pulls from WhatsApp into Dealix using inbound webhook (already live)

This gives Dealix only the leads worth processing — no noise.

## What activates with `lead_intake_email` TARGET service (Wave 8)

When triggered (per Article 11 — first customer asks), the following becomes available:

### Step-by-step (future state)

1. Customer creates email alias: `dealix-<handle>@dealix.me` (auto-allocated by wizard)
2. Customer sets up Gmail/Outlook forward to that alias
3. Dealix runs IMAP polling every 5 min OR webhook on inbound (provider TBD: SendGrid/Resend/Mailgun)
4. Each email → MIME parse → classification (lead vs spam vs reply) → Lead Inbox

### Endpoints (when activated)

- `POST /api/v1/webhooks/email/inbound` — provider-side webhook
- `GET /api/v1/leads?source=email&handle=<customer>` — query inbound leads

## Current outbound (already PRODUCTION_READY)

For sending email replies (after approval):

1. Sami connects Gmail OAuth via `/api/v1/email/connect/gmail` (founder side)
2. Drafts generated by Dealix → preview on `/decisions.html`
3. Sami clicks Approve → `POST /api/v1/email/send-approved` → goes through compliance check (line 200-216 in `email_send.py`)
4. Compliance gate: opt-out · risk_score · suppression list · 24h-window · daily-cap
5. If passed → Gmail sends; audit log written

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| "Email not configured" | Gmail OAuth not refreshed | Re-run `/api/v1/email/connect/gmail` |
| Outbound send blocked | Compliance gate triggered | Check `email_send_log` for `blocked_compliance` reason |
| Customer's forwarded leads not appearing | Manual workflow — Sami forgot to ingest | Reminder in Friday review (per `SALES_OPS_SOP.md` §10) |

## Hard rules (immutable in code)

- ❌ NO_BLAST: every email is 1-to-1 with explicit approval
- ❌ NO_COLD_OUTREACH: opt-out + suppression list checked before every send
- ❌ NO_LIVE_SEND in test env (`APP_ENV=test` blocks all outbound)
- ✅ Every send has audit_log entry
- ✅ Customer's email forwarded data is processor-only (controller stays customer)
