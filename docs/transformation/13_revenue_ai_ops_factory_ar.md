# 13 — Revenue + AI Ops Factory (Arabic Operating Blueprint)

## Objective

تحويل Dealix إلى شركة تشغيل إيراد وذكاء اصطناعي محكومة، تعمل حول المؤسس، وتمنع أي تنفيذ خارجي غير معتمد.

## Operating identity

Dealix = **AI-native Revenue Operating Company**

المعادلة التنفيذية:

- Founder-led trust
- Proof-led funnel
- Automated revenue ops
- Agentic execution loops
- Disciplined approval system
- Evidence ledger
- Delivery-to-product learning
- Partner leverage
- Security-by-design

## Golden operating doctrine

كل نشاط يجب أن يمر عبر السلسلة التالية:

`Signal → Source → Approval → Action → Evidence → Decision → Value → Asset`

أي تنفيذ خارج هذه السلسلة يعتبر drift تشغيلي.

## Layered operating design

1. Signal Layer
2. Agent Layer
3. Workflow Layer
4. Approval Layer
5. Action Layer
6. Evidence Layer
7. Learning Layer
8. Productization Layer

## Automation law (three levels)

### Level 1 — Fully Automated

يسمح بتنفيذ داخلي كامل (capture, scoring, drafting, reminders, evidence logging) بدون إرسال خارجي.

### Level 2 — Agent-Assisted

الوكيل يقترح (ICP, message, offer, pricing band, next action, upsell, partner) والقرار النهائي للمؤسس.

### Level 3 — Founder Approval Required

لا يسمح بالتنفيذ قبل اعتماد صريح:

- أول إرسال خارجي
- إرسال scope
- إرسال invoice
- التشخيص النهائي
- نشر case study
- أي claim أمني/امتثال
- أي فعل يمس المال/السمعة

## Core policies (non-negotiable)

- No Autonomous External Send
- No cold WhatsApp
- No LinkedIn automation
- No unsupported claims
- Every claim requires source (or `is_estimate=true`)
- Every risky action requires approval
- Every action must be traceable in Evidence Ledger

## Core loops to operate weekly

1. Demand Loop
2. Sales Loop
3. Delivery Loop
4. Product Learning Loop
5. Partner Loop
6. Governance Loop
7. Weekly CEO Loop

## Approval Center (mandatory)

Minimal approval types:

- `external_message`
- `scope_send`
- `invoice_send`
- `diagnostic_final`
- `case_study`
- `security_claim`
- `agent_tool_action`
- `refund_or_discount`
- `partner_terms`

Risk levels:

- Low: internal-only
- Medium: controlled external follow-ups
- High: scope/invoice/public claims
- Critical: security claims, client data transfer, autonomous external attempts

## Evidence Ledger (moat)

الدفتر يجب أن يسجل الأحداث الإيرادية والتشغيلية كاملة (lead → meeting → scope → invoice → payment → delivery → proof → upsell).

الحد الأدنى لحقول الحدث:

- `event_id`
- `timestamp`
- `account_id`
- `contact_id`
- `event_type`
- `source`
- `summary`
- `approval_required`
- `confidence`
- `created_by`

## Implementation order (strict)

1. Sprint 1: Revenue control base (stages, scoring, approval queue, evidence events)
2. Sprint 2: Agent drafting (outreach, brief, scope, follow-up, content)
3. Sprint 3: Delivery automation (onboarding, diagnostic checklist, proof pack draft)
4. Sprint 4: Governance automation (claim checker, approval enforcement, payment gating)
5. Sprint 5: Productization only after repeatability signals

## Build vs Automate rule

- New workflow: manual + checklist
- Repeated twice: template
- Repeated three times: automation
- Repeated by 2+ clients: internal module
- Repeated by 3+ clients with retainer: product feature

## Control artifact

Machine-readable contract:

- `dealix/transformation/revenue_ai_ops_factory.yaml`

This blueprint is intentionally approval-first, evidence-first, and anti-drift by design.
