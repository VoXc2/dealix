# Dealix CEO/Board Operating Plan — Governed Revenue & AI Operations

## 1) الهوية النهائية (Category Positioning)

**EN:** Dealix — Governed Revenue & AI Operations.  
**AR:** Dealix — تشغيل الإيراد والذكاء الاصطناعي بحوكمة، أدلة، وموافقات.

صيغة الرسالة الرسمية:

- **EN:** Dealix helps companies turn AI experimentation and revenue operations into governed, measurable workflows - with source clarity, approval boundaries, evidence trails, and proof of value.
- **AR:** تساعد Dealix الشركات على تحويل تجارب الذكاء الاصطناعي وعمليات الإيراد إلى تشغيل محكوم وقابل للقياس، عبر وضوح المصادر، حدود الموافقة، مسارات الأدلة، وإثبات القيمة.

## 2) نجم الشمال

**Governed Value Decisions Created**

تعريف قياسي:
- قرار له `source` واضح.
- قرار له `approval` واضح.
- قرار له `evidence` موثق.
- قرار له أثر قيمة قابل للقياس (`value`).

## 3) العقيدة التشغيلية (Golden Chain)

كل workflow يمر بهذا التسلسل:

`Signal -> Source -> Approval -> Action -> Evidence -> Decision -> Value -> Asset`

أي نشاط لا يدخل السلسلة أعلاه يعتبر drift حتى يثبت العكس.

## 4) نموذج الشركة (Execution Model)

الترتيب المعتمد:

1. Diagnostic
2. Sprint
3. Retainer
4. Reusable Playbook
5. Internal Platform
6. Client Workspace
7. SaaS Module

قاعدة تشغيل:
- Service-led
- Software-assisted
- Evidence-first
- Retainer-backed
- Platform-later

المبدأ:

## 7) State Machine (Evidence Levels)

- `prepared_not_sent = L2`
- `sent = L4`
- `replied_interested = L4`
- `meeting_booked = L4`
- `used_in_meeting = L5`
- `scope_requested = L6`
- `pilot_intro_requested = L6`
- `invoice_sent = L7_candidate`
- `invoice_paid = L7_confirmed`

قواعد عدم الكسر:
- لا `sent` بدون `founder_confirmed=true`.
- لا L5 بدون `used_in_meeting`.
- لا L6 بدون `scope_requested|pilot_intro_requested`.
- لا L7_confirmed بدون `invoice_paid`.
- لا claim revenue قبل `invoice_paid`.

## 8) Decision Gates (Commercial Progression)

- **Gate 1:** First Market Proof (`sent` + تصنيف الرد/الصمت)
- **Gate 2:** Meeting Proof (`used_in_meeting = L5`)
- **Gate 3:** Pull Proof (`scope_requested = L6`)
- **Gate 4:** Revenue Proof (`invoice_paid = L7_confirmed`)
- **Gate 5:** Repeatability (بيع نفس العرض مرتين)
- **Gate 6:** Retainer (MRR مثبت)
- **Gate 7:** Platform Signal (تكرار workflow `>=3`)

## 9) No-Build Doctrine

لا نبني ميزة جديدة إلا إذا تحقق واحد أو أكثر:

- طلب عميل صريح.
- تكرار workflow ثلاث مرات.
- تقليل خطر حقيقي.
- تسريع تسليم مدفوع.
- فتح احتمال retainer واضح.

غير ذلك: **No build**.

## 10) Full Ops تعريفًا (Safety-first)

Full Ops لا يعني autonomous external execution.  
Full Ops يعني النظام:

- يجهز
- يقترح
- يحذر
- يسجل
- يتحقق
- يصنف
- يولد draft

والأفعال الخارجية دائمًا approval-first.

## 11) First 30 Execution Loop (Founder Cadence)

1. Run readiness: `bash scripts/run_ceo_one_session_readiness.sh`
2. Select first 5 warm contacts.
3. Send manually (no automation).
4. Log `sent` evidence.
5. If interested -> meeting.
6. If scope requested -> diagnostic scope.
7. If scope approved -> invoice draft -> invoice sent.
8. On `invoice_paid` -> proof pack + retainer path.

## 12) Governance Operating Frame (NIST AI RMF aligned)

- **GOVERN:** ownership + policy + approvals.
- **MAP:** workflow risks + data risks + use-case map.
- **MEASURE:** KPI + evidence quality + outcome quality.
- **MANAGE:** mitigations + logs + review cadence + no-build decisions.
