# سياسة ضبط النطاق وحارس النطاق (Scope Guard)
## منع تضخم النطاق وحماية العلاقة مع العميل

> **مبدأ أساسي:** كل طلب يقع خارج `scope` في `DHO-XXXX` يمر عبر Scope Guard — لا يُمتَص بصمت، أبدًا.
> **لماذا:** تضخم النطاق يضر العميل (توقعات غير واضحة) والفريق (استنزاف طاقة بلا مقابل).

---

## 1. ما هو تضخم النطاق (Scope Creep)

تضخم النطاق يحدث عندما تتراكم مهام وطلبات إضافية على التسليم دون:
- موافقة رسمية
- تعديل الجدول الزمني
- تعديل السعر أو الموارد

النتيجة: فريق مرهق، عميل محبط، تسليم متأخر، وعلاقة مهترئة.

---

## 2. من هو Scope Guard

الـScope Guard ليس شخصًا واحدًا بالضرورة — بل دور ومسؤولية:

| الدور | المسؤولية |
|---|---|
| **Delivery Owner** | يُشغّل Scope Guard في التسليم اليومي |
| **المؤسس** | القرار النهائي في طلبات تغيير النطاق |
| **العقد/DHO-XXXX** | المرجع الموضوعي للحكم على كل طلب |

**قاعدة:** أي طلب تغيير يذهب للمؤسس أولًا — قبل أي وعد للعميل.

---

## 3. أنواع الطلبات وكيفية التعامل معها

| نوع الطلب | مثال | الإجراء |
|---|---|---|
| **ضمن النطاق** | "هل يمكن تعديل عنوان التقرير؟" | نفّذ مباشرة |
| **توضيح** | "هل الخطة تشمل أسبوعين أو أربعة؟" | رجع لـ DHO + وضّح |
| **تغيير صغير** | "أضف عمودًا في الجدول" | قيّم الجهد + أبلغ المؤسس |
| **توسع نطاق** | "هل تقدر تدير إعلاناتنا كذلك؟" | Scope Guard → لا |
| **منتج جديد** | "نريد نظام CRM كامل" | Scope Guard → عرض منفصل |

---

## 4. عملية Scope Guard خطوة بخطوة

```
العميل يطلب شيئًا إضافيًا
        ↓
هل هو ضمن scope في DHO-XXXX؟
        ↓
   نعم ────────────────────────→ نفّذ (L3)
        ↓
        لا
        ↓
وثّق الطلب في scope_change_log
        ↓
أبلغ العميل: "سنراجع هذا الطلب ونعود إليك"
(لا وعود الآن — Approval First)
        ↓
رفع الطلب للمؤسس + تقييم: جهد؟ قيمة؟ تأثير على الـtimeline؟
        ↓
المؤسس يقرر:
  ┌─────────────────────┬──────────────────────┐
  رفض / خارج النطاق   قبول بتعديل عقد      قبول مجاني
        ↓                     ↓                   ↓
  أبلغ العميل بأدب    عرض Scope Change      نفّذ + وثّق
  + اقترح عرضًا       (سعر + timeline)
    منفصلًا
```

---

## 5. كيف تخبر العميل بـ "خارج النطاق"

الرسالة يجب أن تكون:
1. محترمة وإيجابية
2. واضحة في الحدود
3. تفتح بابًا للحل المدفوع

**نموذج رسالة (يُعدّ للمراجعة قبل الإرسال):**

> "شكرًا لمشاركتك هذا الطلب. بمراجعة نطاق عملنا المتفق عليه في [DHO-XXXX]، يقع هذا الطلب خارج النطاق الحالي. يسعدنا تقييم إدراجه كخدمة إضافية — سأعود إليك خلال يوم عمل بتقييم واضح."

## 6. نموذج Scope Change Request

عند تسجيل طلب تغيير نطاق، يُملأ النموذج التالي:

| الحقل | المحتوى |
|---|---|
| **التاريخ** | |
| **الشركة** | |
| **DHO المرجعي** | |
| **الطلب** | (وصف دقيق بكلمات العميل) |
| **هل هو ضمن النطاق؟** | نعم / لا |
| **الجهد المقدّر** | (ساعات / أيام) |
| **التأثير على الـtimeline** | |
| **التوصية** | رفض / عرض إضافي / قبول مجاني |
| **قرار المؤسس** | |
| **approved_by + approved_at** | |

---

## 7. طلبات تغيير النطاق في مرحلة القبول (ACC-XXXX)

الـ`scope_changes_requested` في `ACC-XXXX` هو المكان الرسمي لتسجيل هذه الطلبات. الحقيقة المهمة:

**لا تقبل التسليم مشروطًا بتنفيذ طلبات جديدة.**

| الوضع | الإجراء |
|---|---|
| العميل يقبل مع طلبات جديدة | سجّل الطلبات في `scope_changes_requested`، اقبل التسليم الحالي |
| العميل يرفض بسبب طلبات جديدة | راجع DHO-XXXX — هل كانت ضمن النطاق أصلًا؟ |
| طلب ضمن النطاق فعلًا | انظر DHO-XXXX وأكمل ما هو مطلوب |

---

## 8. المؤشرات التحذيرية لتضخم النطاق

راقب هذه العلامات مبكرًا:

| المؤشر | الإجراء الفوري |
|---|---|
| "بس شيء صغير إضافي..." يتكرر | توثيق + Scope Guard |
| طلب اجتماعات إضافية بمواضيع جديدة | "سنضيف هذا لأجندة عرض منفصل" |
| مقارنة بمزود آخر "يعطي أكثر" | مراجعة DHO + توضيح القيمة |
| عدم رضا عن نتائج ضمن النطاق | تميّز بين "جودة" و"نطاق" |

---

## 9. ما لا يجوز للـScope Guard

| ممنوع | لماذا |
|---|---|
| قبول طلب خارج النطاق بدون موافقة المؤسس | approval_first إلزامي |
| رفض الطلب دون عرض بديل | يُضر بالعلاقة |
| تأخير الرد أكثر من يوم عمل | يبني توترًا بلا داعٍ |
| وعد بـ"نشوف" بدون موعد محدد | غموض = توقعات خاطئة |

---

## 10. ربط Scope Guard بدورة التسليم الكاملة

| متى | الإجراء |
|---|---|
| عند إنشاء DHO-XXXX | تحديد `out_of_scope` بوضوح |
| Day 1-3 (Intake) | مراجعة `scope` مع العميل وتوقيع الاستلام |
| Day 8-10 (Review) | تفعيل Scope Guard عند ظهور طلبات جديدة |
| Day 11-14 (Acceptance) | `scope_changes_requested` في ACC-XXXX |
| مستمر (أسبوعي) | مراجعة طلبات التغيير في كل اجتماع |
