# تقرير القيمة الأسبوعي (Weekly Value Report)
## القيمة المُسلَّمة بأدلة — لا ضمان عائد

> **السكيمة المرجعية:** [`schemas/weekly_value_report.schema.json`](../../schemas/weekly_value_report.schema.json)
> **قاعدة لا تُكسر:** لا رقم في WVR بلا `evidence_level`. لا وعد ROI مضمون.
> **قناة التسليم:** البوابة الآمنة فقط (`delivered_via: "secure_portal"`).

---

## 1. لماذا تقرير القيمة الأسبوعي

تقرير القيمة الأسبوعي هو الوثيقة التي تُثبت للعميل أن استثماره يُنتج نتائج قابلة للقياس. بدونه:
- لا يرى العميل ما حُقِّق
- يصعب تبرير التجديد عند انتهاء العقد
- لا يمكن لفريق Dealix إثبات أثره

**متى يبدأ:** بعد انتهاء أول أسبوع تسليم فعلي (Day 7 تقريبًا).
**التكرار:** أسبوعي، كل يوم اثنين أو الثلاثاء.
**قاعدة التجديد:** أي مسودة تجديد تتطلب `evidence_level ∈ {client_data, measured, verified}`.

---

## 2. الحقول الإلزامية في `weekly_value_report.schema.json`

| الحقل | النوع | الوصف |
|---|---|---|
| `id` | string | معرّف `WVR-XXXX` |
| `created_at` | datetime | وقت إنشاء التقرير |
| `company` | string | اسم الشركة |
| `week_of` | date | تاريخ بداية الأسبوع |
| `value_delivered` | string | وصف نصي للقيمة المُسلَّمة |
| `metrics` | array | قائمة المقاييس مع `evidence_level` |
| `evidence_level` | enum | المستوى العام للتقرير |
| `next_week_plan` | string | خطة الأسبوع القادم |

---

## 3. هيكل المقاييس (`metrics`)

كل مقياس في المصفوفة يحتوي على:

| الحقل | مطلوب | الوصف |
|---|---|---|
| `name` | ✅ | اسم المقياس |
| `value` | ✅ | القيمة الحالية |
| `evidence_level` | ✅ | مستوى الدليل |
| `baseline` | لا | القيمة الأساسية قبل التدخل |

### مثال حي — WVR-1001 (Digital Rise Agency)

```
week_of: 2026-06-01
value_delivered: "تفعيل تذكير المتابعة استرجع 12 فرصة كانت متروكة."

المقاييس:
┌─────────────────────┬───────────┬─────────┬───────────────┐
│ المقياس             │ Baseline  │ الآن    │ evidence_level│
├─────────────────────┼───────────┼─────────┼───────────────┤
│ نسبة المتابعة       │ 43%       │ 61%     │ measured      │
│ متوسط زمن الرد      │ 4.2 ساعة  │ 1.1 ساعة│ measured      │
└─────────────────────┴───────────┴─────────┴───────────────┘
```

## 4. سلّم `evidence_level` في WVR

| المستوى | المعنى | صالح في WVR؟ |
|---|---|---|
| `none` | لا دليل | لا — يوقف التقرير |
| `assumption` | فرضية داخلية | لا — لا يُرسَل للعميل |
| `benchmark` | معيار صناعي | نعم، مع تحفظ واضح |
| `client_reported` | العميل أبلغ عنه | نعم، مع "كما أبلغ العميل" |
| `client_data` | من بيانات العميل الفعلية | نعم |
| `measured` | قِسناه أثناء التسليم | نعم — مفضّل |
| `verified` | مُتحقَّق مستقلًا | نعم — أعلى جودة |

**قاعدة:** المستوى العام للتقرير (`evidence_level` الجذر) = أضعف مستوى بين المقاييس المُدرجة.

---

## 5. قناة التسليم — البوابة الآمنة فقط

| قيمة `delivered_via` | المعنى | متى تُستخدم |
|---|---|---|
| `secure_portal` | التسليم عبر البوابة الآمنة فقط | الافتراضي دائمًا |
| `portal_and_whatsapp_notice` | البوابة + إشعار واتساب (بعد موافقة) | إذا طلب العميل إشعارًا |

**ممنوع:**
- إرسال تقرير القيمة مباشرة عبر واتساب
- إرفاق الأرقام الحساسة في رسائل البريد الإلكتروني المباشرة
- مشاركة بيانات العميل في أدوات ذكاء عام

---

## 6. تنبيه: لا ضمان ROI

يجب أن يحتوي كل WVR على تنويه صريح:

> "الأرقام الواردة هنا تعكس ما قِسناه أثناء التسليم (`evidence_level: measured`). هي مؤشرات أداء فعلية، لا وعودًا مضمونة باستمرار هذا الأداء أو تحقيق عائد استثمار محدد."

هذا ليس تحفظًا قانونيًا فقط — بل لأن الأداء يعتمد على التزام الفريق وعوامل خارجية لا يتحكم فيها Dealix.

---

## 7. إجراء إعداد وتسليم WVR (خطوة بخطوة)

```
1. [L3] Delivery Owner يجمع بيانات الأسبوع
2. [L3] يملأ WVR-XXXX بكل الحقول الإلزامية
3. [L3] يتحقق: كل مقياس له evidence_level ≥ client_data
4. [L3] يكتب next_week_plan بوضوح
5. [L4] يُحمَّل التقرير على البوابة الآمنة
6. [L4] يُرسَل إشعار للعميل (بموافقة) عبر البوابة
7. تسجيل الإجراء في ai_action_ledger
```

---

## 8. فحص جودة WVR قبل التسليم

- [ ] `id` بنمط `WVR-XXXX`
- [ ] `week_of` صحيح
- [ ] `value_delivered` وصف إيجابي وملموس
- [ ] كل مقياس له `evidence_level` ≥ `client_reported`
- [ ] `evidence_level` الجذر ≠ `none` أو `assumption`
- [ ] `next_week_plan` واضح وقابل للتنفيذ
- [ ] `delivered_via: "secure_portal"`
- [ ] لا بيانات PII في التقرير
- [ ] لا وعود ROI مضمونة في النص

---

## 9. ماذا يحدث إذا كانت النتائج ضعيفة هذا الأسبوع

لا تخفي الأرقام السلبية. WVR صادق > WVR مُلمَّع.

| الوضع | ما تفعله |
|---|---|
| مقياس تراجع | سجّله مع `evidence_level` + سبب التراجع في `risks` |
| لا توجد بيانات كافية | أذكر ذلك صراحةً: "لم تتوفر بيانات كافية هذا الأسبوع" |
| العميل لم يُفعّل التوصيات | وثّق في `risks` + اذكره في `next_week_plan` |

---

## 10. ربط WVR بدورة التجديد

- الـWVR هو الوثيقة الأساسية لمسودة التجديد/الترقية
- لا تُكتب مسودة تجديد إلا بـ `evidence_level ∈ {client_data, measured, verified}`
- مجموع WVRs = Proof Pack شهري
- انظر [`company_os/delivery/proof_pack_template.md`](../../company_os/delivery/proof_pack_template.md)
