# معايير قبول الانتقال من GTM إلى التسليم
## ما الذي يجب أن يكون جاهزًا قبل بدء التسليم

> **مبدأ البوابة:** لا يبدأ التسليم قبل اجتياز جميع معايير البوابة (Gate Criteria). بوابة لا تُتجاوز بذرائع.
> **المرجع:** [`schemas/delivery_handoff.schema.json`](../../schemas/delivery_handoff.schema.json)

---

## 1. لماذا بوابة GTM → Delivery

الانتقال من مرحلة GTM (المبيعات) إلى مرحلة التسليم هو أحد أكثر الانتقالات حساسية في دورة الإيراد. الفشل في هذه المرحلة يُكلّف:
- توقعات عميل غير متطابقة مع ما وُعد به
- استنزاف فريق التسليم على متطلبات لم تُحدَّد
- خلافات نطاق منذ البداية
- فقدان ثقة العميل في الأسابيع الأولى

**البوابة تضمن:** كل ما وعد به المبيعات موثّق، كل ما لم يُوعد به واضح، وكل ما يحتاجه التسليم متاح.

---

## 2. Gate 1 — اكتمال وثيقة الـHandoff (DHO-XXXX)

### متطلبات Gate 1

| المعيار | الفحص | الحالة |
|---|---|---|
| `DHO-XXXX` موجود ومُنشأ | `id` بنمط `DHO-XXXX` | ✅/❌ |
| `product_id` ∈ كتالوج المنتجات | مطابقة الكتالوج | ✅/❌ |
| `scope` محدد بـ ≥ 1 عنصر واضح | قائمة منطوقة | ✅/❌ |
| `out_of_scope` محدد | قائمة منطوقة | ✅/❌ |
| `success_metric` قابل للقياس | ليس "نجاح عام" | ✅/❌ |
| `required_access` من القائمة المسموحة | enum محدد | ✅/❌ |
| `first_workflow` محدد | وصف واضح | ✅/❌ |
| `delivery_owner` مسمّى | اسم حقيقي | ✅/❌ |
| `timeline` محدد | أيام أو تاريخ | ✅/❌ |
| `risks` موثّقة | ≥ 1 خطر | ✅/❌ |
| `weekly_value_report_template` مرتبط | مسار صالح | ✅/❌ |

**نتيجة Gate 1:** اجتياز 11/11 ← Gate 2 مفتوحة. أي حقل ناقص = Gate 2 مغلقة.

---

## 3. Gate 2 — موافقة المؤسس

### متطلبات Gate 2

| المعيار | التحقق |
|---|---|
| المؤسس راجع `DHO-XXXX` | تسجيل مراجعة |
| `approved_by` مملوء بالمؤسس | حقل موجود |
| `approved_at` مسجَّل | timestamp صحيح |
| تسجيل الإجراء في `ai_action_ledger` | سجل موجود |

**لا يوجد تفويض لتخطي هذه الخطوة.**

---

## 4. Gate 3 — جاهزية الوصول والبيانات

### متطلبات Gate 3

| المعيار | متى؟ | الحالة |
|---|---|---|
| رابط البوابة الآمنة مُنشأ | قبل Day 1 | ✅/❌ |
| الصلاحيات المطلوبة محددة من `required_access` | Day 0 | ✅/❌ |
| العميل أُبلغ بطريقة استلام الوصول | Day 0 | ✅/❌ |
| اتفاقية البيانات (PDPL) موقّعة من العميل | Day 0-1 | ✅/❌ |

**تحذير:** لا تُمنح صلاحيات الوصول خارج البوابة الآمنة.

---

## 5. Gate 4 — وضوح التوقعات مع العميل

### ما يجب أن يعرفه العميل قبل Day 1

| الموضوع | الوضوح المطلوب |
|---|---|
| ما هو بالضبط ضمن النطاق | مراجعة `scope` معًا |
| ما هو خارج النطاق | مراجعة `out_of_scope` معًا |
| ما هو `success_metric` | وكيف سيُقاس |
| أن لا ضمان ROI | تحفظ صريح موثّق |
| الجدول الزمني | `timeline` من DHO |
| كيف يُسلَّم WVR | عبر البوابة الآمنة |
| طريقة التعامل مع طلبات التغيير | عبر Scope Guard |

---

## 6. Gate 5 — جاهزية فريق التسليم

| المعيار | الفحص |
|---|---|
| `delivery_owner` متفرغ للتسليم | تحقق من التقويم |
| `ONB-XXXX` مُنشأ ومرتبط بـ `DHO-XXXX` | سجل الإنشاء |
| بيئة العمل الداخلية جاهزة | مجلد العميل، قوالب |
| خطة أول أسبوع (Day 0-7) موثّقة | قائمة مهام واضحة |

---

## 7. ملخص بوابة GTM → Delivery

```
صفقة مربوحة (PROP-XXXX موافَق عليه)
        ↓
Gate 1: اكتمال DHO-XXXX (11 حقل إلزامي) ──── FAIL → أكمل الحقول أولًا
        ↓
Gate 2: موافقة المؤسس (approved_by + approved_at) ── FAIL → لا تكمل
        ↓
Gate 3: جاهزية الوصول (بوابة آمنة + صلاحيات) ─── FAIL → أنشئ البوابة أولًا
        ↓
Gate 4: وضوح التوقعات مع العميل ──────────────── FAIL → اجتماع توضيحي
        ↓
Gate 5: جاهزية فريق التسليم ─────────────────── FAIL → جدولة + تجهيز
        ↓
✅ بدء التسليم الرسمي (ONB-XXXX: in_progress)
```

---

## 8. ما يحدث إذا بدأ التسليم بلا بوابات

| الفشل | العاقبة |
|---|---|
| DHO ناقص | فريق التسليم يخمّن النطاق → تضخم حتمي |
| بلا موافقة مؤسس | قرارات بلا محاسبة → خلافات لاحقة |
| بلا وضوح توقعات | عميل محبط من الأسبوع الأول |
| بلا access جاهز | Day 1 ضائع → تأخر cascade |

---

## 9. حالات الاستثناء

| الحالة | الإجراء |
|---|---|
| المؤسس غير متاح يومًا | تأجيل Gate 2 يومًا (لا تخطيها) |
| العميل يضغط للبدء فورًا | أكمل Gates 1-3 كحد أدنى |
| بيانات access جزئية | ابدأ بما توفر + وثّق الناقص في risks |
| تغيير المنتج بعد الموافقة | DHO جديد + دورة gates من البداية |

---

## 10. مثال تطبيقي — Digital Rise Agency

| Gate | الحالة | ملاحظة |
|---|---|---|
| Gate 1: DHO-1001 | ✅ مكتمل | 11/11 حقل مملوءة |
| Gate 2: موافقة مؤسس | ✅ Day 0 | `approved_by: founder` |
| Gate 3: صلاحيات | ✅ Day 1 | `read_only_crm`, `file_upload` عبر البوابة |
| Gate 4: توقعات | ✅ Day 1 | scope + out_of_scope موضّحان |
| Gate 5: فريق | ✅ Day 0 | `delivery_owner: founder`، ONB-1001 جاهز |
| **النتيجة** | **✅ بدء رسمي** | ONB-1001: in_progress |
