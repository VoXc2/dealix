# Proof Pack — المعيار

**الغرض:** **Proof Pack** هو قلب البيع: العميل لا يشتري «AI» بل **وضوح، دليل، قرار، تقليل هدر، خطوة تالية**.

## أقسام Proof Pack القياسية

1. **Context** — السياق والهدف  
2. **Inputs reviewed** — ما تمت مراجعته (بدون ادعاءات بلا مصدر)  
3. **Lead / workflow status** — لوحة حالة / كروت  
4. **Source quality** — وضوح المصدر وجودته  
5. **Owner gaps** — أين يختفي المالك؟  
6. **Approval risks** — ما الذي قد يخرق سياسة أو يحتاج موافقة؟  
7. **Follow-up gaps** — أين توقفت المتابعة؟  
8. **Draft messages** — مسودات (ليست إرسالاً حياً)  
9. **Recommended next actions** — أفعال مقترحة مع مالك مقترح  
10. **Truth labels** — تصنيف كل ادعاء (أدناه)  
11. **Upgrade path** — Sprint / Retainer / نطاق تالي منطقي

## Truth Labels (إلزامي لكل claim مهم)

| Label | المعنى |
|-------|--------|
| **Estimate** | تقدير — لا يُباع كحقيقة تشغيلية |
| **Observed** | لوحظ من البيانات/السجلات المقدمة |
| **Client-confirmed** | أكده العميل صراحة |
| **Payment-confirmed** | مرتبط بدفع مثبت |
| **Repeated workflow** | تكرر بنفس الشكل عبر حالات |
| **Retainer-ready** | جاهزية تشغيل للاستمرارية (قرار وليس وعداً) |

### أمثلة

- فجوة متابعة: **Observed**  
- قيمة محتملة: **Estimate**  
- ألم العميل: **Client-confirmed**  
- إيراد: **Payment-confirmed** فقط
