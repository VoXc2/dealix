# Productization Path — مسار التحويل إلى منتج

## English

### Do not jump to SaaS

Dealix is Governed Revenue & AI Operations, not a chatbot or an automation
reseller. A product is the last stage of a path, not the first decision. The
path moves from founder-assisted delivery to product only on real, repeated
demand. Skipping stages builds features nobody paid for.

### The six stages

1. **Internal tools** — checklists, scorecards, and Proof templates used by
   the founder to deliver. No client sees them yet.
2. **Client-visible reports** — the Proof Pack and diagnostic outputs shaped
   for the client. The client sees results, not the machine.
3. **Client workspace** — a place where a client follows their own evidence,
   owners, and next actions.
4. **Partner portal** — partners co-sell and track shared work through a
   structured surface.
5. **Self-serve modules** — single capabilities a user can run without the
   founder present.
6. **SaaS** — a packaged product. Only reached after stages one to five have
   each earned repeated paid use.

### The build rule — graduation by repetition

Nothing is built ahead of demand. A workflow graduates only as it repeats:

- New workflow → write a checklist.
- Repeats twice → turn the checklist into a template.
- Repeats three times → automate the template.
- Repeats with two clients → build an internal module.
- Repeats with three clients plus retainers → it becomes a product feature.

### Proof gates

Four gates govern the path and override any wish to move faster:

- No revenue is counted before payment.
- No Stage 5 (self-serve modules) before a real meeting.
- No Stage 6 (SaaS) before a written scope.
- No build before the workflow has actually repeated.

These gates keep the path inside the operating doctrine chain: Signal →
Source → Owner → Approval → Action → Evidence → Decision → Value → Asset.

## العربية

### لا تقفز إلى SaaS

Dealix هي حوكمة الإيرادات وعمليات الذكاء الاصطناعي، ليست روبوت محادثة ولا
بائع أتمتة. المنتج هو المرحلة الأخيرة من مسار، لا القرار الأول. ينتقل المسار من
تسليم بمساعدة المؤسِّس إلى منتج فقط عند طلب حقيقي ومتكرر. تخطّي المراحل يبني
ميزات لم يدفع لها أحد.

### المراحل الست

١. **أدوات داخلية** — قوائم تحقّق وبطاقات تقييم وقوالب إثبات يستخدمها المؤسِّس
   للتسليم. لا يراها أي عميل بعد.
٢. **تقارير ظاهرة للعميل** — Proof Pack ومخرجات التشخيص مصاغة للعميل. يرى
   العميل النتائج لا الآلة.
٣. **مساحة عمل للعميل** — مكان يتابع فيه العميل أدلّته ومُلّاكه وخطواته التالية.
٤. **بوابة الشركاء** — يبيع الشركاء بالتعاون ويتابعون العمل المشترك عبر سطح
   منظَّم.
٥. **وحدات خدمة ذاتية** — قدرات مفردة يشغّلها المستخدم دون حضور المؤسِّس.
٦. **SaaS** — منتج مُحزَّم. لا يُبلَغ إلا بعد أن تكسب المراحل من الأولى إلى
   الخامسة استخدامًا مدفوعًا متكررًا.

### قاعدة البناء — التخرّج بالتكرار

لا يُبنى شيء قبل الطلب. يتخرّج سير العمل فقط مع تكراره:

- سير عمل جديد ← اكتب قائمة تحقّق.
- يتكرر مرتين ← حوّل القائمة إلى قالب.
- يتكرر ثلاث مرات ← أتمت القالب.
- يتكرر مع عميلَين ← ابنِ وحدة داخلية.
- يتكرر مع ثلاثة عملاء مع Retainers ← يصبح ميزة منتج.

### بوابات الإثبات

أربع بوابات تحكم المسار وتتقدّم على أي رغبة في التسريع:

- لا تُحتسب إيرادات قبل الدفع.
- لا مرحلة خامسة (وحدات خدمة ذاتية) قبل اجتماع حقيقي.
- لا مرحلة سادسة (SaaS) قبل نطاق مكتوب.
- لا بناء قبل أن يكون سير العمل قد تكرر فعلًا.

تُبقي هذه البوابات المسار داخل سلسلة الدوكترين: إشارة ← مصدر ← مالك ← موافقة
← إجراء ← دليل ← قرار ← قيمة ← أصل.
