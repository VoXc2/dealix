# Productization Path — مسار التصنيع

<!-- Layer: Empire | Owner: Founder | Date: 2026-05-17 -->
<!-- Arabic primary — العربية أولاً -->

> **قاعدة حاكمة:** لا قفز إلى SaaS. لا تُبنى ميزة منتج قبل ظهور طلب
> متكرر مُثبت (المعتقد الأساسي رقم 9 في الدستور). البناء يتبع الطلب،
> لا الطموح.

---

## المبدأ — التدرّج لا القفزة

أكثر ما يقتل المؤسسات الصغيرة هو بناء منتج قبل وجود طلب عليه. Dealix
لا تبني SaaS لأنها تريد أن تكون شركة SaaS؛ تبنيه فقط حين يُجبرها
الطلب المتكرر. المسار يصعد درجة درجة، وكل درجة تُموَّل من الدرجة
السابقة وتُثبت قبل الانتقال.

---

## ست مراحل التصنيع — Six stages

```
Stage 1  Internal tools          ← أدوات تشغيل داخلية للمؤسس فقط
   │
Stage 2  Client-visible reports  ← تقارير يراها العميل لا يشغّلها
   │
Stage 3  Client workspace        ← مساحة عمل يدخلها العميل
   │
Stage 4  Partner portal          ← بوابة للشركاء وإعادة البيع
   │
Stage 5  Self-serve modules      ← وحدات يستخدمها العميل ذاتياً
   │
Stage 6  SaaS                    ← منتج مستقل قائم بذاته
```

لا تُتخطى مرحلة. كل مرحلة تتطلّب إثبات استخدام حقيقي قبل التالية.
اليوم Dealix في Stage 1–2؛ ما بعدها مشروط ومستقبلي.

---

## سلّم قاعدة البناء — Build rule ladder

متى يُبنى شيء؟ ليس عند الفكرة، بل عند تكرّر الطلب. هذا السلّم يُفعّل
المعتقد رقم 9: لا طلب متكرر، لا ميزة منتج.

```
workflow جديد
   │  أنجزه يدوياً ووثّقه
   ▼
checklist
   │  تكرّر مرتين؟
   ▼
template
   │  تكرّر 3 مرّات؟
   ▼
automation
   │  تكرّر مع عميلَين؟
   ▼
internal module
   │  تكرّر مع 3 عملاء + retainers؟
   ▼
product feature
```

| المحفّز | المخرج المسموح |
|---------|-----------------|
| workflow جديد | أنجزه يدوياً، لا تبنِ شيئاً بعد |
| تكرّر مرّة ثانية | حوّله إلى checklist |
| تكرّر 3 مرّات | حوّله إلى template |
| تكرّر مع عميلَين | اسمح بـ automation |
| تكرّر مع عميلَين (طلب مستقر) | internal module |
| تكرّر مع 3 عملاء + retainers | product feature |

---

## لماذا هذا السلّم

- يمنع بناء أصول لا تُستخدم — كل بناء مدفوع بطلب مُثبت.
- يحوّل المعرفة إلى منتج تدريجياً (انظر [`CAPITAL_MODEL.md`](./CAPITAL_MODEL.md)).
- يبقي الهامش مرتفعاً: الأتمتة تأتي بعد إثبات القيمة لا قبلها.
- يحمي من ادعاء «منتج جاهز» قبل أن يكون كذلك.

محفّزات البناء التفصيلية موثّقة في
[`../sales-kit/CONDITIONAL_BUILD_TRIGGERS.md`](../sales-kit/CONDITIONAL_BUILD_TRIGGERS.md)
— هذا المستند هو الدوكترين، وذاك هو التشغيل.

## بوابات الانتقال بين المراحل

| الانتقال | الشرط |
|----------|-------|
| Stage 1 → 2 | تقرير داخلي أثبت قيمته في تسليم حقيقي |
| Stage 2 → 3 | ≥ 3 عملاء طلبوا الاطلاع المباشر على التقرير |
| Stage 3 → 4 | شريك واحد على الأقل بعقد موقّع |
| Stage 4 → 5 | طلب متكرر من عملاء على وظيفة ذاتية الخدمة |
| Stage 5 → 6 | استخدام ذاتي مستقر + اقتصاد وحدة مُثبت |

كل انتقال قرار مؤسس موثّق، لا انزلاق تلقائي.

---

## ما لا يفعله هذا المسار

- لا يَعِد بأن Dealix ستصبح SaaS — هذا مشروط بالطلب.
- لا يبني ميزة لأن منافساً بناها.
- لا يصف وحدة قيد البناء كأنها متاحة اليوم.
