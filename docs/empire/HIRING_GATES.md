# Hiring Gates — بوابات التوظيف

## English

### Do not hire early — hire after signals

Dealix is Governed Revenue & AI Operations, not a chatbot, not an AI agency,
not an automation reseller. Hiring follows the same operating doctrine chain as
everything else: Signal → Source → Owner → Approval → Action → Evidence →
Decision → Value → Asset. A hire is an Action, and an Action needs a Signal and
Evidence before it. The signal is never "the founder is busy." The signal is a
workflow that has repeated. Headcount added before repetition turns a fixed
cost into noise the company has not earned.

### The current state

Today there is no team to gate. The founder holds three roles at once: Sales,
Delivery, and Approval. The System carries the load that headcount would
otherwise carry: it produces drafts, scorecards, and Proof Pack templates. This
is intentional. It keeps the cost base low while the offer ladder is still
being proven and keeps every external action under one named approver, in line
with the non-negotiable that every external action needs human approval.

### The gates

Each role opens only when its signal is observed and Payment-confirmed:

| Signal observed | Role that opens | Why |
|---|---|---|
| 3 paid pilots delivered | Part-time Ops Assistant | Repeated delivery admin justifies relief |
| 5–8 paid pilots delivered | Delivery Operator | Delivery is now a repeated workflow, not a one-off |
| First Retainer signed | Partner Manager / Growth Assistant | Recurring revenue can fund channel work |
| 3 Retainers running | Customer Success + Product Engineer | Expansion and product demand are now real |

Every gate is keyed to paid, confirmed work — pilots, not conversations, and
Retainers, not proposals. A gate that has not been crossed is not a hiring
plan; it is a wish.

### The rule

Do not hire because you are busy. Hire because a workflow repeated. Busyness is
a Signal that something must change, but the change may be a template, an
automation, or a dropped low-value rung — see
[`PRODUCTIZATION_PATH.md`](PRODUCTIZATION_PATH.md) for graduation by repetition.
A hire is the most expensive answer; it is correct only when the repeated
workflow is documented, paid for, and ready to hand to a named owner. Estimated
demand is not confirmed demand.

## العربية

### لا توظّف مبكرًا — وظّف بعد الإشارات

Dealix هي حوكمة الإيرادات وعمليات الذكاء الاصطناعي، ليست روبوت محادثة، ولا
وكالة ذكاء اصطناعي، ولا بائع أتمتة. يتبع التوظيف سلسلة الدوكترين التشغيلي نفسها
التي يتبعها كل شيء: إشارة ← مصدر ← مالك ← موافقة ← إجراء ← دليل ← قرار ← قيمة ←
أصل. التوظيف إجراء، والإجراء يحتاج إشارة ودليلًا قبله. الإشارة ليست أبدًا «أن
المؤسِّس مشغول». الإشارة هي سير عمل تكرّر. إضافة رأس قبل التكرار تحوّل تكلفة
ثابتة إلى ضجيج لم تكسبه الشركة.

### الوضع الحالي

اليوم لا فريق ليُبوَّب. يحمل المؤسِّس ثلاثة أدوار معًا: البيع، والتسليم،
والموافقة. ويحمل النظام العبء الذي كانت الرؤوس ستحمله: ينتج المسودات، وبطاقات
التقييم، وقوالب Proof Pack. هذا مقصود. يُبقي قاعدة التكلفة منخفضة بينما لا يزال
سلّم العروض قيد الإثبات، ويُبقي كل إجراء خارجي تحت معتمِد واحد مُسمّى، اتساقًا
مع المبدأ غير القابل للتفاوض بأن كل إجراء خارجي يحتاج موافقة بشرية.

### البوابات

يُفتح كل دور فقط حين تُرصد إشارته وتُؤكَّد بالدفع:

| الإشارة المرصودة | الدور الذي يُفتح | لماذا |
|---|---|---|
| تسليم ٣ تجارب مدفوعة | مساعد عمليات بدوام جزئي | إدارة التسليم المتكررة تبرّر التخفيف |
| تسليم ٥–٨ تجارب مدفوعة | مشغّل تسليم | التسليم الآن سير عمل متكرر لا مرة واحدة |
| توقيع أول Retainer | مدير شركاء / مساعد نمو | الإيراد المتكرر يستطيع تمويل عمل القناة |
| تشغيل ٣ Retainers | نجاح عملاء + مهندس منتج | طلب التوسّع والمنتج صار حقيقيًا |

كل بوابة مرتبطة بعمل مدفوع ومؤكَّد — تجارب لا محادثات، وRetainers لا عروض.
البوابة التي لم تُعبَر ليست خطة توظيف؛ بل أمنية.

### القاعدة

لا توظّف لأنك مشغول. وظّف لأن سير عمل تكرّر. الانشغال إشارة بأن شيئًا يجب أن
يتغيّر، لكن التغيير قد يكون قالبًا، أو أتمتة، أو إسقاط درجة منخفضة القيمة —
انظر [`PRODUCTIZATION_PATH.md`](PRODUCTIZATION_PATH.md) للتخرّج بالتكرار.
التوظيف هو الجواب الأغلى؛ ويكون صحيحًا فقط حين يكون سير العمل المتكرر موثَّقًا
ومدفوعًا وجاهزًا لتسليمه إلى مالك مُسمّى. الطلب المُقدَّر ليس طلبًا مؤكَّدًا.
