# سياسة الرفع الآمن للعملاء

> **نظام Dealix — Saudi B2B Revenue Operating System**
> الإصدار: 1.0 | التاريخ: 2026-06-03 | المالك: Agent #2
> السكيمة المرجعية: `schemas/client_upload.schema.json`
> المسار: `/client/upload`

---

## 1. المبدأ الأساسي

كل ملف يرفعه العميل يجب أن يمر عبر البوابة الآمنة حصرًا. لا يُقبل أي ملف عبر:
- واتساب
- بريد إلكتروني شخصي أو عام
- روابط Google Drive أو Dropbox مشاركة عامة
- أي قناة غير البوابة

**السبب:** الملفات كثيرًا ما تحتوي PII لعملاء العميل (أسماء، أرقام جوال، بيانات مالية). هذه المعلومات تتطلب تعاملًا آمنًا وفق PDPL/SDAIA.

---

## 2. حقول السجل (client_upload.schema.json)

| الحقل | النوع | الوصف |
|---|---|---|
| `id` | `UPL-XXXX` | معرّف فريد للملف المرفوع |
| `company` | string | اسم الشركة العميلة |
| `session_id` | string / null | ربط بجلسة البوابة `CPS-XXXX` |
| `filename` | string | اسم الملف الأصلي |
| `content_type` | string | نوع الملف (مثل `text/csv`, `application/pdf`) |
| `size_bytes` | integer | الحجم بالبايت |
| `storage_ref` | `portal://` أو `s3://` | **مرجع تخزين خاص** — لا رابط عام أبدًا |
| `scanned` | boolean | هل مرّ الملف بالفحص الأمني؟ |
| `pii_flag` | boolean | هل يحتمل الملف PII لأشخاص آخرين؟ |
| `retention_until` | date | تاريخ انتهاء الاحتفاظ (90 يومًا من الرفع) |
| `status` | enum | `received → scanning → ready / quarantined / deleted` |

---

## 3. دورة حياة الملف المرفوع

```
العميل يرفع الملف عبر /client/upload
        ↓
[status: received] — تسجيل فوري في النظام
        ↓
[status: scanning] — فحص فيروسات + كشف PII
        ↓
    [pii_flag = true?]
    نعم → تحذير وإخفاء هوية قبل أي تحليل
    لا  → متابعة عادية
        ↓
[status: ready] — متاح للتحليل المصرّح به
    أو
[status: quarantined] — مشبوه أو ضار → لا يُستخدم
        ↓
بعد 90 يومًا → [status: deleted] (حذف تلقائي)
```

---

## 4. قاعدة `storage_ref` — لا روابط عامة

- `storage_ref` يبدأ بـ `portal://` أو `s3://` — هو **مرجع** منطقي داخلي.
- لا يُخزَّن في أي ملف بيانات أو سجل أو تقرير أو مخرَج رابط `presigned` أو URL قابل للوصول العام.
- الوصول للملف يتم فقط من خلال البوابة بجلسة مُصادَق عليها.

**مثال واقعي:** UPL-1001 (Digital Rise Agency) — ملف `leads_export_q2.csv`، مرجعه `portal://uploads/digitalrise/leads_export_q2.csv`، تم فحصه، `pii_flag=true` (يحتوي بيانات استفسارات عملاء)، يُحذف بتاريخ 2026-09-01.

---

## 5. إدارة PII (البيانات الشخصية)

عندما يكون `pii_flag=true`:

| الخطوة | الإجراء |
|---|---|
| قبل التحليل | إخفاء هوية الأفراد (masking) — أسماء/جوال/بريد |
| أثناء التحليل | العمل على بيانات مُجمَّعة أو مُخفاة الهوية |
| في التقارير | لا أسماء/أرقام حقيقية لعملاء العميل |
| مع أدوات الذكاء | لا تُرسَل PII لأدوات ذكاء عام خارجية |

---

## 6. الاحتفاظ والحذف (PDPL § 5)

| القاعدة | التفصيل |
|---|---|
| مدة الاحتفاظ الافتراضية | 90 يومًا من `created_at` |
| `retention_until` | يُحسَب تلقائيًا عند الرفع |
| الحذف التلقائي | بعد انتهاء `retention_until` → `status: deleted` |
| الحذف بطلب العميل | فوري في أي وقت (حق صاحب البيانات) |
| ما يُحتفَظ به بعد الحذف | رؤى مجهولة الهوية فقط (بموافقة العميل) |

---

## 7. أنواع الملفات المقبولة وغير المقبولة

**مقبول:**
- تصدير CRM (CSV، XLSX)
- تقارير أداء الإعلانات
- سجلات استفسارات (مجهّلة إن أمكن)
- بيانات pipeline المبيعات

**غير مقبول:**
- ملفات تنفيذية (.exe, .sh, .bat)
- ملفات تحتوي مفاتيح API أو كلمات مرور
- بيانات بطاقات بنكية أو أرقام حسابات
- صور هوية وطنية أو جوازات سفر

---

## 8. مسؤوليات العميل

- العميل يؤكد امتلاكه حق مشاركة البيانات.
- العميل يُبلّغ عن أي PII لطرف ثالث في الملف.
- العميل يوافق على سياسة الاحتفاظ 90 يومًا.
